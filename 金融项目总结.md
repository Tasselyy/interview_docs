

# 完整链路

用户提交问题后，主 Agent 先做规划和维度拆解，再把任务分发给不同领域 Agent。每个领域 Agent 在 Skill 的约束下按标准 SOP 调用底层 MCP 工具，最后产出统一的结构化 DomainResult。后处理阶段，做 claim 提取、evidence 汇总、来源优先级排序，以及冲突检测。最后交给 Final Synthesizer 生成最终研究报告，并把任务、轨迹、结论、证据全链路沉淀下来。

# 架构设计

## multi agent对比单agent

1. 上下文窗口拥挤，长任务易"遗忘"

单 agent 配置（`us_stock_research.yaml`）给了 `max_turns: 200`，所有工具结果都堆在同一个 `message_history` 里。当用户问"特斯拉最新财报 + 近期新闻 + 宏观利率影响"时：

- 财报数据、新闻文本、宏观数据**全部挤在一个上下文**中
- 前期检索到的关键数字容易被后续大量文本"冲掉"
- `keep_tool_result: 5` 只保留最近 5 条工具结果，早期财报数据可能被丢弃

多 agent 方案中，每个子 agent 有**独立的 message_history**（各 `max_turns: 20`），互不干扰。

2. 无法并行，延迟线性叠加

3. 角色模糊导致工具滥用

单 agent 持有**全部工具**（`search_and_scrape_webpage` + `jina_scrape_llm_summary` + `tool-python` + `us_stock_research` 的所有子工具），LLM 需要自己判断"何时该查财报、何时该查新闻"。

多 agent 通过 `tool_blacklist` 做了**硬隔离**：

- `agent-earnings`：不允许调 `us_stock_news`（新闻工具）
- `agent-news`：不允许调 `company_earnings_summary`（财报工具）

单 agent 没有这层约束，容易出现：用搜索工具查财报数据（而非用专用 API）、反复查相同维度而遗漏其他维度。

4. **Prompt** 职责过载

单 agent 的 system prompt 要同时承担：

- 财报分析（理解 SEC 文件、EPS、forward guidance）
- 新闻检索（时效性、催化剂识别）
- 宏观分析（利率、CPI、Fed 政策）
- 任务分解与综合

多 agent 中每个子 agent 有**专精的角色 prompt**（`agent-earnings`、`agent-news`、`agent-macro`），主 agent 只负责分解和综合。**单一 prompt 越长，LLM 的指令遵循率越低。**

5. **错误传播无隔离**

单 agent 中，一次工具调用失败的 rollback 会影响整个任务流。多 agent 中：

- 子 agent 失败被限制在自己的循环内（独立的 `consecutive_rollbacks` 计数器）
- 主 agent 收到子 agent 的失败结果后可以决定是否重试或跳过
- DEV_SPEC 引用论文数据：Independent 架构错误放大 **17.2×**，Centralized 仅 **4.4×**

6. **缺乏交叉验证**

多 agent 的主 agent prompt 明确要求：

> **Validation**: After receiving sub-agent results, cross-check numbers and sources. If there are contradictions, investigate further.

单 agent 不存在这个"汇审"环节——它自己查的数据自己用，缺少一个独立的视角来发现矛盾或遗漏。

## 为什么选择中心化编排不是去中心化？

主要有三个考虑：

第一，**金融研究的本质是有层级的**。用户问一个问题，比如"分析某个公司的投资价值"，这不是各个Agent平等地讨论，而是需要一个主管级的决策者来理解问题、规划研究维度、汇总结论。

第二，**去中心化协商成本高，金融场景风险高**。如果让多个Agent互相协商、投票、达成共识，这个过程会很冗长，而且对于金融分析这种需要快速给出结论的场景，成本太高。

第三，**可观测性更好**。中心化编排下，每一步决策都是可追溯的，所有的信息流向都清晰。而去中心化的协商过程是黑盒的，对于需要证据链可解释的金融场景，这是个很大的问题。

**Trade-off是什么呢？** 主Agent确实可能成为瓶颈。我们的解决方案是：主Agent只负责规划、派发、汇总三件事，具体的细节研究全部交给子Agent。主Agent本身不做任何IO密集的操作，只做决策。这样它就不会成为系统的瓶颈。

## 主Agent的职责边界怎么划定的？

主Agent的职责有四件明确的事：

**第一个是理解问题**。用户问进来一个自然语言问题，主Agent需要把它转化成结构化的研究计划。比如"分析腾讯的投资价值"，主Agent需要理解这需要财报分析、新闻事件、宏观背景三个维度。

**第二个是规划和派发**。根据问题类型，主Agent决定调用哪些Domain Agent。比如只需要财报分析的问题，就只调earnings Agent。需要综合分析的，就并行调三个Agent。

**第三个是补查决策**。如果某个子Agent的结果不完整或不清晰，主Agent可以决定是否需要补查。这不是让主Agent自己去补查，而是派给对应的子Agent再做一轮。

**第四个是汇总和冲突解决**。各个子Agent的结论可能有冲突，主Agent根据证据的优先级、来源的可信度，来决定最终的结论。

**为什么不把所有事都放在主Agent里？** 因为那样主Agent会成为一个万能的执行体，什么都会做，什么都做不深。分工明确反而能让系统更清晰、更易维护、更易扩展。

## 主 Agent 和子 Agent

主 Agent 和子 Agent 之间不是直接互相聊天，它们是通过**编排器把子 Agent 暴露成一种特殊工具**连接起来的，本质上是‘主 Agent 发起 tool call，编排器去跑子 Agent，再把结果当 tool result 写回主 Agent 上下文’。

## 子Agent的划分依据是什么？

我们的第一版本是三个Agent：**earnings Agent、news Agent、macro Agent**。

划分的依据是**金融分析的天然维度**，不是技术维度。一个专业的金融研究员会从这三个维度来分析一个公司：

- **earnings Agent** 负责基本面。财报数据、利润、增长、财务指标，这些都是硬数据，需要系统化的提取和分析。
- **news Agent** 负责事件催化剂。公司的重大事件、产品发布、高管变动、监管消息，这些信息来自新闻，需要理解其对股价的潜在影响。
- **macro Agent** 负责宏观背景。利率、CPI、汇率、政策变化，这些影响整个市场，也影响这个公司。

## 主Agent规划结果不合理，纠错机制？

**纠错发生在两个地方：**

第一个是**子Agent的执行结果评估**。子Agent返回结果后，主Agent会看这个结果是否完整。如果某个维度的结果质量太差或信息太少，主Agent可以决定对同一个Agent做二轮查询。

第二个是**证据驱动的冲突检测**。如果多个子Agent的结论互相矛盾，这本身就表明主Agent的规划可能有问题，比如某个Agent遗漏了重要信息。这时候主Agent可以指导某个Agent补查。

## DeepResearch 架构

第一，**它不是单轮问答，而是多轮研究**。主 Agent 会先理解问题，再决定要不要拆成财报、新闻、宏观这些维度；独立的维度可以同轮并行下发给子 Agent，有依赖关系的就分轮处理，所以它更像 research workflow，不是一次 prompt 生成。

第二，**它不是只搜一轮，而是持续调工具补证据**。主 Agent 和子 Agent 都有自己的 ReAct 循环，可以反复调用搜索、抓取、财报、Python 分析这些工具，把信息一步步补全。项目本身也强调长上下文和高 tool-call 上限，这就是它能做 deep research 的基础。

第三，**它强调证据链，而不是只给结论**。输出不是随便写一段总结，而是要求 Key Findings -> Evidence & Sources -> Limitations，而且会看来源优先级，像 SEC/IR 这种官方来源优先，冲突时要交叉核对。也就是说，它不是‘生成答案’，而是‘生成带依据的答案’。

第四，**它能处理长任务的工程问题**。比如上下文保留、压缩、失败摘要、子 Agent 超时、partial failure 这些都做了，不会因为一个子 Agent 卡住就整轮挂掉。这一点其实很关键，因为 deep research 真正难的往往不是第一步推理，而是长链路跑到后面还能不能稳住。

第五，**它是可回放、可评测的**。整个过程会记 task log、main/sub agent message history、tool call trace，所以最后不仅知道答案是什么，还知道它怎么得出的。

## 多轮检索什么时候停止

是一个“模型决策 + 工程兜底”的机制，停到模型认为当前证据已经足够，不再请求新工具为止。

具体来说，主 Agent 每一轮都会先看当前已经拿到的信息，再决定下一步要不要继续调工具、继续检索。如果模型这一轮已经不再发起 tool call 了，系统就认为它觉得证据已经够了，开始进入总结和答案生成阶段。所以本质上，‘停不停’主要是由模型基于当前上下文自主判断信息是否足够。

与此同时，工程上还有几层硬约束兜底，比如到了 max_turns、就会被强制停止，避免死循环或者成本失控。

## Agent多步任务中工具的调用出错了，反思纠错

1. 即时纠错：格式错误回滚

当 LLM 输出的工具调用格式不合规（MCP XML标签缺失/畸形）或触发拒绝关键词时，orchestrator 会：

- 回滚当前 turn（`turn_count -= 1`）
- 从 `message_history` 中弹出错误的 assistant 消息
- 重新尝试，最多连续回滚 **5 次**

2. 重复查询检测 

系统维护 `used_queries` 缓存，检测 agent 是否对搜索类工具发出了相同的查询。重复时触发回滚，迫使 agent 换一种策略。

3. 工具执行错误检测 

`should_rollback_result()` 方法检测三类错误结果：

- `"Unknown tool:"` — 工具不存在
- `"Error executing tool"` — 执行异常
- Google Search 空结果 — 查询无效

检测到后同样触发 turn 回滚。

4. 失败经验总结（核心反思机制）

这是最关键的反思设计。当任务未成功完成时，系统会让 LLM 生成结构化的**失败总结**：

```
Failure type: [incomplete / blocked / misdirected / format_missed]
What happened: [描述采取的方法及失败原因]
Useful findings: [列出可复用的中间发现和结论]
```

这些总结会被注入到下次重试的 prompt 中（`prompt_utils.py:24-63`），以 `=== Previous Attempts Analysis ===` 的格式呈现，让 agent 基于历史失败选择新策略。

5. 上下文管理 + 多次重试

当 `context_compress_limit > 0` 时启用。达到 max turns 后：

- 如果不是最后一次重试 → **跳过生成最终答案**，直接生成失败总结
- 下一次重试带着失败经验重新开始

6. 最终答案重试 (`answer_generator.py:255-354`)

提取 `\boxed{}` 格式答案时最多重试 **3 次**，每次弹出错误的 assistant 消息后重新生成。

7. API 层重试 (`anthropic_client.py:100-107`)

使用 tenacity 库对 LLM API 调用做 5 次重试、每次间隔 10 秒。

## 容错：外部 API 调用超时或频繁报错

成了分层机制：最底层对外部 API 调用做有限次 retry 和 timeout 控制，比如搜索、抓取、LLM 调用都会有超时和退避重试；如果这一层还失败，编排器不会立刻把整条链路判死，而是通过 rollback 让模型换一个 query、换一个工具路径，避免机械重复同一次错误调用；再往上，如果某个子 Agent 或某个数据源彻底不可用，系统会走 fallback 或 partial failure，把这一维标记成超时/失败，但保留其它维度的结果继续综合；最后在答案收尾阶段，如果标准格式输出失败，还会做最终总结重试，必要时回退到中间结果做兜底。所以它的核心思路是：局部错误先局部重试，重试不行就局部降级，始终尽量保证主流程继续向前，而不是因为一个外部 API 挂掉就整任务失败。

# Skills

## 怎么做的

放在 Agent 和底层 MCP Tools 之间来做的，它本质是一层 SOP 执行层。Skill 本身会包含几类东西，比如 manifest、输入输出 schema、SOP 流程、tool allowlist、以及结果校验规则。运行时 Skill Executor 会先读取 manifest，完成 Skill 的装载和边界初始化，再按 schema、和 SOP 去约束 Agent 执行

如果拿财报分析举例，比如 earnings 这个 Skill，限定只能访问财报相关工具和允许的网页源，再要求输出必须带关键指标、证据来源、日期和 limitation。

## skill约束层面

1. Prompt / SOP 层软约束
   Agent 会被明确告知这个 Skill 的目标、步骤、证据要求、输出格式，比如只能做财报分析、必须给来源、不能写投资建议。这一层约束的是“怎么想、怎么答”。
2. Tool Policy 层硬约束
   通过 allowlist / blacklist 限制它能看到和能调用的工具。比如 earnings Skill 只能用财报和公告相关工具，看不到新闻专用工具。这一层约束的是“能做什么、不能做什么”。
3. Schema / Checks 层结果约束
   输入要符合 input schema，输出要符合 output schema；如果字段缺失、结构不对、证据不全，就重试或者降级。这一层约束的是“产出必须长什么样”。
4. Runtime / Orchestrator 层流程约束
   编排器控制它在哪个子任务里运行、最多跑多少轮、超时多久、何时停止、失败怎么回退。这一层约束的是“在什么边界内执行”。

## Skill 的完整流程，生命周期

主 Agent 触发 Skill，Skill Executor 加载约束，子 Agent 按 SOP 调工具，最后返回一个经过校验的结构化结果，而不是随便写段总结。

拿 earnings_snapshot 举例，先是 Main Agent 判断这个问题需要财报分析，于是把任务路由给 agent-earnings，再触发 earnings_snapshot 这个 Skill。接着 Skill Executor 会先加载这个 Skill 的定义，包括输入输出 schema、SOP、工具权限和质量检查规则。输入校验通过后，才让子 Agent 在受控工具集合里执行，比如先拉财务数据，再查 IR、SEC、财报发布，抓正文、抽关键信息，然后组装成结构化结果，把结论、证据这些字段带出来。最后再做输出 schema 校验和质量检查，不通过就重试，还是不行就降级成部分结果返回。通过的话，这个结果会回到主链路，交给后面的汇总、冲突处理和最终报告生成。

## Schema校验失败的原因是什么？

**第一类（40%）：格式问题**

- Agent输出了多余的markdown符号，比如三个反引号、多余的空格
- JSON数组或对象没有正确关闭
- 这些通常是模型偶发的格式问题，和模型温度、prompt设计有关

**第二类（35%）：逻辑问题**

- 必填字段缺失，比如evidence字段为空
- 这些通常反映了Agent没有真正理解Skill的输出要求

**第三类（25%）：超时或调用失败**

- Agent在规定时间内没有完成，timeout了
- 工具调用失败，比如网络不通，但Agent没有妥善处理

##  Skills和Tools的职责边界划分？

**MCP Tool是原子操作**，比如：

- 搜索新闻（调Serper API）
- 提取文章内容（调Jina API）
- 查询财务数据（调Bloomberg或爬虫）
- 比较数据（简单的算术或字符串匹配）

**Skill是工作流**，比如：

- earnings_snapshot：搜索最新财报 -> 提取关键指标 -> 分析增长趋势 -> 输出结构化结论
- news_catalyst_scan：搜索最近的重要新闻 -> 提取事件信息 -> 评估影响 -> 输出事件清单
- macro_impact_brief：查询宏观数据 -> 分析变化趋势 -> 评估对行业的影响 -> 输出宏观背景

# 结构和证据对齐

这个项目里，结论-证据对齐不是‘先让模型写结论，最后再补几个链接’，而是从流程上就把它做成显式约束。每个子 Agent 输出的不是自由文本，而是标准化结果，里面同时有结论候选、证据集合，以及结论和证据的映射关系。后面系统会先把结论做归一化，比如统一公司、指标、期间 这些口径，再冲突检测。然后再看每条结论到底挂了哪些有效证据，证据至少要能追到来源、日期和具体内容，而且来源还有优先级，像 SEC、IR 这种官方来源会高于媒体转载。最后，没有有效证据支撑的结论不能进高置信输出；

# 冲突检测

第一步，先从各个子 Agent 的结果里抽 claim 和 evidence。
第二步，做**归一化**，把公司、指标、单位、期间、口径统一掉。比如 revenue / net sales / top line 先归成同一个指标；Q4 2025 和 latest quarter 先映射到同一期间；GAAP 和 non-GAAP 要分开。
第三步，按 comparison_group_key 分桶。只有“同一公司、同一指标、同一期间、同一口径”的结论才放进一个桶里比较。期间不同、口径不同，都不算硬冲突。
第四步，在同一个桶里看值、方向和来源。如果值不一致，就进入冲突分类。我一般分四类：时间口径冲突、统计口径冲突、来源可信度冲突、真实事实冲突。
第五步，做来源排序。像 SEC、IR、official transcript 这种官方来源优先级最高，Reuters、Bloomberg 这类主流媒体次之，二次摘要最低。
第六步，做决策。如果高优来源和低优来源冲突，直接按规则解决，高优覆盖低优。



# 上下文管理

## 压缩策略

平时主要用**规则式截断**，快到边界时再用 **LLM 摘要压缩**。
常态下，系统用 keep_tool_result 做滑动窗口，只保留最近 K 组工具结果，老的结果不直接删，而是替换成短占位文本。这一步是规则式的，优点是稳定、便宜、不引入摘要失真。

但如果任务已经很长，或者准备进入收尾阶段，就会切到 **LLM 压缩**。典型有两种：
第一，子 Agent 跑完后不会把原始搜索结果全塞给主 Agent，而是先生成一份强结构化 summary，再回传给主 Agent。
第二，如果达到最大轮数或者这次尝试失败了，系统会生成一份 failure summary，把这轮做了什么、卡在哪、哪些发现还能复用压成高密度摘要，下一次 retry 直接吃这个摘要，而不是重放整段历史。

## 近因策略

只保留最近 K 条工具结果的完整内容，更早的工具结果不删消息，只把 content 替换成「Tool result is omitted to save tokens」这类占位，这样既省 token，又保留完整推理与动作序列。因为最近几次工具返回对当前决策更有用，更早的可以弱化。

我们有两层「动态」处理 ：

1. 生成长度：若本次回复因 `finish_reason == "length"` 被截断，不会直接报错，而是把本次请求的 `max_tokens` 提高约 10% 后重试（有上限），多次仍截断则接受当前结果继续跑，保证 ReAct 能继续。
2. 上下文长度：在每轮结束、准备做 summary 前，会估一下「当前 history + summary prompt + 预留回复」是否超过 `max_context_length`；若超了，就从尾部动态删掉最后一对 assistant–user，再进入 summary，相当于用「少一轮的对话」换「能执行 summary」的空间，既避免超长，又为 summary 腾出空间。

## 上下文管理的优化性能提升？

**延迟优化：**

- 不做任何context优化：平均延迟 45 秒
- 启用缓存：延迟降低20%

**成本优化：**

- 原始方案：平均 30k tokens/query
- 启用摘要压缩：20k tokens/query (-33%)

**质量影响：**

- 原始方案：报告质量评分 7.8/10
- 启用优化后：质量评分 7.6/10 (only -2.5%, 可接受)

## Token消耗

**Token消耗统计：**

- 一次简单查询（比如"腾讯的最新股价"）：约2-3k tokens
- 一次标准Deep Research（综合三个维度）：约15-25k tokens
- 一次复杂Deep Research（加上风险分析、对标对比）：约30-50k tokens

**有没有做过token用量的优化？**

有的。通过实验我们发现：

- earnings Agent通常最耗token（因为财报数据多）
- 通过压缩财报输出摘要，能省40% token而质量只下降5%

# 持久化

**PostgreSQL负责：**

- 任务的完整执行记录
- 每个Agent的运行日志
- 最终的报告和结论
- Evidence的完整记录
- 这些数据需要长期保存，可能被审计、被用户查阅

**Redis负责：**

- 任务的实时状态（is_running, current_step, eta等）
- 中间的工具调用结果（当前没有必要永久保存）
- 已执行Agent的缓存结果
- 这些数据只需要在任务完成前有效

# 部署

## SGLang优势

第一，**这类场景最浪费的是重复 prefill**。主 Agent 多轮调用、子 Agent 独立会话、还有大段相同 system prompt 和历史上下文，前缀重复非常多。SGLang 的核心优化之一是 **RadixAttention / prefix caching**，可以复用前缀 KV cache，避免每一轮都从头算一遍，所以在多轮 Agent 工作流里收益很明显。

第二，**Agent 系统的请求形态很不均匀**。有的请求是超长上下文 prefill，有的是短回复 decode，还有主 Agent 和多个子 Agent 交替进来。SGLang 提供 **continuous batching、chunked prefill、prefill-decode disaggregation** 这类机制，本质上是在提升 GPU 利用率，减少长请求把短请求完全堵住的问题，所以对长链路、多请求混跑更友好。

第三，**长上下文不仅慢，还很吃显存**。SGLang 在 attention 和 KV cache 管理上做了很多运行时优化，比如 paged attention、层级缓存、量化支持等，所以更容易把大模型和长上下文稳定跑起来，而不是一上来就被显存和吞吐卡死。

# 评测与性能指标

## 性能评测

**第一层是子 Agent 的单元评测。**

每个 Agent 单独测，比如财务 Agent 就拿一批已知答案的财务问题去跑，看它的准确率和召回率。这层主要保证每个模块自己是靠谱的。

------

**第二层是端到端的集成评测。**

我们维护了一批**金融领域的标注数据集**，覆盖常见的查询类型，每条数据都有人工标注的参考答案。系统跑完之后，用几个维度来打分：

- **准确性**：关键事实有没有答对
- **完整性**：该覆盖的点有没有都覆盖到
- **一致性**：有没有自相矛盾的地方
- **证据质量**。关键结论里有多少真的挂上了有效 evidence

打分的方式一部分是规则匹配，比如数字对不对；另一部分是用 LLM 来做裁判，让它对比参考答案给系统输出打分。

## 端到端平均延迟是多少？

**目标延迟：20-30秒**

**延迟分解：**

1. 请求解析和规划（主Agent）：5秒
2. 子Agent并行执行（earnings/news/macro）：30秒（取决于网络和搜索）
3. 冲突解决和汇总：5秒
4. 报告生成和存储：1-2秒

总计：**40秒**（实际数据）

## 并行调度对延迟的优化有多大？

**对比数据：**

| 方案                | 延迟 | 相对改进 |
| ------------------- | ---- | -------- |
| 串行执行（A->B->C） | 50秒 | baseline |
| 并行执行（A\|B\|C） | 30秒 |          |

# Prompt

## 一、System Prompt 的组装方式

最终注入 LLM 的系统提示 = **MCP工具定义 + Agent角色定义**，在 `orchestrator.py` 中拼接：

```python
system_prompt = generate_mcp_system_prompt(date, mcp_servers) 
              + generate_agent_specific_system_prompt(agent_type)
```

------

## 二、MCP 工具定义层 (`generate_mcp_system_prompt`)

`prompt_utils.py:85-164` — 动态生成工具列表和调用格式指令：

- 遍历所有 MCP server，列出每个工具的名称、描述、参数 schema

- 规定调用格式必须用 XML 标签：

  ```xml
  <use_mcp_tool>
    <server_name>xxx</server_name>
    <tool_name>xxx</tool_name>
    <arguments>{"key": "value"}</arguments>
  </use_mcp_tool>
  ```

- 约束：**每条消息只能调用一个工具**，工具调用必须在回复末尾

------

## 三、Agent 角色定义层 (`generate_agent_specific_system_prompt`)

`prompt_utils.py:204-313` — 根据 `agent_type` 注入不同角色 prompt：

| agent_type            | 角色         | 核心要求                                                     |
| :-------------------- | :----------- | :----------------------------------------------------------- |
| `"main"`              | 通用任务解决 | 使用工具逐步解决问题，追求完整准确                           |
| `"agent-browsing"`    | 网页搜索     | 只输出可验证事实，不推测                                     |
| `"us_stock_research"` | 美股研究主控 | **首轮必须调用工具**；非投资建议；结构化输出；子agent调度规则 |
| `"agent-earnings"`    | 财报子agent  | 专注SEC文件、收入、EPS、前瞻指引，**首轮必须调用工具**       |
| `"agent-news"`        | 新闻子agent  | 近期催化剂和新闻，**首轮必须调用工具**                       |
| `"agent-macro"`       | 宏观子agent  | 利率、CPI、就业、美联储政策，**首轮必须调用工具**            |

关键设计点：

- **首轮强制工具调用** — 防止 agent 说"我将搜索..."却不实际行动
- **信息来源优先级** — 官方文件(SEC/IR) > 主流媒体 > 二手摘要
- **子agent调度** — 独立任务并行，依赖任务串行

------

## 四、最终答案生成层 (`generate_agent_summarize_prompt`)

`prompt_utils.py:316-414` — 在所有工具调用完成后，引导 LLM 输出结构化答案：

- **main**: 严格 `\boxed{}` 格式（用于benchmark评测），数字不加逗号，不加单位
- **us_stock_research**: `\boxed{}` + 结构化报告（Key Findings → Evidence & Sources → Limitations）+ 免责声明
- **agent-earnings/news/macro**: 领域专属报告结构（Key Points → Evidence → Confidence & Limitations）

------

## 五、失败反思 Prompt

`prompt_utils.py:24-63` — 任务未完成时，要求 LLM 生成结构化复盘：

```
Failure type: [incomplete / blocked / misdirected / format_missed]
What happened: [描述方法及失败原因]
Useful findings: [可复用的中间发现]
```

注入下次重试时作为 `=== Previous Attempts Analysis ===` 前缀。
