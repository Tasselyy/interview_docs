# RAG

## 什么是RAG

简单来说，RAG就是就是给LLM配了一个外挂的知识库

是为解决LLM的两个核心缺陷：

1. 容易产生幻觉
2. 知识库过时，并且无法直接访问私有数据

标准的RAG一般分为三个核心阶段：

1. **数据入库：**把私有数据（pdf，word）解析，切分成文本块，然后用embedding模型转化成高维向量，存入Milvus，Faiss这种向量数据库
2. **检索召回：**当用户提问的时候，把用户的问题也变成向量，然后去向量数据库中进行相似度计算，把最相关的TOP-K文档片段召回
3. **增强生成：**把召回的文档作为上下文背景，和用户的原始问题一起喂给LLM。

## RAG和SFT的区别，取舍

RAG：成本低，更新简单，但是响应慢，依赖检索质量

SFT：成本高，更新困难，依赖训练数据，但是响应快

RAG场景：实时/高度动态的数据，需要极高的事实准确度和可溯源性

SFT场景：想要模型严格遵循特定格式输出，或者是需要学特定场景下的 Function Calling 范式

## RAG和上下文的区别以及优缺点

**核心区别：**

上下文是静态的，你在 prompt 里写死了什么信息模型就看到什么。RAG 是动态的，根据 query 实时检索，不同问题拿到不同的信息。

**上下文的优势：**

简单直接，没有检索环节就没有检索出错的可能。信息完整可控，你塞进去什么模型就看到什么，不存在漏检的风险。延迟低，不需要额外的检索调用。适合信息量不大、内容固定的场景，比如 system prompt 里放几页产品文档。

**上下文的劣势：**

受 context window 限制，塞不下太多信息。token 成本随信息量线性增长。信息更新需要改 prompt，不够灵活。如果塞太多无关信息反而会干扰模型输出质量。

**RAG 的优势：**

知识库可以无限大，不受 context window 限制。按需检索只注入相关内容，token 利用率高。知识更新只需要更新向量库，不需要动 prompt 或重新训练模型。能引用来源做溯源，可解释性更好。

**RAG 的劣势：**

引入了检索环节，检索质量直接决定最终质量，检索不到或者检索错了比没有更糟。系统复杂度高，要维护 embedding 模型、向量库、chunking 策略、reranker 这一整套 pipeline。延迟更高，多了检索这一跳。还有语义鸿沟的问题，用户 query 和文档的表述方式不一样，可能导致相关内容检索不到。

## RAG 评估

**检索端评估：**

- **Recall@K**：相关文档有没有被捞回来，这是最基础的。你K设5，真正相关的文档能召回几个。

- **Precision@K**：捞回来的里面有多少是真正有用的，噪声太多会干扰生成。

- **MRR（Mean Reciprocal Rank）**：相关文档排第几位，排越前面越好，因为LLM对上下文位置是敏感的。

  

**生成端评估：**

- **Faithfulness（忠实度）**：生成的内容有没有"编"，是不是都能在检索到的文档里找到依据。这个是防幻觉的核心指标。
- **Answer Relevance**：回答跟问题相不相关，有没有答非所问。
- **Completeness**：回答是否完整覆盖了问题要求的信息点。

**最重要的是什么？**

我认为最重要的是**Faithfulness（忠实度）**。原因有三：

1. **RAG存在的意义就是减少幻觉**。如果生成的内容不忠于检索到的文档，那整个RAG pipeline就失去了意义，还不如直接让LLM裸生成。
2. **它是用户信任的基础**。特别在学术场景（像Paper Pilot）和金融场景，一旦出现无中生有的信息，用户对系统的信任就崩了。
3. **其他指标某种程度上依赖它**。如果内容都是编的，relevance再高也没用——答案看着相关但其实是假的，危害更大。

当然，检索端的Recall也很关键，因为如果相关文档根本没捞回来，faithfulness再好也是"忠于错误的上下文"。所以实际中是**Recall保底，Faithfulness把关**的逻辑

## 怎么发现召回数据不准确导致幻觉，如何解决

检索不准是幻觉的上游原因，核心思路就是在检索端提高 recall 和 precision，在生成端加约束和验证。

**离线评估**：

用标注好的 query-document 对，跑 Recall@K、MRR、NDCG 这些检索指标。如果检索阶段的 recall 就低，后面 LLM 生成大概率会编。你 Paper Pilot 里可以用 LLM-as-Judge 做自动评估，让一个模型判断生成的答案是否有检索文档的支撑。

**在线检测**：

- **Faithfulness 检测**：把生成的答案和检索到的 chunks 做对比，看答案中的 claim 是否都能在 chunks 中找到依据。RAGAs 框架就是干这个的，核心指标是 faithfulness score。
- **置信度/不确定性信号**：如果 LLM 输出中出现模糊表述（"可能"、"大概"），或者 logprob 很低，往往说明检索给的上下文不够。
- **空检索 / 低相似度检测**：检索结果的 score 低于阈值时直接标记为"无可靠来源"，比让 LLM 硬编好得多。

## 怎么解决

**检索端优化**（治本）：

- **混合检索**：dense + sparse（BM25）+ RRF 融合，你做 Paper Pilot 时用 Milvus 2.5 原生 BM25 就是这个思路，语义和关键词互补，减少漏召回。
- **Query 改写 / 扩展**：用 LLM 对原始 query 做 rewrite、HyDE（生成假设文档再检索）、multi-query 拆分，提升召回覆盖度。
- **Reranker**：检索完用 cross-encoder（比如 BGE-reranker）做精排，把真正相关的排到前面。
- **Chunk 策略优化**：chunk 太大太小都不行，可以用 parent-child chunking（小 chunk 检索，返回大 chunk 给 LLM），或者加 metadata filter 缩小检索范围。

**生成端优化**（治标 + 兜底）：

- **Prompt 约束**：明确告诉 LLM "只根据以下文档回答，如果找不到答案就说不知道"。
- **Citation / 溯源**：要求 LLM 标注每个 claim 的来源 chunk，这样不仅方便用户验证，也能在后处理中自动检查是否有依据。你的金融多 Agent 系统里的 conclusion-evidence traceability 就是这个思路。
- **Self-RAG / CRAG**：让 LLM 先判断检索结果是否相关，不相关就重新检索或拒绝回答，而不是硬生成。

## RAGAS

核心理念是 LLM-as-a-judge

分为两个模块：

1. 评估generator的质量-LLM回答的好不好：
   * 忠实度：回答是不是百分百来自于检索到的上下文
   * 回答相关性：回答和问题是不是高度相关的
2. 评估Retriever的质量：在向量库中找的准不准：
   * 上下文准确度：TOP-K的质量，也就是看是不是把最相关的排在最前面
   * 上下文召回率：需要提供Ground Truth，评价检索到的上下文是否包含了回答问题所需要的必要信息

## Ragas的Context Precision如果很低，你怎么优化？

**优化可以从几个层次切入：**

> Context Precision 低本质是排序质量问题，优先级上我会先加 Reranker 因为性价比最高，然后检查 chunk 策略是否合理，再调混合检索权重和 top-k 截断，最后考虑 query rewriting

**第一层：检索端优化**

最直接的，检索质量本身有问题。可以从几个角度改：

- **混合检索加权调优**——比如 Paper Pilot 里用 dense + BM25 做 RRF 融合，如果 precision 低，可能是 RRF 的权重不合适，dense 和 sparse 的相对比重需要调。可以在 dev set 上做 grid search 找最优权重。
- **top-k 缩小**——你返回 10 个 chunk 但其实只有 3 个相关，precision 自然低。适当减小 k 值，或者加一个 score threshold 做截断，低于阈值的不返回。
- **embedding 模型升级**——换一个在你领域表现更好的 embedding，比如论文场景可以试学术领域微调过的模型。

**第二层：chunk 策略优化**

检索准不准很大程度取决于 chunk 怎么切的：

- **chunk 太大**——一个 chunk 里混了多个主题，和 query 部分相关部分无关，precision 必然低。缩小 chunk size 或者用语义分段（按段落、section 切）。
- **chunk 太小**——语义不完整，embedding 质量差，检索结果不稳定。
- **加 overlap 或 parent-child 结构**——small chunk 用来检索，命中后返回 parent chunk 提供完整上下文。

**第三层：加 Reranker**

检索是粗筛，reranker 做精排。用 cross-encoder（比如 bge-reranker 或 Cohere rerank）对初步召回的 chunk 重新打分排序，把真正相关的排到前面。这对 precision 提升通常最显著，因为 Context Precision 本质上是个排序质量指标。

**第四层：Query 端优化**

有时候不是检索差，是 query 本身太模糊：

- **Query Rewriting**——用 LLM 把用户模糊的问题改写成更精确的检索 query
- **HyDE**——让 LLM 先生成一个假设性答案，用答案的 embedding 去检索，有时候比直接用 query 效果好

## BM25

### 核心思想

BM25 是 TF-IDF 的改进版。核心直觉：**一个词在文档里出现得越多、在语料库里越稀有，相关性贡献越大。**



$$ \text{BM25}(Q, D) = \sum_{i=1}^{n} \text{IDF}(q_i) \cdot \frac{f(q_i, D) \cdot (k_1 + 1)}{f(q_i, D) + k_1 \cdot \left(1 - b + b \cdot \dfrac{|D|}{\text{avgdl}}\right)} $$

### 三个组成部分

**① IDF — 词的稀有度**

$$ \text{IDF}(q_i) = \ln\left(\frac{N - n(q_i) + 0.5}{n(q_i) + 0.5} + 1\right) $$

$N$ = 总文档数，$n(q_i)$ = 包含该词的文档数。越稀有 → IDF 越高。

**② TF 饱和 — 词频贡献有上限**

通过 $k_1$（典型值 **1.2~2.0**）控制饱和速度。出现 100 次不会比 10 次高 10 倍。

**③ 文档长度归一化**

通过 $b$（典型值 **0.75**）惩罚长文档。$|D|$ 为文档长度，$\text{avgdl}$ 为平均长度。$b=0$ 不惩罚，$b=1$ 完全按比例惩罚。

------

### BM25 vs TF-IDF

| 对比     | TF-IDF       | BM25                 |
| -------- | ------------ | -------------------- |
| 词频     | 线性，无上限 | 饱和，$k_1$ 控制上限 |
| 文档长度 | 不考虑       | $b$ 参数归一化       |

## 检索优化，加速

**核心问题是：暴力搜索（brute-force）是 O(n) 的，数据量大了扛不住，所以需要近似最近邻搜索（ANN）来加速，用一点精度换大量速度。**

**主流的 ANN 方案分四大类：**

**1. 基于图的索引——HNSW（Hierarchical Navigable Small World）**

目前最主流，Milvus 默认推荐的就是这个。核心思想是构建一个多层跳表式的图结构，顶层稀疏做粗定位，逐层往下细化搜索。查询时间复杂度大概是 O(log n)，精度非常高，通常 recall 能到 95% 以上。缺点是内存占用大，因为要存图的邻接关系，构建索引也比较慢。适合对精度要求高、数据规模中等（百万到千万级）的场景。

**2. 基于量化的方法——IVF + PQ**

IVF（Inverted File Index）先用 K-Means 把向量空间聚成若干簇，查询时只搜最近的几个簇，缩小搜索范围。PQ（Product Quantization）把高维向量切成子空间分别量化，大幅压缩存储和计算量。IVF-PQ 组合起来既能缩搜索范围又能降计算成本，适合大规模数据（亿级），内存友好，但精度比 HNSW 低一些。

**还有一些工程层面的加速手段：**

GPU 加速——Faiss 支持 GPU 版的 IVF-PQ，批量查询吞吐量能提升一个数量级。向量降维——入库前用 PCA 或 Matryoshka Embedding 降维，维度低了所有算法都快。预过滤——先用 metadata 过滤缩小候选集，再在子集上做 ANN，避免全库搜索。

## Chunking

### 为什么要切

1. Embedding 模型限制：
   * 输入长度有限
   * 长文本会稀释关键信息，影响检索多大精度和准度
2. 检索回来的片段要喂给LLM，但LLM上下文窗口有限制

### 如何避免语义丢失

* 滑动窗口靠重叠部分
* 直接语义分割
* 分割之后评估语义，丢失严重就重新分割

## Graph RAG

传统向量检索擅长找语义相似的内容，但很难处理跨文档的关联推理。比如问人物之间的关系，信息散落在十几个段落里，靠embedding相似度很难串起来。Graph RAG的做法是先从文档里抽取实体和关系建成知识图谱，检索时在图上做关系遍历，特别适合需要全局理解的问题。

#### **架构上分两大阶段：**

**Indexing阶段（离线构建）：**

- 用LLM对文档做**实体抽取和关系抽取**，识别出人名、机构、方法、概念等实体，以及它们之间的关系
- 构建Knowledge Graph，节点是实体，边是关系
- 在图上做**社区发现**（比如Leiden算法），把密切关联的实体聚成社区
- 对每个社区生成**community summary**，这就是全局知识的压缩表示

**Query阶段（在线检索）：**

- **Local Search**：从query中提取实体，在图上找到相关的邻居节点和关联关系，连带相关的原始text chunk一起作为context。适合具体的、局部的问题。
- **Global Search**：利用社区summary做全局性的归纳总结。把各个社区的summary拿出来，用map-reduce的方式让LLM做分层汇总。适合"趋势是什么""有哪些主要流派"这类全局问题。

#### GraphRAG的难点

**1. 图构建质量严重依赖LLM抽取能力**

实体和关系是LLM抽取的，不是规则匹配的。这意味着：

- 抽取会漏、会错，尤其是领域术语和隐含关系
- 同一个实体的不同表述（"BERT"、"bert模型"、"Bidirectional Encoder"）需要做**实体消歧和对齐**，这本身就是个NLP老大难问题
- 关系类型定义得太粗或太细都有问题，需要针对领域做schema设计

**2. 构建成本极高**

微软原版GraphRAG的indexing阶段要对每个chunk都调LLM做抽取，文档量大的时候token消耗巨大。据社区反馈，处理几百篇文档就可能花掉几十到上百美元的API cost。而且构建时间很长，不像传统RAG切chunk+embedding那么快。

**3. 图的规模和查询效率**

图大了之后，subgraph检索的效率怎么保证？社区发现算法跑一次可以，但图结构变了要重新跑，开销不小。Local search的时候遍历多少跳邻居、怎么剪枝，都需要调。

**4. 噪声传播**

图上一个错误的边可能导致推理链路走偏。传统RAG检索到一个noisy chunk影响还可控，但图上错误关系会通过路径传播，影响范围更大。

#### GraphRAG如何应对增量场景

**挑战在于：**

增量不只是"加几个节点几条边"这么简单，因为新文档可能：

- 引入新实体和新关系
- 改变已有实体之间的关系权重
- 影响社区结构——原来两个社区可能因为新关系被合并

**实际解决思路：**

**1. 增量实体/关系抽取 + 图merge**

新文档进来后只对新文档做抽取，然后把新的实体和关系merge进已有的图。关键难点是**实体对齐**——新文档里提到的实体跟图里已有的是不是同一个？需要做entity linking或者用embedding相似度做匹配。

**2. 增量社区更新**

不全量重跑社区发现，而是只对受影响的局部子图做社区重新划分。判断受影响范围的方法：看新增的边连接了哪些已有社区，只重新计算这些社区及其邻居社区。对应的community summary也只重新生成受影响的那几个。

**3. 版本化或时间戳机制**

给节点和边打上时间戳或版本号，这样可以支持：

- 按时间范围过滤图谱内容
- 处理信息过时的问题（旧的关系可能不再成立）
- 做知识的生命周期管理

**4. 实际工程上的trade-off**

> "完全的实时增量更新其实很难做到完美，工程上通常是混合策略——小增量用增量merge，累积到一定量之后做一次全量rebuild来修正累积误差。就像数据库里LSM-tree的思路，小的写入先buffer，定期做compaction。"

# Agent

## 基础

### agent的运行流程

**核心循环就一句话**：**感知 → 思考 → 行动 → 观察 → 再思考**，不断循环直到任务完成。

**具体展开**：

**1. 接收输入**：用户发来一个 query，加上 system prompt（角色定义、可用工具描述、约束条件）一起送给 LLM。

**2. 规划/推理**：LLM 分析任务，决定下一步该做什么。这里有两种主流模式：

- **ReAct**：Thought → Action → Observation 循环，每一步都有显式的推理过程，LLM 先想（我需要查什么），再决定调什么工具。
- **Function Calling**：模型直接输出结构化的工具调用（tool_calls），不需要显式的 thought chain，更紧凑高效。你 Paper Pilot 用的就是这种。

**3. 工具执行**：框架拿到 LLM 输出的 tool call，去实际执行——调 API、查数据库、跑检索，拿到结果。

**4. 观察反馈**：把工具返回的结果拼回 messages（作为 tool result），再送给 LLM。

**5. 迭代判断**：LLM 看到工具结果后，决定是继续调工具（回到第 2 步），还是信息够了直接生成最终回答（输出 END）。

**6. 输出结果**：返回最终答案给用户。

### agent模块

Agent 系统五个核心模块：感知层理解输入、规划层分解任务、工具层执行动作、记忆层管理状态、输出层整合结果。再加上贯穿全流程的反思机制做自我纠错，和安全护栏做输入输出防护。本质上就是一个感知-规划-执行-反馈的闭环，每个模块对应闭环中的一个环节。



## 常见框架对比

### LangChain

它提供了一整套模块化组件：prompt template、LLM调用封装、tool/function calling、memory、document loader、vector store对接等等。核心抽象是**Chain**——把多个处理步骤串成一条链，比如先检索再生成，就是一个Retrieval QA Chain。

优点是生态极其丰富，基本市面上所有的LLM、向量数据库、工具API都有现成的集成

LangChain更适合做**单链路的LLM应用**

### LangGraph

专门解决LangChain做不了复杂workflow的问题。

核心抽象从Chain变成了**有状态的有向图**——节点是处理步骤，边是条件跳转，整张图共享一个State对象。这意味着你可以做循环、分支、条件路由、human-in-the-loop、甚至多Agent协作。

显式的全局State所有节点共享，支持checkpoint可以断点恢复，支持streaming和human-in-the-loop中断。

### AutoGPT

核心理念是给LLM一个高层目标，它自己做任务分解、自己决定用什么工具、自己执行、自己反思，完全自主循环，不需要人类干预。

它的执行循环大概是：Think→Plan→Act→Observe→Reflect→继续或结束。看起来很酷，但实际体验问题很多：容易陷入无限循环，一个简单任务可能调几十次LLM；token消耗极大；中间过程不可控，你不知道它为什么突然决定去搜索某个东西。

### Dify

是一个**低代码/无代码的LLM应用开发平台**

它提供了一个可视化的界面，你可以通过拖拽节点来搭建workflow——RAG pipeline、chatbot、Agent都可以。底层支持接各种LLM provider、向量数据库、工具API。

Dify的核心优势是**降低门槛**。产品经理、运营人员这些非技术角色也能搭LLM应用，不用写代码。它还自带prompt管理、数据集管理、监控日志、API发布这些运维功能，是一个比较完整的平台。

### CrewAI

专注**多Agent角色协作**

核心抽象是三个概念：Agent（有Role、Goal、Backstory的角色）、Task（具体任务）、Crew（一组Agent组成的团队）。你定义好角色和任务，框架自动驱动协作。

## LangGraph

### 状态怎么流转的

**流转方式**：

1. **普通边 `add_edge(A, B)`** — A 执行完无条件跳到 B。
2. **条件边 `add_conditional_edges(A, route_fn, mapping)`** — `route_fn` 接收当前 state，返回一个字符串 key，mapping 把 key 映射到下一个节点名。这是实现分支/循环的核心。
3. **入口/出口** — `set_entry_point` 定义起始节点，`set_finish_point` 或返回 `END` 结束。

**状态更新机制**：每个节点是一个函数，接收 state，返回一个 partial dict。LangGraph 用 **reducer** 合并更新到全局 state。默认是覆盖，但你可以用 `Annotated[list, operator.add]` 这种方式做追加合并（比如 messages 列表）。

### 适用场景

**1. 带自我反思/纠错循环的Agent**

这是LangGraph最经典的场景。比如我Paper Pilot里做的：生成完答案之后有一个reflection节点，让LLM自己评估回答质量——有没有幻觉、有没有遗漏关键信息。如果不合格就循环回去重新检索、重新生成。这种"生成→检查→不行就重来"的循环，在LangGraph里就是图上加一条回边加一个条件判断，很自然。LangChain做不了这种循环，它的Chain是单向的。

**2. 多步骤条件路由的workflow**

比如用户query进来，先做意图分类：如果是事实型问题走RAG检索路径，如果是闲聊走直接生成路径，如果是敏感问题走拒答路径。每条路径后面可能还有不同的后处理逻辑。这种多分支的路由用LangGraph的条件边来定义非常清晰，每个分支就是图上的一条路径。

我在Paper Pilot里就有类似的设计——query先过一个路由节点，判断需不需要检索、需要检索哪些数据源，然后走不同的子图。

**3. 需要Human-in-the-loop的场景**

有些步骤需要人来确认或干预。比如Agent要执行一个高风险操作——发邮件、写数据库、调外部API，你希望执行前人先确认一下。LangGraph支持在任意节点设置interrupt，暂停执行等人确认后再继续，而且配合checkpoint机制可以把当前state存下来，人确认后从断点恢复。

**4. 多Agent协作编排**

多个Agent各自负责不同子任务，需要一个Orchestrator来协调。LangGraph可以把每个Agent封装成一个子图或一个节点，Orchestrator作为顶层图来调度。Agent之间的通信通过共享State来完成，比消息传递更可控。

### 创建agent方式

**1. 预置ReAct Agent（create_react_agent）**

LangGraph提供了一个`create_react_agent`的工厂函数，你只需要传入LLM和一组tools，它自动帮你搭好一个标准的ReAct循环——LLM思考→决定调哪个tool→执行tool→拿到观察结果→继续思考或输出最终答案。

底层其实也是一张图，有两个核心节点：一个是LLM调用节点，一个是tool执行节点，中间有条件边判断"要不要继续调tool"。但这些你都不用自己写，一行代码就出来了。

适合场景是**标准的tool-calling Agent**，比如给LLM接一堆API工具让它自己决定怎么用。但workflow是固定的ReAct循环，你没法加自定义节点、没法改控制流。

**2. StateGraph自定义图（核心方式）**

你自己定义State schema，自己定义节点函数，自己定义边和条件路由，完全手动搭图。流程大概是：

第一步，定义State——就是一个TypedDict或Pydantic model，声明图里所有节点共享的状态字段。比如messages、retrieved_docs、generation_count这些。

第二步，定义节点函数——每个节点接收当前state，做一些处理，返回state的更新。比如retrieval节点接收query从state里取出来，去检索，把结果写回state的docs字段。

第三步，用`StateGraph`把节点和边组装起来——`add_node`加节点，`add_edge`加普通边，`add_conditional_edges`加条件路由。最后`compile`编译成可执行的图。

**3. 子图嵌套（Subgraph）做多Agent**

每个sub-Agent本身是一个完整的StateGraph，有自己的节点和状态。父图的节点就是这些子图，父图负责决定调用顺序、数据怎么在子图之间流转。

子图可以有自己独立的State schema，跟父图的State做mapping。这样每个Agent的内部逻辑是封装的，对外只暴露输入输出接口，关注点分离做得比较好。

适合场景是**复杂多Agent系统**，每个Agent内部有自己的workflow，Agent之间又有协作关系。

### 多轮上下文怎么管理

LangGraph管理多轮上下文的核心就是它的**State机制**。整张图共享一个State对象，每一轮对话的所有信息都在State里面流转和累积。

#### **问题1：上下文越来越长，token爆炸**

多轮对话聊久了messages列表会很长，直接全塞给LLM会超context window，而且token成本也顶不住。

**解决方案是做上下文压缩**，几种策略：

滑动窗口——只保留最近N轮对话，早期的消息丢弃。最简单但会丢失重要的早期信息。

摘要压缩——在State里加一个summary字段，每隔几轮让LLM把之前的对话压缩成一段摘要，后续对话的context就是summary加最近几轮messages。这样既控制了长度又保留了关键信息。我在Paper Pilot里用的就是类似思路。

token budget控制——设一个上限，比如context最多给4000 tokens，超了就触发压缩。可以在图里专门加一个"context管理"节点，每轮对话前先检查token量，超了就压缩。

#### **问题2：不是所有信息都该放在messages里**

多轮对话中会产生很多中间信息——检索到的文档、工具调用结果、内部推理过程。这些全塞messages里会导致对话历史很脏，LLM容易被噪声干扰。

**做法是State里分字段管理**：

python

```python
class State(TypedDict):
    messages: Annotated[list, add_messages]     # 用户和LLM的对话
    retrieved_docs: list                         # 当前轮检索到的文档
    user_intent: str                             # 当前用户意图
    conversation_summary: str                    # 历史摘要
    tool_results: dict                           # 工具调用结果
```

messages只存用户可见的对话内容，其他信息用独立字段管理。各节点按需读取各自需要的字段，不会互相污染。这是LangGraph比纯message passing模式强的地方——State可以携带结构化的上下文，不是所有东西都要塞进message列表。

#### **问题3：多轮之间需要跨session持久化**

用户关掉页面再回来，对话历史不能丢。LangGraph提供了**Checkpointer机制**，可以把State持久化到外部存储。

内置支持MemorySaver（内存，开发用）和SqliteSaver / PostgresSaver（生产用）。每一步执行完都会自动保存一个checkpoint，包含完整的State快照。恢复的时候传一个thread_id就能从上次的状态继续。

实际用法：

python

```python
from langgraph.checkpoint.memory import MemorySaver

checkpointer = MemorySaver()
graph = workflow.compile(checkpointer=checkpointer)

# 每次调用传thread_id
config = {"configurable": {"thread_id": "user_123"}}
graph.invoke({"messages": [user_input]}, config)
```

同一个thread_id的调用会自动接上之前的State，实现跨session的多轮上下文。

### 流式处理

`stream_mode="updates"` 或 `"values"`

这两种模式用于跟踪多智能体图的流转过程。

- `"updates"`：每次某个节点（Agent）执行完毕，只返回该节点对全局状态（State）做的**增量修改**。
- `"values"`：每次节点执行完，返回**完整的全局状态**。

`stream_mode="messages"`用于 Token 逐字输出

- 在这个模式下，LangGraph 将内部节点生成的每一个 Token 作为 `Chunk` 抛出来。

### LangGraph 和动态逻辑链的核心区别，底层原理

**LangGraph** 的核心是一个**可循环的状态机**。它用图的节点和边来**显式定义**工作流，侧重于**流程的绝对可控性**。

LangGraph 的底层维护着一个全局的 Pydantic  State。在图的每一次流转中，在各个节点之间传递。

**Nodes：** 通常是一个 Python 函数或一个 Agent。它接收当前的 State，执行逻辑（比如调用 LLM 或工具），然后返回对 State 的更新（通过 Reducer 机制合并数据）。

**Edges：** 决定下一步去哪个节点。条件边根据 LLM 的输出动态决定是继续向下走，还是循环回去重做。

**底层运行机制：** 它本质上是一个**基于图的迭代器**。系统在后台维护着一个执行栈和Checkpoints，每次节点执行完毕后，框架会评估边逻辑，如果是条件边，则调用路由函数决定下一个入栈的节点。

## openclaw

### OpenClaw 是什么?

OpenClaw 是一个开源的、可自托管的个人AI助手平台。本质上是一个事件驱动的Agent执行引擎，前面放了一个多渠道网关。用TypeScript编写，运行在你自己的机器上，所有数据和密钥不离开本地。

### OpenClaw 的架构是什么?

三层架构:触发层、网关层、Agent层。触发层有五种事件源(Messages、Heartbeats、Crons、Hooks、Webhooks)不断产生事件;网关层负责协议转换、认证和路由分发;Agent层负责推理(Think)、工具执行(Act)和记忆(Remember)。本质是一个事件驱动的执行管道:事件产生网关路由Agent执行状态持久化。

### Gateway 作用是什么?

Gateway是一个本地长驻的Node.js进程，监听单一端口(默认18789)，同时提供HTTP和WebSocket。它是触发层和Agent层之间的中间件，负责渠道连接管理、协议转换、会话路由、认证鉴权、插件加载和设备配对。关键边界是:Gateway不做推理、不调LLM、不执行工具，只负责"消息到了该给谁"和"回复该送回哪里"。

### 有哪些消息源/触发源?


五种:
a.Messages一用户通过聊天渠道发消息，事件驱动
b.Heartbeats一固定间隔定时器(如每15分钟)，让Agent 主动唤醒
c.Crons一用户定义的计划任务(cron表达式)，执行具体的定时工作
d.Hooks一Gateway/Agent 内部生命周期事件触发的回调
e.Webhooks一外部系统通过HTTP POST 触发加上Agents可以派生 Subagents，形成递归执行。

### 为什么能接入各个平台?

因为Gateway 层的Channel Manager 把每个渠道封装为一个独立的Channel Plugin，每个插件实现统一的接口。不同渠道的协议差异(Telegram BotAPI、Discord WebSocket)被封装在各自插件内部，对上层Agent 暴露一致的消息接口。加一个新渠道只需写一个Channel Plugin，Agent代码零改动。目前支持20+个渠道。

### 长短期记忆如何组织的

短期记忆就是当前会话的对话上下文，包括用户消息、助手回复、工具调用结果这些内容，全部在ContextWindow 里以消息列表的形式存在。它的特点是容量受模型上下文窗口限制，比如 Claude是200K tokens,满了之后旧的对话就会被压缩成一段结构化摘要，只保留最近约20,000tokens的原始消息。所以短期记忆本质上是临时的、会被压缩的。

长期记忆则完全基于磁盘上的Markdown 文件，分成两种组织方式。第一种是MEMORY.md，这是一个策展式的长期记忆文件，存放的是用户偏好、持久性决策、重要事实这些需要长期保留的内容，由Agent主动维护和更新，每次会话启动时注入上下文，不受压缩影响。第二种是memory/YYYY-MM-DD.md，按日期组织的追加式日志，存的是当天的运行笔记、阶段性进展、临时但值得回顾的上下文，会话启动时只读入今天和昨天的内容。

长期记忆还配有一套检索索引机制。系统会对所有记忆文件建立混合搜索索引，包括向量语义搜索和 BM25关键词搜索，Agent 在需要的时候通过 memory_search 工具按需检索，不用把全部历史记忆都塞进上下文里。这样即使记忆文件越积越多，也不会撑爆上下文窗口。两者之间的衔接靠的是压缩前记忆冲刷:当短期记忆快满需要压缩时，系统会先静默提醒 Agent把当前对话中重要的内容写入长期记忆文件，然后再做压缩。这就保证了短期记忆中真正有价值的信息能沉淀到长期记忆里，不会因为压缩而彻底丢失。

### 四层记忆系统原理

第一层是 Bootstrap Files, 相当于硬盘上的件。包括 SOUL.md、AGENTS.md、USER.md、MEMORY.md等Markdown文件，存放的是 Agent 的人格设定、行为规则、用户身份和策展过的长期知识。它们每次会话启动时都从磁盘重新读取注入上下文，完全免疫压缩，是最稳定的一层记忆。不过有字符上限，单文件20,000字符，总计150,000字符，所以要保持精简。

第二层是Session Transcript，相当于文件系统日志。完整的对话历史用JSONL 格式逐条追加写入磁盘，包括用户消息、助手回复、工具调用和压缩摘要。它是跨重启持久的，继续会话时从JSONL重建上下文。但模型不是看全部记录，而是只看当前上下文窗口能装下的部分，更早的内容通过压缩变成摘要。

第三层是Context Window,相当于 RAM。就是模型每轮推理时实际能看到的所有内容，包括系统提示、BootstrapFiles、对话历史、工具返回结果都挤在这个固定大小的空间里。容量由模型决定，满了就触发Compaction。压缩时把最近约20,000 tokens 原样保留，更早的消息用LLM生成结构化摘要替代，摘要里必须保留任务状态、决策理由、TODO和关键标识符。压缩前还会静默提醒 Agent 把重要信息写入记忆文件，防止信息丢失。

第四层是Retrieval Index，相当于搜索引擎。系统对所有记忆文件建立混合搜索索引，同时使用向量语义搜索和BM25关键词搜索，按7:3 权重融合排序。Agent 通过 memory_search和memory_get 这两个工具按需检索，不需要把全部记忆都塞进上下文。检索是模型自主决策的，不是每轮自动触发。索引还支持时间衰减和去重，新笔记排名更靠前

这四层的协同逻辑是:Bootstrap Files提供稳定的身份和规则基座，Session Transcript 保存完整的历史记录，Context Window 管理当前可见的工作记忆并在满时通过压缩保持可用，Retrieval Index 让 Agent 能按需回忆历史而不占用上下文空间。关键设计原则是只有写入磁盘文件的信息才是真正持久的记忆，纯对话中没有持久化的内容压缩后就不可恢复了。

### 采用了什么Agent架构?

递归Centralized(中心化)架构。Parent Agent作为中心，工具创建子通过LLM的tool call机制调用sessions_spawnAgent执行子任务，子Agent完成后 push结果回父，父Agent等所有子任务完成后整合输出。子Agent 可以继续 spawn孙Agent，形成一棵 Orchestrator-Worker 树，每一层都是该层的中心

### 为什么要选择中心化架构?

根本原因是对话本身的结构是中心化的--有确定的入口(用户发消息)，有确定的出口(给用户一条完整回复)，中间的执行可以分发，但责任链不能断。如果换成去中心化(P2P协商)，最终谁来发回复没有确定答案;如果换成Independent(各自独立)，结果无法拼合成一条完整回复。Centralized 的父子结构正好对应这个"一进一出”的模型:Parent Agent 是唯一的回复责任人，子Agent只是它延伸执行能力的手段。

### Centralized系统性能受限?


1.减少 Parent 的工作量:Parent 只做编排和最终整合，不做具体执行。OpenClaw 在子Agent 的系统提示词里规范。子 Agent 不允许跑题、不允许主动发消息Parent 只需要等结果、做最终决策，每轮turn 的工作量被压到最低。

2.并行化子任务:Parent 可以同时 spawn多个子Agent 并行执行，不需要串行一个一个等。总耗时取决于最慢的那个子Agent，而不是所有子任务时间之和。Parent 在提示词里也被告知可以同时 spawn多个任务，依靠 push-based announce各自回来触发汇总。

3.并发数量限制:并行是加速手段，但无限并行会耗尽系统资源，所以OpenClaw 同时设置了两层上限作为反向约束:单个session的活跃子Agent 数（默认5），全局所有子 Agent 的并发数（默认8）。超出上限时‘sessions_spawn 直接返回‘forbidden',防止雪崩。

4.递归下推编排:子Agent 本身也可以变成 Orchestrator，继续向下 spawn 孙Agent，把编排压力分散到子层。Parent 只负责顶层任务拆解，不需要直接管所有叶子节点。

5.按角色分配模型:sessions_spawn的参数支持‘model和‘thinking 覆盖，可以给Parent 配置推理能力强的模型做决策，给执行型子Agent 配置更快更轻量的模型做具体任务，按角色分配算力而不是一刀切。

### Agent是如何通信的?

通信模式上是工具调用，控制权始终留在 Parent Agent。具体实现是 push-based异步消息:子Agent完成后通过调用 Gateway把结果以"用户消息”的形式投递到父Agent的消息队列，触发父Agent的下一轮LLM推理。父Agent不轮询、不阻塞线程，等消息触发即可，
传递内容上只传最终结果，子Agent的中间推理步骤全部留在私有 session里，不传给父Agent，有效控制上下文膨胀。



## harness

### 定义

Harness Engineering是围绕 AI 模型构建运行环境、调度机制和管控基础设施的一种工程实践。它的目标不是单纯让模型“更聪明”，而是让AI Agent在长时、复杂、多步骤的真实任务中，能够稳定、可靠、可控地持续运行。
如果用一句更通俗的话来说，Harness Engineering做的事情，就是不再把AI 当成一个一次性回答问题的黑盒，而是把它放进一个被设计好的工作系统里，让它按规则做事。
它背后的设计哲学也很重要:模型只是原材料，环境才决定结果能不能落地。
再强的模型，如果直接裸跑长任务，也会遇到上下文腐化、失忆、幻觉完成、任务漂移等问题。所以Harness Engineering的核心，不是继续往模型上堆能力，而是通过外部记忆、任务拆解、测试门禁、状态交接和人工审核，把AI的执行过程工程化。

### 解决问题

它主要解决的是 AI在长时任务里的可靠性问题。
第一个问题是ContextRot，也就是上下文腐化。模型做长任务时，上下文窗口会不断被历史对话、错误尝试和无关信息填满，最后导致它丧失对原始目标的把握。
第二个问题是Hallucinated Completion，也就是幻觉完成。模型在复杂任务中迷失后，往往不会老实说“我不会了"，而是会输出一个看起来完成、实际上并不正确的结果。
第三个问题是Model Drift，也就是模型漂移。多步执行中，模型会逐渐偏离最初目标，最后做出来的东西方向就歪了。
所以我理解Harness Engineering的价值，不是让模型在单轮回答里更强，而是让它在几十步、几百步的连续工作流里，依然能沿着正确轨道往前走。

### 和 Prompt ,Context 的关系是什么?

我会把它理解成一个演进关系，而不是替代关系。
Prompt Engineering主要解决的是“对AI说什么”，也就是如何设计指令，让单次输出更好。
Context Engineering进一步解决的是“让AI知道什么”，也就是在一次session或上下文窗口里，给它什么信息、给多少信息、怎么组织信息。
而Harness Engineering更进一步，它解决的是“让AI 在什么环境里做事”。
也就是说，Harness Engineering并没有取代前两者，而是把它们包进了一个更大的系统里。Harness内部的每个session还是要做Context Engineering，每一次给 agent 的具体任务描述还是要做Prompt Engineering。只是现在关注点从“优化一次对话”升级成了“设计一整个运行体系”。

### 如果让你用Harness工程来实现一个项目

从一个Initializer+Coding Loop的最小闭环开始，而不是一开始就做复杂的多agent 平台。第一步，我会先准备一份清晰的App Spec/PRD，把产品目标、功能范围和验收标准定义清楚。

第二步，我会实现一个Initializer Agent。它的职责不是写业务代码，而是读取需求文档，把大任务拆成一组细粒度、可执行、可验证的feature，并生成类似feature_list.json 这样的状态文件。同时它还会初始化项目骨架、代码仓库，以及后续循环执行所需的基础环境。

第三步，我会实现一个Coding Loop。每一轮都启动一个全新的coding agent，让它只读取当前状态文件、进度记录和代码库信息，只完成一个明确的小任务。任务完成后，不是让agent自己宣布成功，而是必须经过测试、lint 或类型检查等确定性验证;只有验证通过，才更新状态文件、写入交接记录并提交代码。然后这个agent立即退出，下一轮由新的agent接力。

第四步，我会把外部记忆做成系统核心，而不是把连续性寄托在模型上下文里。让每个新agent启动时都能快速接手，而不是依赖前一轮session的“记忆”。

第五步，我会在关键节点加入Human-in-the-loop。在高风险功能、关键界面变更或者多轮失败之后，我会设计人工确认断点，确保系统不会沿着错误方向持续推进。

## MCP

### 定义

MCP是一种标准化接口协议，统一LLM和外部资源的连接方式

用于让LLM能访问外部**数据，工具，提示词**

MCP采用的是服务端-客户端的架构，包括 client， host 和 server

host就是运行LLM的程序，client在host里面运行，负责和 MCP server通信

server中暴露**数据，工具，提示词**

典型的优势是：动态能力更新（只更新工具参数），接口的标准化

### MCP vs Function Calling

| **维度**       | **Function Calling**                   | **MCP (Model Context Protocol)**                  |
| -------------- | -------------------------------------- | ------------------------------------------------- |
| **层级**       | 大模型的原生能力 (Model capability)    | 系统级通信协议 (System architecture protocol)     |
| **核心解决**   | 结构化输出动作指令                     | 标准化 AI 客户端与外部系统的连接                  |
| **包含内容**   | 仅限工具调用 (Tools)                   | 资源 (Resources), 提示词 (Prompts), 工具 (Tools)  |
| **生态复用性** | 低（每个应用都要自己写集成和调度逻辑） | 高（一次编写 Server，各种标准化 Client 通用接入） |

### 远端 Server vs 本地 Server

| **维度**         | **本地 Server (Local)**                               | **远端 Server (Remote)**                                     |
| ---------------- | ----------------------------------------------------- | ------------------------------------------------------------ |
| **通信协议**     | `stdio` (进程间通信)                                  | Streamable HTTP，可选SSE                                     |
| **部署位置**     | 客户端同机（如开发者的个人电脑）                      | 云服务器 / Docker 容器 / 企业内网                            |
| **扩展性**       | 受限于本地                                            | 扩展性强                                                     |
| **安全与鉴权**   | 无需网络鉴权，安全责任依赖客户端拦截                  | 远端Server端通常会有完善的认证机制(OAuth)，安全性更高，支持加密、访问控制 |
| **网络延迟**     | 极低（几乎为零）                                      | 取决于网络环境，可能带来额外的延迟                           |
| **资源访问能力** | 读写本地文件、执行本地命令行脚本、访问 localhost 服务 | 访问内部微服务、企业级数据库、调用受限的第三方 API           |

### MCP接入多个测评工具时，如果两个工具对同一个问题回答格式不统一，怎么处理？

分层处理——优先在 MCP Server 端用 Adapter 模式统一输出 schema，改不了就在 Orchestrator 层做归一化映射，兜底可以用 LLM 动态解析。另外格式统一之外还要注意评分标准的对齐，不同工具的分数不能直接横向比较。

## Skills

### 定义

skill 是对 Prompt，Tool Use，和 Workflow 的包装

是一个预定义的，可以复用的功能模块，用来让agent执行特定SOP任务

包含了：

1. SKILL.md: 包含了 Yaml 头和Body。Yaml 是md文件中的头，告诉AI是否加载这个SKILL，Al必定会加载。而body是正文，是渐进式披露的，只有AI读取了YAML头，认为这个SKILL应该被加载，才会加载这个正文部分。
2. Scripts: 可选。包含着在AI执行过程中需要使用的脚本文件。
3. Reference: 可选。存放执行SKILL相关的文档，说明，内容以MD为主。
4. Assets:可选。存放执行SKILL是需要的模板，静态资源，比如图片等。

### Skill vs MCP



Skills和MCP的核心差别是: MCP提供给模型数据，而Skils教会模型如何使用这些数据 .

 二者区别为:

* MCP侧重于提供工具调用，而Skills则侧重于提示词。MCP更像是一个标准的工具箱，提供规范，可复用的工具。而Skills则是一个带目录的说明书。
* MCP的本质是一个独立运行的程序，而Skills本质是一个文档。他们本质不同，决定了他们性质也不同，Skills更适合处理轻量级的，简单的逻辑，安全性和稳定性都不及MCP。
* Skills采用渐进式披露提示词的机制，Tokens消耗低。而MCP则是一次性塞入所有提示词，Token消耗高。一般只会连cc几个MCP工具，而skills可以有十几个。
* Skills的开发难度相比于MCP，开发难度低很

总结来说，SKILL和MCP二者不是相互替代，而是相互补充的。

在同时使用了多个MCP Server的时候，也可以用SKILL对多个编排多个MCP Server的工作流

在开发MCP SERVER的时候，可以在MCP Server的文档中添加一个链接，指向一个SKILL的链接，这个SKILL让用户/AI更清楚知道该MCP如何使用，遇到问题如何解决，让模型调用MCP时取得更稳定的结果。

### Skill vs Prompt

Skills相比于Prompts的核心差别在于他们的形态和生命周期，Prompts是一次性对话内容，你需要自己保存，复制和粘贴。

模型不会知道什么时候用哪一个Prompt，而且你需要把Prompts 全部塞到上下文里面。这会大量占用你的上下文空间。

Skill则是将提示词以一种更加**工程化**的包装方式打包，他是一个文件夹的形式。他会动态加载/渐近式披露提示词，更加节省提示词。**容易复用和版本管理**，易于协作和共享。

总结来说，Skill 不是一个更复杂的提示词，而是把复杂的提示词，变成了一个有结构，可复用，可以自动调用的模块。

### 如何写 skill

**你永远不可能在第一次尝试时就写出一个完美的Skill**。前几次运行Skill时，输出可能带有很强的“AI生成”痕迹，但当你运行并迭代10到20、30次后，它会变得越来越完美。

**反馈循环的核心步骤可以总结为：调用（Invoke） -> 观察（Watch） -> 给予反馈（Feedback） -> 修复完善（Fix） -> 重复测试**

 **针对不同症状的“调试与迭代指南”** 为了将一个完成度90%的Skill提升到接近100%，视频提供了一套针对性的除错和迭代方法：

**如果AI执行的步骤错误或顺序颠倒：** 直接去修改 `skill.md` 文件中的步骤指令。

**如果输出缺乏特定的语调、风格或上下文：** 需要添加参考文件（Reference files），并确保在 `skill.md` 中正确指向了该文件路径。

**如果AI反反复复犯同一个错误：** 在Skill中增加一条明确的“规则（Rule）”作为护栏。

**如果AI在使用某个工具时遇到困难，或反复搜索相同的内容：** 为该工具或内容创建一个专门的参考文档供其读取。

**如果Skill运作良好，但你觉得还能更好：** 只能“暴力破解”——不断地反复运行，对它做错或可以改进的地方进行挑剔和微调。

## 上下文

### 上下文工程

Agent 上下文工程包含七个模块：系统指令定身份、工具描述定能力、任务上下文定目标、执行轨迹记历史、检索知识补外部信息、长期记忆补用户信息、全局状态协调多 Agent。核心工程挑战是动态组装和 token 预算管理——在有限的 context window 里，让每一步都看到最相关的信息。

### 上下文压缩

### 在长对话中，除了截断，你了解哪些更高效的上下文压缩方法？

按工程实践中的优先级，我会首选滑动窗口加摘要的混合方案，简单可靠。如果对话特别长或者需要跨 session 记忆，就上外挂向量检索，本质上把 context 管理从"塞进窗口"变成"按需检索"。再进一步优化可以加关键信息抽取做结构化存储，以及用 LLMLingua 这类工具做 token 级压缩。核心原则是让每一个 token 都承载高信息密度。

## Memory

第一层是**短期 memory**，靠最近几轮上下文和摘要维持对话连续性；第二层是**长期 memory**，把用户偏好、任务状态、关键事实存到外部存储中，再做检索增强；第三层是**memory policy**，也就是决定写入、更新、删除和遗忘的策略。传统系统大多做到前两层，但如果想把系统做得更稳，真正关键的是第三层。MemSkill 给我的启发就是，memory policy 不应该完全写死，而应该被学习、被评估、再持续演化。

### MemSkil

“我最近看了一篇 **MemSkill**，它研究的是 LLM Agent 的长期记忆管理。它的核心不是再用固定的 add、update、delete 这类手写规则去做 memory，而是把 memory operation 抽象成一个 **可学习、可进化的 skill bank**。具体来说，系统里有一个 **controller**，会根据当前的 text span 和检索到的已有 memory，选择一组最相关的 Top-K skills；然后一个 **LLM executor** 再基于这些 skills，一次性生成 skill-guided memory update。这个设计把传统的 turn-level、规则驱动 memory pipeline（langgraph short term memory），变成了 span-level、skill-conditioned 的 memory construction。更进一步，它还引入了一个 **designer**，会从训练里的 hard cases 里分析失败模式，去 refine 旧 skill，或者提出新 skill，所以它优化的不只是 memory 内容本身，而是‘如何构造 memory’这套策略空间。作者整体是用一个 closed-loop 的方式在做：一边用 RL 训练 controller 学会选 skill，一边让 designer 迭代 skill bank。我觉得这篇论文最有价值的点，就是它把 agent memory 从静态模块，推进成了一个可以自我演化的系统。” ([arxiv.org](https://arxiv.org/html/2602.02474v1))



从系统设计角度看，我觉得它最强的地方在于把 **memory policy** 和 **memory content** 分开建模了，前者由 skill bank 和 controller 负责，后者由 executor 负责，这样扩展性和泛化性都会更强。

## 性能评估

agent 最大问题往往不是不会调工具，而是不会做决策

agent 的评估不能只看最后成功率，是一个执行过程，所以我一般会看四件事

第一层是**结果层**。
 就是最后任务有没有完成，这个肯定要看。但我不会只看一个很粗的成功率，因为很多真实任务不是“做成了”这么简单，它往往有很多约束条件，比如时间对不对、预算有没有超、用户偏好有没有满足、工具调用结果是不是符合要求。所以我会把任务拆成更细的检查项，而不是只看最后对不对。

第二层是**过程层**。
 我会看 agent 在执行过程里的表现，比如它有没有先澄清问题，有没有做合理规划，工具调用顺序对不对，前后信息有没有对上。因为 agent 和普通问答不一样，它是一个长流程任务，很多问题其实不是出在最后结果上，而是出在中间某一步走偏了。所以我觉得评估不能只看 final result，还要看 trajectory，也就是整条执行轨迹。

第三层是**稳定性**。
 这个我觉得特别重要。因为有些 agent 不是不会做，而是有时候能做对，有时候又不行。那这种在真实业务里其实很危险。所以我会区分两件事：一种是“多跑几次，至少有一次做对”，另一种是“多跑几次，次次都做对”。前者说明它有能力，后者才说明它稳定、能落地。这个区别我觉得特别关键。

第四层是**按复杂度拆开评估**。

失败归因，要同时看**推理复杂度、工具复杂度、交互复杂度**

因为不同任务难点不一样。有的任务难在推理，也就是 reasoning；有的难在工具之间的依赖关系，也就是 tool use 不只是会不会调用，而是什么时候调、按什么顺序调；还有的难在交互，因为用户说得不清楚、会改需求、会漏信息，所以要看是否能识别出模糊意图，主动澄清。这三种复杂度其实要分开看，不然你只看总分，很难知道 agent 到底卡在哪。

VitaBench 对这个问题讲得挺好。它的一个核心观点就是，真实世界里的 agent 评估，不能只看 function calling 对不对，而是。而且它还专门区分了“偶尔成功”和“稳定成功”。这点我很认同，因为这比单纯看一个成功率更接近真实落地。

## Multi-Agent 架构

1. 先判断：这个任务到底适不适合 multi-agent

“我会先判断任务是不是真正 agentic。也就是它是不是需要持续和环境交互、是不是部分可观测、是不是要根据反馈不断调整策略。因为很多人以前在非 agentic benchmark 上测出来的 multi-agent 优势，放到真正 agentic 任务里并不一定成立

2. 再判断：任务能不能自然拆分

“我最先看的其实不是复杂度，而是能不能可拆解。Finance 这种能拆成并行子任务的场景，多 agent 就会好很多；但 像是强顺序、强状态依赖的任务，所有 multi-agent 架构都变差。所以真正决定要不要上 multi-agent 的，往往不是任务难不难，而是能不能拆。

3. 看 single-agent baseline 高不高

“我一般会先做一个 strong single-agent baseline。当 single-agent baseline 已经超过大概 45% 的时候，继续加 coordination 往往收益很小，甚至转负。也就是说，单 agent 已经做得不错时，多agent的沟通和管理成本可能比收益还大。

4. 最后再选架构

如果任务可分解，而且需要结果汇总和校验，我更偏向 centralized；如果任务更像搜索、探索、网页导航这种高熵空间并行探索，我会更偏向 decentralized，因为它可以实时交流，更新计划；如果任务本身强顺序依赖，我反而倾向先坚持 single-agent，而不是硬上 multi-agent。

## Prompt

## 最佳实践

**1. 角色设定（System Prompt）**

开头明确告诉模型它是谁、要做什么。"你是一个金融分析师，专注于A股市场分析"比"帮我分析股票"效果好得多。角色设定会锚定模型的知识范围和回答风格。

**2. 指令要具体，不要模糊**

差的："帮我总结一下这篇文章"。好的："用3个要点总结这篇文章的核心论点，每个要点不超过50字，使用中文"。把格式、长度、语言、风格都写清楚，模型就不需要猜。

**3. 结构化输入输出**

用 XML 标签或者 Markdown 分隔不同的内容区域，比如 `<context>` 放背景信息，`<question>` 放问题，`<output_format>` 放输出要求。让模型能清晰区分指令和数据，避免 prompt injection 风险。输出也一样，如果需要结构化结果就明确指定 JSON schema 或者模板。

**4. Few-shot 示例**

给2-3个输入输出示例，尤其是输出格式复杂或者任务有歧义的时候。示例要覆盖典型 case 和边界 case。比如做分类任务，正面、负面、中性各给一个例子。

**5. CoT 引导**

复杂推理任务加一句"请一步步思考"或者在示例中展示推理过程。对数学、逻辑、多步分析类任务提升很大。

**6. 正面指令优于否定指令**

差的："不要用专业术语"。好的："用高中生能理解的语言解释"。模型对"要做什么"的遵循度远高于"不要做什么"。

**7. 约束和兜底**

明确边界条件："如果信息不足，回答'无法判断'而不是编造"。"如果用户问的超出金融领域，礼貌拒绝"。这些 guardrail 在生产环境里非常关键，防止 hallucination 和越界回答。

**8. 迭代优化**

没有一次写完的完美 prompt。先写一版跑几个 case，看哪里不符合预期，针对性地加约束或示例。常见的调优循环是：输出太长就加长度限制，格式不对就加示例，内容跑偏就加角色约束。

### Prompt Chain 模式

不是让模型一次把所有事情做完，而是把一个复杂任务拆成多步，每一步用一个单独的 Prompt 去完成，再把上一步结果传给下一步。

### prompt 压缩

**主要方案分三类：**

**第一类：Token 级压缩——LLMLingua 系列**

核心思路是用一个小模型（比如 GPT-2 级别）算每个 token 的 perplexity，perplexity 低的说明信息量小、可预测性强，直接删掉。保留 perplexity 高的 token，因为它们承载了更多信息。压缩率能做到 2-5 倍，而且对下游任务的性能影响通常很小。LLMLingua-2 进一步用了数据蒸馏的方式训练了一个专门的压缩分类器，比原始 perplexity 方法更准。

**第二类：句子/段落级压缩——抽取式**

不是删 token，而是删整句或整段。对 prompt 中的每个句子评估和 query 的相关性，保留相关的，删掉不相关的。本质上就是在 prompt 层面再做一次检索筛选。可以用 embedding 相似度做，也可以用 cross-encoder 打分。比 token 级更粗粒度，但实现简单，适合 RAG 场景里 chunk 冗余明显的情况。

**第三类：摘要式压缩——生成式**

用 LLM 把长 prompt 压缩成一段短摘要。比如把 5000 token 的检索结果摘要成 500 token。好处是压缩率最高、可读性强。缺点是本身需要一次 LLM 调用，有额外成本和延迟，而且摘要过程会有信息损耗。适合离线预处理或者对同一段内容反复使用的场景。

### CoT FewShot

**Few-shot**

在 prompt 里给几个示例，让模型照着学。比如你要让模型做情感分类，给它三个"输入→输出"的例子，它就能理解你要的格式和标准。本质是通过 in-context learning 让模型理解任务模式，不需要微调。对应的 zero-shot 就是不给例子直接让模型做。

**CoT（Chain-of-Thought）**

让模型在给出最终答案之前先把推理过程一步步写出来。最简单的方式就是在 prompt 里加一句"let's think step by step"。核心思想是把复杂问题分解成中间推理步骤，每一步的输出作为下一步的输入，降低单步推理难度。

**CoT 为什么能提升质量：**

本质原因是 Transformer 的每一次 token 生成都是一次前向计算，计算量是固定的。如果让模型直接输出答案，相当于要求它用一次前向计算完成整个推理，复杂任务根本不够用。CoT 通过让模型生成中间 token，等于给了模型更多的计算步骤来处理问题。每个中间步骤的输出会进入后续的 context，影响后面的 attention 计算，相当于把复杂推理拆成了多次简单推理的串联。

另外一个角度是，中间步骤让模型"自我暴露"推理过程，如果中间某步推错了，后续步骤有机会基于错误结果进一步修正（虽然不保证），比直接跳到答案的黑盒推理可控性更强。

**怎么组合使用：**

最强的组合是 **Few-shot CoT**——在 prompt 里给几个带完整推理过程的示例。不只是给输入输出，而是给"输入→推理步骤→输出"的完整链条。模型同时学到了任务格式和推理模式。

**一个实际 prompt 的结构通常是这样的：**

先是 system prompt 定义角色和约束，然后几个 few-shot 示例展示推理过程，最后是用户的实际问题。模型会模仿示例的推理模式来处理新问题。

**进阶变体：**

**Self-Consistency**——同一个问题用 CoT 跑多次（调高 temperature），取多数投票的答案。用冗余换准确性。

**CoT + Tool Use**——推理到某一步发现需要外部信息，就调用工具获取，再继续推理。这其实就是 ReAct 的思路，把 CoT 和 Agent 的工具调用结合起来。

# LLM 基础原理

## **解释Transformer的核心架构，特别是Self-Attention机制。**

------

Transformer的核心思想是完全抛弃了RNN的循环结构，用注意力机制来建模序列中任意两个位置之间的依赖关系。整体架构是一个编码器-解码器结构，编码器负责理解输入，解码器负责生成输出，但现在很多模型只用其中一半——比如BERT只用编码器，GPT和Qwen这类只用解码器。

**自注意力机制是整个架构的核心。** 它的作用是让序列中的每个token都能"看到"其他所有token，并且根据相关性决定该关注谁。

具体计算过程是这样的：对于输入序列中的每个token，通过三个不同的线性变换矩阵，分别映射出三个向量——Query、Key和Value。直觉上理解，Query是"我在找什么信息"，Key是"我能提供什么信息"，Value是"我实际携带的信息内容"。

然后计算注意力权重：用Query和所有Key做点积，得到的分数表示两个token之间的相关程度。点积结果除以根号下Key的维度，这个缩放是为了防止点积值过大导致softmax之后梯度过小。缩放后过一个softmax归一化成概率分布，最后用这个概率分布对所有Value做加权求和，得到当前token的输出表示。

用公式来说就是：Attention(Q, K, V) = softmax(QK^T / √d_k) × V。

**多头注意力是在此基础上的扩展。** 不是只算一组Q、K、V，而是把向量空间切成多个子空间，每个"头"独立算一组注意力，最后把所有头的输出拼接起来再做一次线性变换。好处是不同的头可以关注不同类型的关系——比如有的头关注语法上的临近依赖，有的头关注语义上的远距离关联。

**除了注意力之外，每一层还有两个关键组件。** 一个是前馈网络，就是两层全连接加一个激活函数，负责对注意力的输出做非线性变换。另一个是残差连接加层归一化——残差连接解决深层网络的梯度消失问题，层归一化稳定训练。

**还有一个重要的设计是位置编码。** 因为自注意力本身是对位置无感知的——打乱输入token的顺序，计算结果是一样的。所以需要额外注入位置信息。原始Transformer用的是正弦余弦函数的固定位置编码，现在主流模型比如Qwen2.5用的是旋转位置编码也就是RoPE，它把位置信息编码到Q和K的旋转角度里，好处是天然支持相对位置建模，而且对长度外推更友好。

## **GPT和BERT的区别是什么？**

**架构上，** GPT用的是Transformer的解码器部分，BERT用的是编码器部分。解码器有因果掩码，每个token只能看到左边的上文，看不到右边；编码器没有这个限制，每个token可以同时看到左边和右边的所有token。所以BERT是双向的，GPT是单向的。

**训练目标上，** GPT做的是自回归语言建模，就是给定前面的token预测下一个token，本质上是在学"怎么往下写"。BERT做的是掩码语言建模，随机把输入中15%的token遮住，让模型根据上下文去预测被遮住的token，本质上是在学"怎么理解整句话"。

**这两个设计选择直接决定了它们的能力倾向。** BERT因为能双向看上下文，对语义理解特别强，天然适合分类、命名实体识别、语义相似度这类理解型任务。但它不擅长生成，因为训练时没学过"一个一个往外蹦token"这件事。GPT因为是自回归的，天然就是一个生成模型，擅长文本生成、对话、续写这类任务。而且后来的发展证明，把GPT足够放大之后，通过生成的方式也能做理解任务——比如你让它生成一个分类标签，效果也不差。

**用法上也有明显区别。** BERT的典型用法是预训练加微调——在下游任务上加一个分类头做微调。GPT的典型用法是提示词工程或者指令微调，通过设计好的提示词引导模型生成想要的输出，不一定需要在下游任务上单独训练。

**如果面试官追问为什么现在大模型都走GPT这条路线而不是BERT，** 核心原因是自回归目标的扩展性更好。不管什么任务都可以转化成"给定上文生成下文"的形式，所以一个模型可以通过扩大规模来覆盖越来越多的任务。BERT的掩码预测目标相对局限，它学到的是填空能力，很难自然地泛化到开放式生成。而且自回归模型的规模效应更明显——参数量从几亿到几千亿，能力是持续涌现的，这就是为什么GPT这条路线最终成了主流。

## **什么是Token？Tokenization的常见方法有哪些？**

A：Token是模型处理文本的最小单元，可以是一个字、一个词、或者一个子词片段。比如"unhappiness"可能被切成"un"+"happiness"。

常见方法有BPE（GPT系列用的）、WordPiece（BERT用的）、SentencePiece（LLaMA用的，对多语言支持好）。

BPE的原理是从字符级别开始，反复合并出现频率最高的相邻对，直到达到预设词表大小。这样常见的词会作为一个整体保留，罕见的词会被拆成子词。

## **什么是大模型的"幻觉"（Hallucination）？如何缓解？**

A：幻觉就是模型一本正经地胡说八道——生成的内容看起来很合理，但实际上是错的或编造的。比如你问它某口井的数据，它可能编一个看起来像模像样但完全虚假的数字。

缓解方法我总结了几个：第一，用RAG，让模型基于检索到的真实数据回答，而不是凭空生成；第二，在Prompt里加约束，告诉模型"不确定的时候要说不知道"；第三，对关键输出做事实核查，比如生成的SQL先验证再执行；第四，降低temperature减少随机性；第五，在领域数据上微调提高准确性。

## Temperature，top k， top p对比

**Temperature**

LLM 输出时，最后一层 softmax 会给每个 token 一个概率。Temperature 就是在 softmax 之前对 logits 除以一个系数 T。T=1 是原始分布；T<1 会让概率分布更尖锐，高概率 token 更突出，输出更确定更保守；T>1 会让分布更平坦，低概率 token 也有机会被选中，输出更随机更有创意。T 趋近 0 就退化成 greedy decoding，永远选概率最高的。

**Top-K**

在采样前，只保留概率最高的 K 个 token，其余的概率直接归零。比如 K=50，不管词表有多大，只从前 50 个候选里采样。好处是简单直接地砍掉长尾噪声。问题是 K 是固定的——有时候模型很确定只有两三个合理选项，K=50 就引入了不必要的噪声；有时候合理选项有上百个，K=50 又太小了。

**Top-P（Nucleus Sampling）**

解决了 Top-K 固定 K 值的问题。不是选前 K 个，而是按概率从高到低累加，累加到总概率超过 P（比如 0.9）就截断。这样当模型很确定时，可能只选两三个 token；不确定时，候选集自动扩大。比 Top-K 更自适应。

**它们之间的关系：**

不是互斥的，实际使用中经常组合。比如先用 Top-K 粗筛，再用 Top-P 精筛，最后用 Temperature 调整分布的锐度。执行顺序一般是 Temperature 先作用于 logits，然后 Top-K 截断，再 Top-P 截断，最后从剩余候选里采样。

## KV cache

KV Cache 就是把已经算过的 Key 和 Value 缓存起来，避免每生成一个新 token 都重新算一遍前面所有 token 的 KV。

**代价是什么：**

内存。KV Cache 的大小 = 层数 × 2（K和V）× 序列长度 × 隐藏维度 × batch size。对于大模型这个开销非常大，比如一个 70B 模型、序列长度 4K，KV Cache 可能要占好几个 GB 显存。所以长上下文场景下 KV Cache 本身就是瓶颈。

**围绕这个瓶颈的优化方案：**

**MQA（Multi-Query Attention）**——所有 attention head 共享同一组 KV，只有 Q 是多头的。KV Cache 直接缩小到原来的 1/head_num。

**GQA（Grouped-Query Attention）**——MQA 和 MHA 的折中，把 head 分组，每组共享一组 KV。LLaMA 2 70B 用的就是这个，平衡了精度和显存。

**PagedAttention**——vLLM 的核心技术。把 KV Cache 从连续显存改成分页管理，像操作系统的虚拟内存一样，解决了显存碎片化问题，大幅提升 batch 吞吐。

**量化 KV Cache**——把缓存的 KV 从 FP16 量化到 INT8 甚至 INT4，显存减半或更多，精度损失很小。

### 为什么不cache query

**因为 Query 没有复用价值。**

每一步生成时，只有**当前新 token** 的 Query 参与 attention 计算。上一步的 Query 算完就用完了，后面永远不会再用。

而 Key 和 Value 不一样——每一步生成时，**所有历史 token 的 K 和 V 都要参与计算**。当前 token 的 Q 要和所有历史的 K 做点积算 attention score，再和所有历史的 V 做加权求和拿到输出。所以 K 和 V 每一步都被重复使用，缓存它们才有意义。

## FP8是什么

**FP8 就是用 8 个 bit 表示一个浮点数，是专门为深度学习推理和训练设计的低精度格式。**

**先回顾常见的浮点格式做对比：**

FP32 是 1 位符号 + 8 位指数 + 23 位尾数，精度高但占 4 字节。FP16 是 1+5+10，2 字节。BF16 是 1+8+7，牺牲尾数精度换更大的指数范围，训练中很常用。FP8 再砍一半，只有 1 字节。

**FP8 有两种变体，这是关键：**

**E4M3**——1 位符号 + 4 位指数 + 3 位尾数。尾数多一点，精度相对更高，但动态范围小。适合**前向传播**，因为权重和激活值的分布相对集中，需要精度多于范围。

**E5M2**——1 位符号 + 5 位指数 + 2 位尾数。指数多一位，动态范围更大，但精度更粗。适合**反向传播**，因为梯度的数值范围波动大，需要更大的指数范围来避免溢出或下溢。

**为什么需要 FP8：**

核心就是省显存和加速。相比 FP16 显存减半，相比 FP32 减到四分之一。而且 NVIDIA H100 等新一代 GPU 原生支持 FP8 Tensor Core 运算，FP8 矩阵乘法的吞吐量是 FP16 的两倍。对大模型推理来说，KV Cache 用 FP8 存储直接省一半显存，计算也更快。

**精度损失怎么办：**

一般配合量化校准（calibration）使用。跑一小批数据统计每层激活值的分布范围，计算一个 scale factor，把数值映射到 FP8 能表示的范围内。只要 scale 选得好，精度损失通常在 1% 以内，对推理质量影响很小。

# 微调

## 大模型微调最重要的是什么

最重要的是数据质量

**1. 大模型本身能力已经很强，微调是在"引导"不是在"教"**

预训练阶段模型已经学到了海量知识和语言能力，微调的本质是告诉模型"在什么场景下用什么方式输出"。所以微调数据不是越多越好，而是要精准地示范你想要的行为模式。几千条高质量数据往往比几万条噪声数据效果好得多。

**2. 坏数据的伤害远大于少数据的伤害**

如果训练数据里有错误的标注、矛盾的指令、低质量的回答，模型会学到错误的pattern，而且这种"学坏了"很难通过加更多数据来纠正。特别是SFT阶段，模型对数据分布非常敏感，几条极端的坏样本就可能带偏整个模型的行为。

**3. 数据决定了微调的上限，训练技巧决定你能逼近这个上限多少**

LoRA vs 全量微调、learning rate怎么调、要不要用cosine scheduler——这些是在已有数据质量基础上的优化。数据不行的话这些技巧全白搭。

## 常见微调方式

**1. 全量微调（Full Fine-Tuning）**

所有参数都参与训练，效果天花板最高，但显存和算力开销巨大。7B模型全量微调至少要2-4张A100。适合数据充足、资源不差钱的场景。

**2. LoRA / QLoRA**

LoRA就是冻住原始权重，在旁边加一对低秩矩阵BA做增量更新，训练参数量压缩到不到1%。

QLoRA更进一步——base model用4-bit量化加载，LoRA参数用BF16训练。这样一张消费级显卡（比如24G的4090）就能微调7B模型。代价是量化带来一点精度损失，但实际测下来跟全量微调差距很小。

**3. Prefix Tuning / P-Tuning**

不改模型参数，而是在输入前面加一组可学习的连续向量（soft prompt）。模型本身完全冻住，只训练这组前缀向量。参数量极少，但效果一般不如LoRA，而且对prefix长度敏感。现在用的人不多了，基本被LoRA替代。

**4. Adapter Tuning**

在Transformer每一层里插入一个小的adapter模块（通常是降维→非线性→升维的bottleneck结构），冻住原始层只训练adapter。思路跟LoRA类似但实现不同，LoRA是加旁路不改结构，Adapter是在结构里插新模块。缺点是推理时多了额外计算，而LoRA可以把BA合并回原始权重，推理零开销。这也是LoRA现在更主流的原因。

## LoRA

### 原理

LoRA的核心思路是：微调时权重的变化量其实是低秩的，没必要更新所有参数。

 具体来说，原始模型某一层的权重矩阵是 **W₀**，维度是 **d × k**。全量微调是学一个同样大小的 **ΔW**。LoRA 的做法是把 **ΔW** 分解成两个小矩阵的乘积： > **ΔW = B × A** > > 其中 B ∈ ℝ^(d × r)，A ∈ ℝ^(r × k)，r ≪ min(d, k)，一般设 4、8、16 这个量级。 训练时冻住原始的 **W₀**，只训练 **A** 和 **B**。前向传播就变成： > **h = W₀·x + B·A·x** 原始输出加一个低秩旁路。初始化上，**A** 用高斯随机初始化，**B** 初始化为零，这样训练开始时 **ΔW = 0**，模型行为跟原始完全一致，起点是稳定的。还有一个缩放系数 **α / r** 控制 LoRA 分支的贡献幅度。 

 对一个 **d × k** 的线性层：

- **全量微调**：d × k 个参数 
- **LoRA**：r × (d + k) 个参数 

预训练模型已经学到了很通用的知识表示，微调只是把这些表示稍微调整一下来适配下游任务。这种调整的本质自由度远小于权重矩阵的维度。原论文也验证了，对全量微调得到的 **ΔW** 做 SVD，前几个奇异值就占了绝大部分能量，rank 设到 8 基本就能逼近全量微调的效果。

### QLoRA

在 LoRA 基础上加了三个优化：

第一，把冻结的原始模型权重量化到 4-bit（NF4 格式），大幅压缩模型体积。NF4 是专门为正态分布的神经网络权重设计的 4-bit 数据类型，比普通 INT4 精度更高。

第二，双重量化（Double Quantization），对量化的 scale factor 再做一次量化，进一步省显存。

第三，分页优化器（Paged Optimizer），用 CPU 内存做 GPU 显存的溢出页，防止训练时显存 OOM。

LoRA 的部分（A 和 B 矩阵）还是保持 BF16 精度训练，保证训练质量。所以 QLoRA 的精髓是**推理用 4-bit 省显存，训练用 BF16 保精度**。

# 后训练

## 常见后训练方式

**1. SFT（Supervised Fine-Tuning，监督微调）**

用标注好的instruction-response对来训练模型，让它学会"按指令办事"。这是从base model到chat model最关键的一步。比如给模型看几万条"用户问什么→应该怎么回答"的样本，模型就学会了对话格式和遵循指令的能力。

**2. 对齐训练（Alignment）**

让模型的输出符合人类偏好，主要两条路线：

RLHF路线：先用偏好数据训一个Reward Model，再用PPO做强化学习，让policy去最大化reward。效果上限可能更高，但工程复杂、训练不稳定。

DPO路线：跳过Reward Model，直接在偏好pair上做supervised learning。工程简单、训练稳定，我Paper Pilot里用的就是这个。

还有一些变体比如KTO（只需要好/坏的单条标注，不需要配对）、IPO（改了DPO的loss解决过拟合问题）。

**3. 持续预训练（Continual Pre-Training）**

用领域语料继续做next token prediction，让模型补充领域知识。比如要做医疗场景，先拿大量医学论文和病历做持续预训练，再做SFT。这步不改训练方式，就是预训练的延续，但数据换成了领域相关的。

典型的后训练pipeline是：持续预训练（可选）→ SFT → 对齐训练。

## RLHF完整流程

**阶段一：SFT（Supervised Fine-Tuning）**

拿一个预训练好的基座模型，用人工标注的高质量问答数据做有监督微调。这一步的目的是让模型学会基本的指令遵循能力，能按照人类期望的格式和风格回答问题。输出的是一个 SFT 模型，是后面两个阶段的起点。

**阶段二：训练奖励模型（Reward Model）**

让 SFT 模型对同一个 prompt 生成多个不同的回答，然后让人类标注员对这些回答做偏好排序——哪个更好哪个更差。用这些偏好对数据训练一个奖励模型，它的输入是一个 prompt + response，输出是一个标量分数，代表这个回答有多符合人类偏好。训练目标是让排序靠前的回答得分高于排序靠后的，通常用 Bradley-Terry 模型做 pairwise ranking loss。

**阶段三：PPO 强化学习优化**

用奖励模型的打分作为 reward signal，通过 PPO（Proximal Policy Optimization）算法优化 SFT 模型的策略。具体流程是：模型生成回答 → 奖励模型打分 → PPO 根据分数更新模型参数，让模型倾向于生成高分回答。

这一步有一个关键设计：**KL 散度惩罚**。在 reward 里减去一个当前模型和 SFT 模型之间的 KL 散度项，防止模型为了刷高奖励分数而过度偏离原始能力，也就是防止 reward hacking——比如模型学会说一堆讨好人的废话但实际内容变差了。

## DPO

**DPO 和 PPO 的核心区别：PPO 是传统强化学习方法，DPO 是一种更简单的偏好对齐方法。**把原来强化学习里的奖励学习过程，隐式地写进了一个更简单的优化目标里。

**PPO**需要先有一个 **reward model**，也就是先训练一个奖励模型，用它给模型输出打分，然后再用强化学习去更新大模型。所以它的流程通常是 **SFT → 奖励模型 → PPO优化**。

而 **DPO** 的思路更直接。它不再单独训练 reward model，也不真的去跑完整的强化学习采样更新。它直接利用 **人类偏好数据**，也就是 chosen 和 rejected 这种成对数据，让模型直接学会“为什么这个回答比那个更好”。

第一个区别是**架构复杂度**。PPO要四个模型同时加载——policy、reference、reward、value，显存开销巨大，分布式训练的协调也很复杂。DPO只需要两个——policy和reference。任何能跑SFT的框架，改改loss函数就能跑DPO。

第二个是**训练稳定性**。PPO是RL算法，有RL的老问题：reward hacking，就是模型学会钻reward model的漏洞拿高分但实际质量不行；超参敏感，clip ratio、KL coefficient调不好容易崩；训练过程方差大，loss曲线经常抖。DPO是supervised loss，训练曲线平稳，超参基本就一个β需要调，控制偏离reference的程度。

第三个是**训练效率**。PPO是online的，每个step要让模型先做generation采样，再算reward，再更新参数，生成这步是瓶颈，非常慢。DPO是offline的，偏好数据预先准备好，训练时只做forward和backward，不需要在线生成，速度快几倍到十倍。

第四个是**能力上限**，这个要客观讲。PPO的优势在于在线探索——训练过程中不断生成新样本拿反馈，理论上能探索到偏好数据覆盖不到的区域。DPO受限于离线数据分布，如果偏好数据覆盖不够或质量不高，天花板可能比PPO低。业界有些实验显示复杂任务上PPO上限更高，但多数场景下DPO性价比更优。

## on-policy 和 off-policy 的区别

**on-policy** 的意思是，
 我现在用哪个策略做决策，就用这个策略刚刚采样出来的数据来训练自己。
 也就是说，**采样策略和训练策略是同一个**。

比如 **PPO** 就是典型的 on-policy。
 模型先按当前策略生成一批回答，再根据这些回答的 reward 去更新自己。更新完之后，之前那批老数据基本就不能直接反复用了，因为它不是“新策略”产生的。

而 **off-policy** 是，
 训练时可以使用**不是当前策略产生的数据**。
 这些数据可以来自旧策略，也可以来自经验回放池，甚至可以是别的策略收集的。
 所以 **采样策略和训练策略可以不一样**。

## GRPO

**对于同一个 prompt，GRPO 会先让模型生成一组 outputs。**
 然后根据 reward，比如答案对不对、格式对不对、推理是否满足要求，给这一组回答打分。接着它不是直接看某个回答的绝对分数，而是看它**相对于组内其他回答**的表现。表现高于组平均的，就是正向更新；低于组平均的，就负向更新。
 所以它学到的其实是：**同样一道题，哪些回答比其他回答更值得保留。** 

它和 PPO 的一个很大区别是：

**PPO 通常要有 value model，GRPO把这部分弱化或者省掉了。**
 这样做的好处是训练链路更简单，显存开销也更低，特别适合长文本生成和 reasoning 场景。也正因为这个特点，GRPO 后来在很多 reasoning model 训练里都很受关注。

**因为 reasoning 任务里，很多题很难直接定义特别精确的逐步价值函数，但很容易比较“这几个回答里哪个更好”。**
 GRPO 就利用了这种“相对比较”信号，把学习问题从“估计绝对价值”变成了“做组内排序”，这通常更稳定，也更符合大模型生成多个 candidate 再筛选的实际过程。

**所以我会把 GRPO 理解成一种更适合大模型推理训练的策略优化方法。它保留了 PPO 那种策略优化的框架，但把优势估计改成了组内相对比较，降低了 value 建模成本，也更贴合 reasoning task 的训练特点。** 

## SFT 和 RL 能使用一套数据集吗

**能用同一批原始数据，但数据格式不一样，不能直接共用。**

**SFT 的数据格式：**

每条数据是一个 (prompt, response) 对，只有一个标准答案。模型学的是"看到这个问题就生成这个回答"，是监督学习的范式。

**RL（RLHF）的数据格式：**

每条数据是一个 prompt 对应多个 response，加上人类的偏好排序。比如同一个问题有回答 A、B、C，人类标注 A > B > C。模型学的不是"正确答案是什么"，而是"什么样的回答更好"。

**所以区别在于：**

SFT 需要的是绝对标准——这个回答是对的。RL 需要的是相对偏好——这个回答比那个好。同一批原始问题可以复用，但标注方式完全不同。

**实际操作中的常见做法：**

先用一批高质量 (prompt, response) 数据做 SFT。然后拿同样的 prompt，让 SFT 之后的模型生成多个不同的 response，再让人类标注员做偏好排序，生成 RL 阶段的训练数据。所以 prompt 是复用的，但 response 部分是重新生成和标注的。

**能不能偷懒把 SFT 数据直接转成 RL 数据：**

理论上可以，把 SFT 的标准答案当作 chosen，让模型随机生成一个差的当作 rejected，就构造出了偏好对。DPO 里经常这么干。但这样数据质量不如真实人类标注，因为 rejected 是人为构造的，不一定代表模型真实会犯的错误。

## AI+脚本做归因报告分析

**第一层：定时触发层**

用 Airflow 做定时调度，比如每周一早上 8 点触发"周报任务"。触发时会带上任务类型和时间范围等参数，传给下面的 Agent。

**第二层：Orchestrator Agent（唯一用 LLM 的地方）**

这个 Agent 用内部模型驱动，职责严格限定为三件事：

第一是任务分解。接到"生成上周归因周报"这个指令后，把它拆成一系列 Tool 调用：先拉数据、再跑归因模型、再跑异常检测、最后生成图表。

第二是流程编排。决定调用顺序、传递参数、处理中间结果的衔接。

第三是报告撰写。把所有脚本返回的结构化 JSON 翻译成自然语言报告。

关键约束：Agent 的 System Prompt 里写死了硬规则——不能编造数字、不能自己做数学运算、每个数据点必须标注来源 Tool 和调用 ID。如果某个数据没查到，必须写"该数据未查询"，不能猜。

**第三层：固定脚本层（Tool Pool）**

这是保证零幻觉的核心，所有计算都封装成确定性脚本，以 Function Call / Tool 的形式暴露给 Agent。

每个脚本都有严格的 JSON Schema 定义输入输出，经过单元测试和代码审查，版本管理。

**第四层：数据访问层**

脚本通过这一层访问埋点数据仓库，核心设计是**参数化 SQL 模板**。我们不让 LLM 写 SQL，因为 LLM 生成的 SQL 可能语法正确但语义错误，比如 JOIN 条件写反、WHERE 漏条件，这种错误在归因场景下极难发现。取而代之的做法是：预先写好 SQL 模板，注册到模板库里，脚本只能调用"模板 ID + 参数"，查询的逻辑结构由人类保证正确。数据库权限设为 Read-Only。

**第五层：验证层（Fact-Checker）**

在报告发布前做最后一道防线。这个模块是**确定性代码，不是 LLM**。它的工作流程是：解析 Agent 生成的报告，用正则或结构化方式抽取所有数值断言（比如"渠道 A 贡献 35%"），然后逐条与脚本原始输出做比对。全部吻合才放行，有一条不一致就打回重生成或标红告警。

## 很多个不同类别的agent持续运行、每天产生几千条轨迹，需要每天报告，设计监控

**分三层设计：**

**第一层：实时指标采集**

每条轨迹记录结构化的元数据：agent_type、task_id、开始结束时间、总步数、每步的工具调用和返回状态、token 消耗、最终状态（成功/失败/超时）。这些是硬指标，不需要 LLM 参与，纯规则就能采集。存到时序数据库（比如 Prometheus）或者直接写日志系统（ELK）。

**第二层：统计异常检测**

对每类 Agent 建立基线指标——平均成功率、平均步数、平均延迟、平均 token 消耗。每天和基线做对比，偏差超过阈值就报警。比如某类 Agent 平时成功率 95%，今天掉到 70%，触发告警。可以用简单的统计方法，比如滑动窗口均值加标准差，不需要复杂模型。按 Agent 类别分组统计很关键，因为不同类别的正常基线差异很大，混在一起看会掩盖问题。

**第三层：质量抽样评估**

硬指标发现不了的问题，比如 Agent 成功返回了但内容有 hallucination，或者推理路径不合理。每天从每类 Agent 的轨迹里按比例抽样，用 LLM-as-Judge 做质量评估——推理链是否连贯、结论是否有依据、是否偏离任务目标。这一层成本高，所以只做抽样，重点抽失败的和边界 case。

**每日报告怎么生成：**

汇总三层的数据，按 Agent 类别出一份报告。每类 Agent 包含：成功率及趋势、失败 top 原因分类（工具报错、超时、质量不合格）、和昨天/上周基线的对比、异常轨迹的典型案例。报告本身可以用 LLM 生成自然语言摘要，但底层数据必须是结构化的指标，不能纯靠 LLM 去读几千条轨迹。

**还有一个容易忽略的点：错误模式聚类。**

几千条轨迹里可能有几十种不同的失败原因，逐条看不现实。把失败轨迹的错误信息做 embedding 然后聚类，每个聚类代表一种失败模式。报告里只展示 top N 的失败模式及其频率和典型案例，这样运维人员能快速定位系统性问题而不是淹没在个案里。

## 设计一个智能导购助手 Agent？描述其感知、规划、记忆和执行四大模块在分布式架构下的协同逻辑。

**四大模块的设计：**

**一、感知模块（Perception Service）**

独立部署为一个微服务，负责理解用户输入。做三件事：意图识别（浏览、比价、咨询、投诉）、实体抽取（品类、品牌、预算、使用场景）、情绪检测（满意、犹豫、不满）。可以用轻量分类模型做意图识别，用 NER 模型做实体抽取，不需要每次都调大模型，省成本。输出一个结构化的理解结果，发到消息队列（比如 Kafka）供下游消费。

多模态场景下还要处理图片输入——用户拍一张鞋的照片说"我想要类似的"，需要接图像检索服务。

**二、记忆模块（Memory Service）**

独立的存储服务，分三层。短期记忆用 Redis 存当前会话状态——聊了几轮、已经看过哪些商品、表达了什么偏好，TTL 设成会话超时时间。长期记忆用向量库加关系型数据库存用户画像——历史购买记录、品牌偏好、尺码信息、价格敏感度。工作记忆也在 Redis 里，存当前任务的中间状态——已经检索了哪些商品、比价结果、候选推荐列表。

感知模块每次处理完都更新短期记忆，规划模块每次决策前都读记忆。关键设计是记忆的读写要异步，不能阻塞主链路。

**三、规划模块（Planning Service）**

Agent 的大脑，调用 LLM 做决策。拿到感知结果和记忆上下文后，决定下一步执行什么动作。这里有两种模式：

简单场景用规则路由——用户明确说"查一下这双鞋的价格"，直接路由到商品查询，不需要 LLM。

复杂场景用 LLM 规划——用户说"帮我挑一双适合马拉松的鞋，最好耐磨"，LLM 分解成：检索跑鞋 → 筛选马拉松适用 → 过滤耐磨属性 → 按预算排序 → 生成推荐理由。

规划模块输出的是一个执行计划（action list），推到任务队列里让执行模块消费。

**四、执行模块（Action Service）**

一组独立的工具微服务，每个对应一个能力：商品检索服务（对接商品库 + 向量检索做语义搜索）、比价服务（跨平台价格对比）、优惠计算服务（满减、券、会员折扣）、库存查询服务、订单服务（加购、下单引导）。

每个工具服务独立部署、独立扩缩容。执行模块从任务队列拿到 action，调对应的工具服务，拿到结果后写回工作记忆。如果某个工具服务挂了或超时，执行模块做降级处理——比如比价服务挂了就跳过比价直接推荐。

**分布式协同的完整链路：**

用户输入 → 感知服务解析意图和实体 → 写入消息队列 → 规划服务消费，读取记忆上下文，生成执行计划 → 执行计划推入任务队列 → 执行服务消费，并行调用多个工具服务 → 结果写回工作记忆 → 规划服务检查是否完成，未完成则继续规划下一步 → 完成后生成最终回复返回用户。

**几个关键的分布式设计考量：**

服务间通过消息队列解耦，任何一个模块挂了不会拖垮整条链路。规划和执行是异步的，执行模块可以并行调多个工具。需要一个全局的 trace ID 串联整个请求链路，方便排查问题。LLM 调用是延迟瓶颈，规划模块要做缓存——相似的决策场景缓存执行计划，不用每次都调 LLM。

## 在电商或导购场景下，用户的请求往往高度模糊，Agent 怎么来精准理解这种需求？

**第一层：意图和实体的结构化抽取**

用户说"想买个礼物"，这几乎没有可执行的信息。但可以先抽取出有的东西：意图是"购买"，场景是"送礼"。缺失的信息也很关键——没有品类、没有预算、没有送礼对象。把已知和缺失都结构化出来，为下一步决策做准备。

**第二层：记忆补全**

在主动追问之前，先看记忆里有没有能用的信息。长期记忆里可能有：这个用户上个月买过女士护肤品、历史客单价在300左右、之前买礼物偏好包装精美的。这些信息能直接补全一部分缺失字段，减少追问轮次。用户体验上，不追问就能猜对是最好的体验。

**第三层：主动澄清，但要聪明地问**

缺失信息太多不能一次全问，会吓跑用户。策略是按信息增益排序，先问最能缩小搜索空间的那个问题。比如"送礼对象是男士还是女士"这一个问题就能砍掉一半商品空间，比问"预算多少"更优先。而且问的方式要带选项而不是开放式——"是送给女朋友、家人、还是同事？"比"你想送给谁？"转化率高得多。控制在1-2轮追问内收敛，超过3轮用户就流失了。

**第四层：试探性推荐**

有时候不问比问好。先根据已有信息推几个方向性的选项——"猜你可能想看这几类：护肤礼盒、鲜花、数码小物件"，让用户点选或者否定。用户的反馈（点了什么、说了"不要这种"）本身就是在表达偏好，比直接追问更自然。本质上是把澄清过程包装成推荐过程，降低用户的认知负担。

**第五层：隐式信号利用**

不只看用户说了什么，还看用户做了什么。浏览记录（看了10分钟运动鞋）、点击行为（反复点同一个价格区间）、停留时长、加购又删除。这些行为信号能推断出文字里没表达的偏好。比如用户嘴上说"随便看看"，但行为上一直在看500-800价位的跑鞋，那就按这个区间推荐。

**第六层：对话过程中持续修正**

理解不是一次性的，是整个对话过程中不断修正的。用户说"便宜点的"，这是相对概念，需要结合已经推荐过的商品价格来理解。如果刚推了800的鞋用户说便宜点，大概是400-600区间；如果推的是200的用户说便宜点，可能是100以内。每一轮交互都更新用户画像的置信度。
