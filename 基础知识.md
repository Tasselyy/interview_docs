# python

## **Q：Python中list、tuple、set、dict的区别是什么？分别适用于什么场景？**

A：简单来说，list是有序可变的，想加就加想删就删，用得最多，比如存一组井的测深数据；tuple也是有序的，但创建之后就不能改了，适合当字典的key或者保护数据不被误改；set是无序不重复的集合，去重特别方便，比如我想知道两口井共同经过了哪些地层，用集合求交集就行；dict是键值对映射，查找是O(1)的，适合快速检索，比如用井号查井的信息。

## **Q：说一下Python的装饰器（decorator）原理和常见用法。**

A：装饰器说白了就是一个函数，它接收一个函数作为参数，然后返回一个新函数。用@语法糖来用，写起来很优雅。

最常见的场景有这些：写日志记录、做权限验证、加缓存（比如@lru_cache）、重试机制、还有计时——我之前就用装饰器给函数加过耗时统计，debug性能问题特别好用。

## **Q：解释生成器（generator）和迭代器（iterator）的区别。**

A：迭代器是实现了`__iter__`和`__next__`方法的对象；生成器是一种特殊的迭代器，用yield关键字创建，每次yield会暂停执行并保存当前状态，下次调用再继续。

生成器最大的好处是惰性求值，不会一次把所有数据加载到内存。比如我要处理一个几个GB的井轨迹数据文件，用生成器逐行读取就不会爆内存，而list会一次全读进来。

还有生成器表达式 `(x**2 for x in range(10))`，跟列表推导长得像，但不占内存。

# RAG

## 什么是RAG

简单来说，RAG就是就是给LLM配了一个外挂的知识库

是为解决LLM的两个核心缺陷：

1. 容易产生幻觉
2. 知识库过时，并且无法直接访问私有数据

标准的RAG一般分为三个核心阶段：

1. **数据入库：**把私有数据（pdf，word）解析，切分成文本块，然后用embedding模型转化成高维向量，存入Milvus，Faiss这种向量数据库
2. **检索召回：**当用户提问的时候，把用户的问题也变成向量，然后去向量数据库中进行相似度计算，把最相关的TOP-K文档片段召回
3. **增强生成：**把召回的文档作为上下文背景，和用户的原始问题一起喂给LLM。

## RAG和SFT的区别，取舍

RAG：成本低，更新简单，但是响应慢，依赖检索质量

SFT：成本高，更新困难，依赖训练数据，但是响应快

RAG场景：实时/高度动态的数据，需要极高的事实准确度和可溯源性

SFT场景：想要模型严格遵循特定格式输出，或者是需要学特定场景下的 Function Calling 范式

## RAGAS

核心理念是 LLM-as-a-judge

分为两个模块：

1. 评估generator的质量-LLM回答的好不好：
   * 忠实度：回答是不是百分百来自于检索到的上下文
   * 回答相关性：回答和问题是不是高度相关的
2. 评估Retriever的质量：在向量库中找的准不准：
   * 上下文准确度：TOP-K的质量，也就是看是不是把最相关的排在最前面
   * 上下文召回率：需要提供Ground Truth，评价检索到的上下文是否包含了回答问题所需要的必要信息

## LangGraph

### 流式处理

`stream_mode="updates"` 或 `"values"`

这两种模式用于跟踪多智能体图的流转过程。

- `"updates"`：每次某个节点（Agent）执行完毕，只返回该节点对全局状态（State）做的**增量修改**。
- `"values"`：每次节点执行完，返回**完整的全局状态**。

`stream_mode="messages"`用于 Token 逐字输出

- 在这个模式下，LangGraph 将内部节点生成的每一个 Token 作为 `Chunk` 抛出来。

### LangGraph 和动态逻辑链的核心区别，底层原理

**LangGraph** 的核心是一个**可循环的状态机**。它用图的节点和边来**显式定义**工作流，侧重于**流程的绝对可控性**。

LangGraph 的底层维护着一个全局的 Pydantic  State。在图的每一次流转中，在各个节点之间传递。

**Nodes：** 通常是一个 Python 函数或一个 Agent。它接收当前的 State，执行逻辑（比如调用 LLM 或工具），然后返回对 State 的更新（通过 Reducer 机制合并数据）。

**Edges：** 决定下一步去哪个节点。条件边根据 LLM 的输出动态决定是继续向下走，还是循环回去重做。

**底层运行机制：** 它本质上是一个**基于图的迭代器**。系统在后台维护着一个执行栈和Checkpoints，每次节点执行完毕后，框架会评估边逻辑，如果是条件边，则调用路由函数决定下一个入栈的节点。

## openclaw

### OpenClaw 是什么?

OpenClaw 是一个开源的、可自托管的个人AI助手平台。本质上是一个事件驱动的Agent执行引擎，前面放了一个多渠道网关。用TypeScript编写，运行在你自己的机器上，所有数据和密钥不离开本地。

### OpenClaw 的架构是什么?

三层架构:触发层、网关层、Agent层。触发层有五种事件源(Messages、Heartbeats、Crons、Hooks、Webhooks)不断产生事件;网关层负责协议转换、认证和路由分发;Agent层负责推理(Think)、工具执行(Act)和记忆(Remember)。本质是一个事件驱动的执行管道:事件产生网关路由Agent执行状态持久化。

### Gateway 作用是什么?

Gateway是一个本地长驻的Node.js进程，监听单一端口(默认18789)，同时提供HTTP和WebSocket。它是触发层和Agent层之间的中间件，负责渠道连接管理、协议转换、会话路由、认证鉴权、插件加载和设备配对。关键边界是:Gateway不做推理、不调LLM、不执行工具，只负责"消息到了该给谁"和"回复该送回哪里"。

### 有哪些消息源/触发源?


五种:
a.Messages一用户通过聊天渠道发消息，事件驱动
b.Heartbeats一固定间隔定时器(如每15分钟)，让Agent 主动唤醒
c.Crons一用户定义的计划任务(cron表达式)，执行具体的定时工作
d.Hooks一Gateway/Agent 内部生命周期事件触发的回调
e.Webhooks一外部系统通过HTTP POST 触发加上Agents可以派生 Subagents，形成递归执行。

### 为什么能接入各个平台?

因为Gateway 层的Channel Manager 把每个渠道封装为一个独立的Channel Plugin，每个插件实现统一的接口。不同渠道的协议差异(Telegram BotAPI、Discord WebSocket)被封装在各自插件内部，对上层Agent 暴露一致的消息接口。加一个新渠道只需写一个Channel Plugin，Agent代码零改动。目前支持20+个渠道。

### 长短期记忆如何组织的

短期记忆就是当前会话的对话上下文，包括用户消息、助手回复、工具调用结果这些内容，全部在ContextWindow 里以消息列表的形式存在。它的特点是容量受模型上下文窗口限制，比如 Claude是200K tokens,满了之后旧的对话就会被压缩成一段结构化摘要，只保留最近约20,000tokens的原始消息。所以短期记忆本质上是临时的、会被压缩的。

长期记忆则完全基于磁盘上的Markdown 文件，分成两种组织方式。第一种是MEMORY.md，这是一个策展式的长期记忆文件，存放的是用户偏好、持久性决策、重要事实这些需要长期保留的内容，由Agent主动维护和更新，每次会话启动时注入上下文，不受压缩影响。第二种是memory/YYYY-MM-DD.md，按日期组织的追加式日志，存的是当天的运行笔记、阶段性进展、临时但值得回顾的上下文，会话启动时只读入今天和昨天的内容。

长期记忆还配有一套检索索引机制。系统会对所有记忆文件建立混合搜索索引，包括向量语义搜索和 BM25关键词搜索，Agent 在需要的时候通过 memory_search 工具按需检索，不用把全部历史记忆都塞进上下文里。这样即使记忆文件越积越多，也不会撑爆上下文窗口。两者之间的衔接靠的是压缩前记忆冲刷:当短期记忆快满需要压缩时，系统会先静默提醒 Agent把当前对话中重要的内容写入长期记忆文件，然后再做压缩。这就保证了短期记忆中真正有价值的信息能沉淀到长期记忆里，不会因为压缩而彻底丢失。

### 四层记忆系统原理

第一层是 Bootstrap Files, 相当于硬盘上的件。包括 SOUL.md、AGENTS.md、USER.md、MEMORY.md等Markdown文件，存放的是 Agent 的人格设定、行为规则、用户身份和策展过的长期知识。它们每次会话启动时都从磁盘重新读取注入上下文，完全免疫压缩，是最稳定的一层记忆。不过有字符上限，单文件20,000字符，总计150,000字符，所以要保持精简。

第二层是Session Transcript，相当于文件系统日志。完整的对话历史用JSONL 格式逐条追加写入磁盘，包括用户消息、助手回复、工具调用和压缩摘要。它是跨重启持久的，继续会话时从JSONL重建上下文。但模型不是看全部记录，而是只看当前上下文窗口能装下的部分，更早的内容通过压缩变成摘要。

第三层是Context Window,相当于 RAM。就是模型每轮推理时实际能看到的所有内容，包括系统提示、BootstrapFiles、对话历史、工具返回结果都挤在这个固定大小的空间里。容量由模型决定，满了就触发Compaction。压缩时把最近约20,000 tokens 原样保留，更早的消息用LLM生成结构化摘要替代，摘要里必须保留任务状态、决策理由、TODO和关键标识符。压缩前还会静默提醒 Agent 把重要信息写入记忆文件，防止信息丢失。

第四层是Retrieval Index，相当于搜索引擎。系统对所有记忆文件建立混合搜索索引，同时使用向量语义搜索和BM25关键词搜索，按7:3 权重融合排序。Agent 通过 memory_search和memory_get 这两个工具按需检索，不需要把全部记忆都塞进上下文。检索是模型自主决策的，不是每轮自动触发。索引还支持时间衰减和去重，新笔记排名更靠前

这四层的协同逻辑是:Bootstrap Files提供稳定的身份和规则基座，Session Transcript 保存完整的历史记录，Context Window 管理当前可见的工作记忆并在满时通过压缩保持可用，Retrieval Index 让 Agent 能按需回忆历史而不占用上下文空间。关键设计原则是只有写入磁盘文件的信息才是真正持久的记忆，纯对话中没有持久化的内容压缩后就不可恢复了。

### 采用了什么Agent架构?

递归Centralized(中心化)架构。Parent Agent作为中心，工具创建子通过LLM的tool call机制调用sessions_spawnAgent执行子任务，子Agent完成后 push结果回父，父Agent等所有子任务完成后整合输出。子Agent 可以继续 spawn孙Agent，形成一棵 Orchestrator-Worker 树，每一层都是该层的中心

### 为什么要选择中心化架构?

根本原因是对话本身的结构是中心化的--有确定的入口(用户发消息)，有确定的出口(给用户一条完整回复)，中间的执行可以分发，但责任链不能断。如果换成去中心化(P2P协商)，最终谁来发回复没有确定答案;如果换成Independent(各自独立)，结果无法拼合成一条完整回复。Centralized 的父子结构正好对应这个"一进一出”的模型:Parent Agent 是唯一的回复责任人，子Agent只是它延伸执行能力的手段。

### Centralized系统性能受限?


1.减少 Parent 的工作量:Parent 只做编排和最终整合，不做具体执行。OpenClaw 在子Agent 的系统提示词里规范。子 Agent 不允许跑题、不允许主动发消息Parent 只需要等结果、做最终决策，每轮turn 的工作量被压到最低。

2.并行化子任务:Parent 可以同时 spawn多个子Agent 并行执行，不需要串行一个一个等。总耗时取决于最慢的那个子Agent，而不是所有子任务时间之和。Parent 在提示词里也被告知可以同时 spawn多个任务，依靠 push-based announce各自回来触发汇总。

3.并发数量限制:并行是加速手段，但无限并行会耗尽系统资源，所以OpenClaw 同时设置了两层上限作为反向约束:单个session的活跃子Agent 数（默认5），全局所有子 Agent 的并发数（默认8）。超出上限时‘sessions_spawn 直接返回‘forbidden',防止雪崩。

4.递归下推编排:子Agent 本身也可以变成 Orchestrator，继续向下 spawn 孙Agent，把编排压力分散到子层。Parent 只负责顶层任务拆解，不需要直接管所有叶子节点。

5.按角色分配模型:sessions_spawn的参数支持‘model和‘thinking 覆盖，可以给Parent 配置推理能力强的模型做决策，给执行型子Agent 配置更快更轻量的模型做具体任务，按角色分配算力而不是一刀切。

### Agent是如何通信的?

通信模式上是工具调用，控制权始终留在 Parent Agent。具体实现是 push-based异步消息:子Agent完成后通过调用 Gateway把结果以"用户消息”的形式投递到父Agent的消息队列，触发父Agent的下一轮LLM推理。父Agent不轮询、不阻塞线程，等消息触发即可，
传递内容上只传最终结果，子Agent的中间推理步骤全部留在私有 session里，不传给父Agent，有效控制上下文膨胀。



## harness

### 定义

Harness Engineering是围绕 AI 模型构建运行环境、调度机制和管控基础设施的一种工程实践。它的目标不是单纯让模型“更聪明”，而是让AI Agent在长时、复杂、多步骤的真实任务中，能够稳定、可靠、可控地持续运行。
如果用一句更通俗的话来说，Harness Engineering做的事情，就是不再把AI 当成一个一次性回答问题的黑盒，而是把它放进一个被设计好的工作系统里，让它按规则做事。
它背后的设计哲学也很重要:模型只是原材料，环境才决定结果能不能落地。
再强的模型，如果直接裸跑长任务，也会遇到上下文腐化、失忆、幻觉完成、任务漂移等问题。所以Harness Engineering的核心，不是继续往模型上堆能力，而是通过外部记忆、任务拆解、测试门禁、状态交接和人工审核，把AI的执行过程工程化。

### 解决问题

它主要解决的是 AI在长时任务里的可靠性问题。
第一个问题是ContextRot，也就是上下文腐化。模型做长任务时，上下文窗口会不断被历史对话、错误尝试和无关信息填满，最后导致它丧失对原始目标的把握。
第二个问题是Hallucinated Completion，也就是幻觉完成。模型在复杂任务中迷失后，往往不会老实说“我不会了"，而是会输出一个看起来完成、实际上并不正确的结果。
第三个问题是Model Drift，也就是模型漂移。多步执行中，模型会逐渐偏离最初目标，最后做出来的东西方向就歪了。
所以我理解Harness Engineering的价值，不是让模型在单轮回答里更强，而是让它在几十步、几百步的连续工作流里，依然能沿着正确轨道往前走。

### 和 Prompt ,Context 的关系是什么?

我会把它理解成一个演进关系，而不是替代关系。
Prompt Engineering主要解决的是“对AI说什么”，也就是如何设计指令，让单次输出更好。
Context Engineering进一步解决的是“让AI知道什么”，也就是在一次session或上下文窗口里，给它什么信息、给多少信息、怎么组织信息。
而Harness Engineering更进一步，它解决的是“让AI 在什么环境里做事”。
也就是说，Harness Engineering并没有取代前两者，而是把它们包进了一个更大的系统里。Harness内部的每个session还是要做Context Engineering，每一次给 agent 的具体任务描述还是要做Prompt Engineering。只是现在关注点从“优化一次对话”升级成了“设计一整个运行体系”。

### 如果让你用Harness工程来实现一个项目

从一个Initializer+Coding Loop的最小闭环开始，而不是一开始就做复杂的多agent 平台。第一步，我会先准备一份清晰的App Spec/PRD，把产品目标、功能范围和验收标准定义清楚。

第二步，我会实现一个Initializer Agent。它的职责不是写业务代码，而是读取需求文档，把大任务拆成一组细粒度、可执行、可验证的feature，并生成类似feature_list.json 这样的状态文件。同时它还会初始化项目骨架、代码仓库，以及后续循环执行所需的基础环境。

第三步，我会实现一个Coding Loop。每一轮都启动一个全新的coding agent，让它只读取当前状态文件、进度记录和代码库信息，只完成一个明确的小任务。任务完成后，不是让agent自己宣布成功，而是必须经过测试、lint 或类型检查等确定性验证;只有验证通过，才更新状态文件、写入交接记录并提交代码。然后这个agent立即退出，下一轮由新的agent接力。

第四步，我会把外部记忆做成系统核心，而不是把连续性寄托在模型上下文里。让每个新agent启动时都能快速接手，而不是依赖前一轮session的“记忆”。

第五步，我会在关键节点加入Human-in-the-loop。在高风险功能、关键界面变更或者多轮失败之后，我会设计人工确认断点，确保系统不会沿着错误方向持续推进。

## MCP

### 定义

MCP是一种标准化接口协议，统一LLM和外部资源的连接方式

用于让LLM能访问外部**数据，工具，提示词**

MCP采用的是服务端-客户端的架构，包括 client， host 和 server

host就是运行LLM的程序，client在host里面运行，负责和 MCP server通信

server中暴露**数据，工具，提示词**

## Skills

### 定义

skill 是对 Prompt，Tool Use，和 Workflow 的包装

是一个预定义的，可以复用的功能模块，用来让agent执行特定SOP任务

包含了：

1. SKILL.md: 包含了 Yaml 头和Body。Yaml 是md文件中的头，告诉AI是否加载这个SKILL，Al必定会加载。而body是正文，是渐进式披露的，只有AI读取了YAML头，认为这个SKILL应该被加载，才会加载这个正文部分。
2. Scripts: 可选。包含着在AI执行过程中需要使用的脚本文件。
3. Reference: 可选。存放执行SKILL相关的文档，说明，内容以MD为主。
4. Assets:可选。存放执行SKILL是需要的模板，静态资源，比如图片等。

### Skill vs MCP

1. MCP 侧重工具调用，Skills 侧重提示词；MCP 像工具箱，Skills 像带目录的说明书
2. MCP是基础服务层，Skill是在应用层
3. MCP的本质是一个独立运行的程序，更适合做安全设计，而Skills本质是一个文档。更适合处理轻量级的。
4. Skills的开发难度相比于MCP，开发难度低很多。

两者不是替代关系，而是**包含与承载**的关系。

* MCP Server可以对外暴露封装好的特定 Skill
* 也可以用 Skill 编排多个MCP Servers 的工作流

### Skill vs Prompt

Skills相比于Prompts的核心差别在于他们的形态和生命周期，Prompts是一次性对话内容，你需要自己保存，复制和粘贴。

模型不会知道什么时候用哪一个Prompt，而且你需要把Prompts 全部塞到上下文里面。这会大量占用你的上下文空间。

Skill则是将提示词以一种更加**工程化**的包装方式打包，他是一个文件夹的形式。他会动态加载/渐近式披露提示词，更加节省提示词。**容易复用和版本管理**，易于协作和共享。

总结来说，Skill 不是一个更复杂的提示词，而是把复杂的提示词，变成了一个有结构，可复用，可以自动调用的模块。

## Memory

第一层是**短期 memory**，靠最近几轮上下文和摘要维持对话连续性；第二层是**长期 memory**，把用户偏好、任务状态、关键事实存到外部存储中，再做检索增强；第三层是**memory policy**，也就是决定写入、更新、删除和遗忘的策略。传统系统大多做到前两层，但如果想把系统做得更稳，真正关键的是第三层。MemSkill 给我的启发就是，memory policy 不应该完全写死，而应该被学习、被评估、再持续演化。

### MemSkil

“我最近看了一篇 **MemSkill**，它研究的是 LLM Agent 的长期记忆管理。它的核心不是再用固定的 add、update、delete 这类手写规则去做 memory，而是把 memory operation 抽象成一个 **可学习、可进化的 skill bank**。具体来说，系统里有一个 **controller**，会根据当前的 text span 和检索到的已有 memory，选择一组最相关的 Top-K skills；然后一个 **LLM executor** 再基于这些 skills，一次性生成 skill-guided memory update。这个设计把传统的 turn-level、规则驱动 memory pipeline（langgraph short term memory），变成了 span-level、skill-conditioned 的 memory construction。更进一步，它还引入了一个 **designer**，会从训练里的 hard cases 里分析失败模式，去 refine 旧 skill，或者提出新 skill，所以它优化的不只是 memory 内容本身，而是‘如何构造 memory’这套策略空间。作者整体是用一个 closed-loop 的方式在做：一边用 RL 训练 controller 学会选 skill，一边让 designer 迭代 skill bank。我觉得这篇论文最有价值的点，就是它把 agent memory 从静态模块，推进成了一个可以自我演化的系统。” ([arxiv.org](https://arxiv.org/html/2602.02474v1))



从系统设计角度看，我觉得它最强的地方在于把 **memory policy** 和 **memory content** 分开建模了，前者由 skill bank 和 controller 负责，后者由 executor 负责，这样扩展性和泛化性都会更强。

## 性能评估

agent 最大问题往往不是不会调工具，而是不会做决策

agent 的评估不能只看最后成功率，是一个执行过程，所以我一般会看四件事

第一层是**结果层**。
 就是最后任务有没有完成，这个肯定要看。但我不会只看一个很粗的成功率，因为很多真实任务不是“做成了”这么简单，它往往有很多约束条件，比如时间对不对、预算有没有超、用户偏好有没有满足、工具调用结果是不是符合要求。所以我会把任务拆成更细的检查项，而不是只看最后对不对。

第二层是**过程层**。
 我会看 agent 在执行过程里的表现，比如它有没有先澄清问题，有没有做合理规划，工具调用顺序对不对，前后信息有没有对上。因为 agent 和普通问答不一样，它是一个长流程任务，很多问题其实不是出在最后结果上，而是出在中间某一步走偏了。所以我觉得评估不能只看 final result，还要看 trajectory，也就是整条执行轨迹。

第三层是**稳定性**。
 这个我觉得特别重要。因为有些 agent 不是不会做，而是有时候能做对，有时候又不行。那这种在真实业务里其实很危险。所以我会区分两件事：一种是“多跑几次，至少有一次做对”，另一种是“多跑几次，次次都做对”。前者说明它有能力，后者才说明它稳定、能落地。这个区别我觉得特别关键。

第四层是**按复杂度拆开评估**。

失败归因，要同时看**推理复杂度、工具复杂度、交互复杂度**

因为不同任务难点不一样。有的任务难在推理，也就是 reasoning；有的难在工具之间的依赖关系，也就是 tool use 不只是会不会调用，而是什么时候调、按什么顺序调；还有的难在交互，因为用户说得不清楚、会改需求、会漏信息，所以要看是否能识别出模糊意图，主动澄清。这三种复杂度其实要分开看，不然你只看总分，很难知道 agent 到底卡在哪。

VitaBench 对这个问题讲得挺好。它的一个核心观点就是，真实世界里的 agent 评估，不能只看 function calling 对不对，而是。而且它还专门区分了“偶尔成功”和“稳定成功”。这点我很认同，因为这比单纯看一个成功率更接近真实落地。

## Multi-Agent 架构

1. 先判断：这个任务到底适不适合 multi-agent

“我会先判断任务是不是真正 agentic。也就是它是不是需要持续和环境交互、是不是部分可观测、是不是要根据反馈不断调整策略。因为很多人以前在非 agentic benchmark 上测出来的 multi-agent 优势，放到真正 agentic 任务里并不一定成立

2. 再判断：任务能不能自然拆分

“我最先看的其实不是复杂度，而是能不能可拆解。Finance 这种能拆成并行子任务的场景，多 agent 就会好很多；但 像是强顺序、强状态依赖的任务，所有 multi-agent 架构都变差。所以真正决定要不要上 multi-agent 的，往往不是任务难不难，而是能不能拆。

3. 看 single-agent baseline 高不高

“我一般会先做一个 strong single-agent baseline。当 single-agent baseline 已经超过大概 45% 的时候，继续加 coordination 往往收益很小，甚至转负。也就是说，单 agent 已经做得不错时，多agent的沟通和管理成本可能比收益还大。

4. 最后再选架构

如果任务可分解，而且需要结果汇总和校验，我更偏向 centralized；如果任务更像搜索、探索、网页导航这种高熵空间并行探索，我会更偏向 decentralized，因为它可以实时交流，更新计划；如果任务本身强顺序依赖，我反而倾向先坚持 single-agent，而不是硬上 multi-agent。

# LLM 基础原理

## **Q：解释Transformer的核心架构，特别是Self-Attention机制。**

A：Transformer由Encoder和Decoder组成，核心是Self-Attention机制。

Self-Attention的直觉是：对于序列中的每个词，去"看看"其他所有词跟它的相关性有多大，然后加权汇总信息。

具体来说，输入会被映射成Q（查询）、K（键）、V（值）三个矩阵，然后算Attention(Q,K,V) = softmax(QKᵀ/√dₖ)V。除以√dₖ是为了防止点积太大导致softmax梯度消失。

Multi-Head Attention就是把注意力分成多个头，每个头关注不同的语义子空间，比如有的头关注语法关系，有的头关注语义关系，最后拼接起来。

还有一点很重要，Attention本身是无序的，所以需要位置编码（Positional Encoding）来注入序列顺序信息。

---

## **Q：GPT和BERT的区别是什么？**

A：最核心的区别是方向性。BERT是Encoder-only架构，用双向注意力，同时看左边和右边的上下文，擅长理解任务，比如分类、命名实体识别、阅读理解，预训练用的是掩码语言模型（随机遮住一些词让模型猜）。

GPT是Decoder-only架构，单向自回归的，只能看到左边的上下文，一个token一个token往右生成，擅长生成任务，预训练目标是预测下一个token。

现在的主流大模型比如GPT-4、Claude、LLaMA基本都是Decoder-only架构，因为这个架构在大规模上的规模效应更好。

---

## **Q：什么是Token？Tokenization的常见方法有哪些？**

A：Token是模型处理文本的最小单元，可以是一个字、一个词、或者一个子词片段。比如"unhappiness"可能被切成"un"+"happiness"。

常见方法有BPE（GPT系列用的）、WordPiece（BERT用的）、SentencePiece（LLaMA用的，对多语言支持好）。

BPE的原理是从字符级别开始，反复合并出现频率最高的相邻对，直到达到预设词表大小。这样常见的词会作为一个整体保留，罕见的词会被拆成子词。

---

## **Q：什么是大模型的"幻觉"（Hallucination）？如何缓解？**

A：幻觉就是模型一本正经地胡说八道——生成的内容看起来很合理，但实际上是错的或编造的。比如你问它某口井的数据，它可能编一个看起来像模像样但完全虚假的数字。

缓解方法我总结了几个：第一，用RAG，让模型基于检索到的真实数据回答，而不是凭空生成；第二，在Prompt里加约束，告诉模型"不确定的时候要说不知道"；第三，对关键输出做事实核查，比如生成的SQL先验证再执行；第四，降低temperature减少随机性；第五，在领域数据上微调提高准确性。

在油气领域这点特别重要，因为错误的数据可能导致严重的安全事故。

---

## **Q：解释一下Temperature和Top-p参数的作用。**

A：Temperature控制输出的随机性。原理是在softmax之前对logits除以T。T接近0的时候，概率分布会非常尖锐，模型几乎总是选概率最高的那个token，输出很确定；T越高分布越平缓，输出越多样、越有创意，但也越可能胡说。

Top-p是核采样，意思是只从累积概率达到p的最小token集合中采样。比如top_p=0.9表示把token按概率从高到低排列，累加到90%就截止，只在这些token里选。

实际用的时候，精确任务（写代码、查数据、做计算）用低temperature，创意任务（写文案、起名字）用高temperature。

# 后训练

## DPO

**DPO 和 PPO 的核心区别，我一般会这样理解：PPO 是传统强化学习方法，DPO 是一种更简单的偏好对齐方法。**

如果展开说的话，**PPO**需要先有一个 **reward model**，也就是先训练一个奖励模型，用它给模型输出打分，然后再用强化学习去更新大模型。所以它的流程通常是 **SFT → 奖励模型 → PPO优化**。
它的优点是比较通用，控制能力强，但缺点也很明显，就是训练链路比较长，工程复杂度高，而且训练容易不稳定。

而 **DPO** 的思路更直接。它不再单独训练 reward model，也不真的去跑完整的强化学习采样更新。它直接利用 **人类偏好数据**，也就是 chosen 和 rejected 这种成对数据，让模型直接学会“为什么这个回答比那个更好”。
所以你可以把 DPO 理解成：**把原来强化学习里的奖励学习过程，隐式地写进了一个更简单的优化目标里。**

所以如果面试官问我两者区别，我会总结成三点：

**第一，训练方式不同。**
PPO 是显式强化学习，要奖励模型；DPO 不是传统RL，更像是直接用偏好数据做优化。

**第二，复杂度不同。**
PPO 训练链路更长，调参更麻烦，显存和算力开销也更大；DPO 更轻量，训练更稳定，工程上更容易落地。

**第三，适用场景不同。**
如果你想要一个比较稳定、成本低的对齐方案，DPO 很常用；如果任务特别依赖长期回报，或者需要和环境多轮交互，PPO 这类RL方法会更合适。

**所以现在很多大模型对齐里，DPO 很受欢迎，就是因为它在效果还不错的前提下，大幅降低了训练复杂度。**

## on-policy 和 off-policy 的区别

**on-policy** 的意思是，
 我现在用哪个策略做决策，就用这个策略刚刚采样出来的数据来训练自己。
 也就是说，**采样策略和训练策略是同一个**。

比如 **PPO** 就是典型的 on-policy。
 模型先按当前策略生成一批回答，再根据这些回答的 reward 去更新自己。更新完之后，之前那批老数据基本就不能直接反复用了，因为它不是“新策略”产生的。

而 **off-policy** 是，
 训练时可以使用**不是当前策略产生的数据**。
 这些数据可以来自旧策略，也可以来自经验回放池，甚至可以是别的策略收集的。
 所以 **采样策略和训练策略可以不一样**。

## GRPO

**对于同一个 prompt，GRPO 会先让模型生成一组 outputs。**
 然后根据 reward，比如答案对不对、格式对不对、推理是否满足要求，给这一组回答打分。接着它不是直接看某个回答的绝对分数，而是看它**相对于组内其他回答**的表现。表现高于组平均的，就是正向更新；低于组平均的，就负向更新。
 所以它学到的其实是：**同样一道题，哪些回答比其他回答更值得保留。** 

它和 PPO 的一个很大区别是：

**PPO 通常要有 value model，GRPO把这部分弱化或者省掉了。**
 这样做的好处是训练链路更简单，显存开销也更低，特别适合长文本生成和 reasoning 场景。也正因为这个特点，GRPO 后来在很多 reasoning model 训练里都很受关注。

**因为 reasoning 任务里，很多题很难直接定义特别精确的逐步价值函数，但很容易比较“这几个回答里哪个更好”。**
 GRPO 就利用了这种“相对比较”信号，把学习问题从“估计绝对价值”变成了“做组内排序”，这通常更稳定，也更符合大模型生成多个 candidate 再筛选的实际过程。

**所以我会把 GRPO 理解成一种更适合大模型推理训练的策略优化方法。它保留了 PPO 那种策略优化的框架，但把优势估计改成了组内相对比较，降低了 value 建模成本，也更贴合 reasoning task 的训练特点。** 

# 场景题

## AI+脚本做归因报告分析

**第一层：定时触发层**

用 Airflow 做定时调度，比如每周一早上 8 点触发"周报任务"。触发时会带上任务类型和时间范围等参数，传给下面的 Agent。

**第二层：Orchestrator Agent（唯一用 LLM 的地方）**

这个 Agent 用内部模型驱动，职责严格限定为三件事：

第一是任务分解。接到"生成上周归因周报"这个指令后，把它拆成一系列 Tool 调用：先拉数据、再跑归因模型、再跑异常检测、最后生成图表。

第二是流程编排。决定调用顺序、传递参数、处理中间结果的衔接。

第三是报告撰写。把所有脚本返回的结构化 JSON 翻译成自然语言报告。

关键约束：Agent 的 System Prompt 里写死了硬规则——不能编造数字、不能自己做数学运算、每个数据点必须标注来源 Tool 和调用 ID。如果某个数据没查到，必须写"该数据未查询"，不能猜。

**第三层：固定脚本层（Tool Pool）**

这是保证零幻觉的核心，所有计算都封装成确定性脚本，以 Function Call / Tool 的形式暴露给 Agent。

每个脚本都有严格的 JSON Schema 定义输入输出，经过单元测试和代码审查，版本管理。

**第四层：数据访问层**

脚本通过这一层访问埋点数据仓库，核心设计是**参数化 SQL 模板**。我们不让 LLM 写 SQL，因为 LLM 生成的 SQL 可能语法正确但语义错误，比如 JOIN 条件写反、WHERE 漏条件，这种错误在归因场景下极难发现。取而代之的做法是：预先写好 SQL 模板，注册到模板库里，脚本只能调用"模板 ID + 参数"，查询的逻辑结构由人类保证正确。数据库权限设为 Read-Only。

**第五层：验证层（Fact-Checker）**

在报告发布前做最后一道防线。这个模块是**确定性代码，不是 LLM**。它的工作流程是：解析 Agent 生成的报告，用正则或结构化方式抽取所有数值断言（比如"渠道 A 贡献 35%"），然后逐条与脚本原始输出做比对。全部吻合才放行，有一条不一致就打回重生成或标红告警。
