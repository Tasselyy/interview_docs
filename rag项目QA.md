

# 你们项目里 Agent 是怎么用工具的？

我们工具主要来自 MCP 的 RAG Server，并且用一个ToolWrapper 统一包一层，做参数归一化、结果解析和错误兜底。

在整体上是编排式的调用工具：没有把工具绑给 LLM，LLM 只产出结构化结果（query、plan、action），**什么时候、用什么参数调 RAG 都由策略代码决定**。

具体来说我们先做意图识别，再路由到不同策略节点。所以真正调 RAG 工具的是各策略节点里的代码：比如说Simple 调一次 search 再合成，Comparative 按实体并行多几次 search。

这和最近berkeley的论文 Measuring Agents in Production 里面说的生产实践是一致的，论文中表明工业界在 Agent 的开发上极度偏好简单性和可控性，常常故意牺牲一定程度的自主性来换取稳定性

我们项目中按意图分组、编排式调工具、用工作流和步数上限做系统级约束，和论文里「有界限的自主性」是一致的。

## Prompt结构怎么设计的

第一层是模板层，按功能拆分，都是独立文件。这样后面调 prompt 的时候，不会影响具体业务流程。

第二层是执行层，这一层负责当前需要的数据，做当前这一步的事。负责把 state 里的结构化信息注入模板，再调用 LLM。

第三层其实就是数据层，或者说状态层。我的做法是用一个统一的 state 来保存流程中间结果，这样每一步都是从统一字段里取数据、再写回统一字段

第四层是编排层，整个流程挂在 LangGraph 上

比较重要的设计点。

第一，尽量用结构化输出

第二，按推理模式拆 prompt，不同策略用不同模板

# 简述

主要是面向学术论文阅读和研究问答场景做的一个 Agentic RAG 系统。架构上 Agent 的推理流程和底层检索能力做了解耦：上层 Agent 这边，我用 LangGraph 搭了一个有向状态图，让系统先做意图识别，再根据问题类型动态路由到不同的推理路径，比如直接问答、对比分析、多跳检索或者探索式推理；下层 RAG 这边，我把检索能力做成模块化服务，通过 MCP 协议和 Agent 做标准化对接，这样后续更换检索后端或者模型都比较方便。

Agent端主要是 Router 和 Critic 这两个模块。Router 这边我基于 Qwen2.5 做了 LoRA 微调，用来做意图分类；Critic 这边我训练了一个偏好对齐后的评分模型，对生成结果从完整性、忠实性这些维度做打分，如果分数不够，结构化错误归因，再做针对性的改善。所以整个系统不是一次生成，而是带自我修正能力的。这样一方面回答质量更稳，另一方面也能把昂贵的大模型调用控制在更必要的环节。

另外在 RAG 侧，我实现了 Dense + Sparse 的 Hybrid Search，用向量召回和 BM25 结合，再通过 RRF 和 rerank 做精排，兼顾语义理解和关键词匹配。数据侧也做了完整摄入流程，包括 文档解析，数据清洗，多模态图片表格解析、父子文档切块，和幂等去重更新。

最后为了方便训练和部署，我还做了AWQ 量化和 vLLM 多 Adapter 部署，能够让同一个量化基座同时服务 Router 和 Critic。整体来说，这个项目让我比较系统地把 Agent 编排、RAG 检索、模型训练和线上部署串起来了。

# 系统架构

## 为什么选择7b不是32b

> Router 和 Critic 是**分类和打分任务**，不是开放式生成，7B 足够且性价比最优。

**1. 任务复杂度不需要 32B**

> Router 本质上是一个 5 分类问题（factual / comparative / multi_hop / exploratory / follow_up），输入是一句用户问题，输出就是一个 label + confidence。这和通用对话或复杂推理完全不同，7B + LoRA 微调后准确率已经很高了。
>
> Critic 也类似，输入是 question + answer + contexts，输出是 score / completeness / faithfulness 三个数值加一句 feedback。这是一个评分/打标任务，不需要模型具备强推理能力。

**2. 部署约束**

|                 | 7B AWQ INT4       | 32B AWQ INT4       |
| --------------- | ----------------- | ------------------ |
| 模型权重显存    | ~4 GB             | ~18 GB             |
| 剩余给 KV Cache | ~18 GB (RTX 4090) | ~4 GB              |
| 并发能力        | 高（几十个并发）  | 很低（几个就满了） |
| 单请求延迟      | ~30-60ms          | ~150-300ms         |

> 我们的 vLLM 同时 serve 两个 LoRA（router-lora + critic-lora），共享一个基座模型。7B AWQ 只占 4GB，剩下的显存全给 KV Cache，并发能力很强。换 32B 的话，光模型权重就吃掉 18GB，一张 4090 基本跑不动并发了。

**3. 架构分工**

> 我们的设计是**模型分层**：
>
> - 本地 7B：只做 Router 分类 + Critic 打分（轻量、高频、低延迟）
> - 云端 gpt-4o-mini：做 slot filling、策略推理、答案合成（需要强生成能力）
>
> 重活交给云端大模型，本地模型只负责它擅长的判别任务。如果 Router 也用 32B，那和直接调云 API 比就没有延迟优势了，违背了本地部署的初衷。

**4. 训练成本**

> 7B 用单卡就能 LoRA 微调，batch_size 开到 16 都没压力。32B 即使用 LoRA + INT4，单卡训练 batch_size 也只能开到 1-2，gradient accumulation 拉长训练时间，迭代效率低很多。我们需要快速实验不同的训练策略（SFT → DPO），7B 可以一两个小时跑完一轮，32B 可能要半天。

**Q：如果 7B 分类准确率不够呢？**

> 我们优先调数据和训练策略，而不是换更大模型。比如 Router 数据增强、hard negative mining、调 LoRA rank。实际测下来 7B + LoRA rank=16 在五分类任务上准确率已经够用了。

**Q：有没有考虑过 1.5B 这种更小的模型？**

> 考虑过。对 Router 这种纯分类来说 1.5B 可能也够。但 Critic 需要理解 question + answer + contexts 的语义关系来打分，1.5B 在这种多文本理解任务上退化比较明显。用同一个 7B 基座同时 serve 两个 LoRA 是当前的最优平衡——一份权重显存，两个任务共用。

## LangGraph State设计

tate 设计的核心思路是**一个 Pydantic Model 贯穿全图，每个节点只写自己负责的字段**。

有两个关键设计：一是 **reducer 区分追加和覆盖**——messages 和 reasoning_trace 是追加的，不会被后续节点覆盖；其他字段是覆盖的，支持 retry 时重写 question 和 draft_answer。

二是 **Intent 分两步填充**——Router 只做分类（轻量，可以用本地 LoRA），Slot Filling 做槽位提取（重活，用 Cloud LLM），这样可以灵活搭配不同的模型部署策略。

控制流变量（retry_count、max_retries）不参与业务逻辑，纯粹驱动条件边的路由决策，实现了 Reflexion 循环：Critic 不通过 → 带着 feedback 改写 question → 重新走一遍 pipeline。



# RAG 简述

先经过 agent 侧把问题整理成更适合检索的结构化查询，比如补全关键词这些信息。然后检索层会基于这个结构化查询，同时走两条召回链路，一条是 Milvus 做语义召回，另一条是 BM25 做关键词召回，而且两边在召回阶段就会结合元数据过滤，把范围先限制住。召回结果出来之后，我们用 RRF 做融合，再用 CrossEncoder 做精排，把最相关的内容排到前面。最后生成阶段不会直接把零散片段塞给模型，而是会按父子分块关系补一些章节上下文，再交给大模型生成答案。整个系统前面用 Marker 做论文解析和父子分块，PostgreSQL 管结构化元数据，Milvus 管向量检索，BM25 管关键词检索

- PDF 解析：Marker
- 分块：父子分块
- 元数据存储：PostgreSQL
- 向量检索：Milvus
- 关键词检索：BM25
- 排序：RRF 融合 + CrossEncoder 精排

| 内容类别                   | 存储位置     | 作用                                                         |
| :------------------------- | :----------- | :----------------------------------------------------------- |
| 原始论文文件与解析中间产物 | 本地文件系统 | 存 PDF、解析后的 markdown/json、图片等中间产物，方便调试和重处理 |
| 文档元数据与结构化信息     | PostgreSQL   | 存文档信息、父子块关系、章节路径、页码、版本、状态等         |
| 向量索引                   | Milvus       | 存分块后的向量，用于语义召回                                 |
| 关键词索引                 | BM25 索引库  | 存分块文本的关键词索引，用于关键词召回                       |

# RAG流程

## 摄入

论文 PDF 先经过结构化解析，然后做清洗和标准化，再做父子分块；分块后的内容写入 PostgreSQL 作为元数据和内容管理层，同时写入 Milvus 建立向量索引，并构建一套 BM25 关键词索引。查询阶段，系统会同时走向量召回和关键词召回两条链路，之后用 RRF 做融合，再用 CrossEncoder 做精排，最后把结果按论文结构组织成上下文，交给大模型生成答案。

先说**摄入阶段**。先拆分页数过长PDF，Pipeline 入口有文档质量检测的前处理（PyMuPDF）。文档进入系统后，不会直接无脑解析，而是会先做一轮质量预检:提取前几页文本，计算有效字符占比和可识别文本密度。如果有效字符率低于阈值，说明这个PDF的文本层本身就有严重问题，索质量根本没法保证。所以直接拒绝入库，在Dashboard上给用户一个明确的提示: 该文档质量不达标，请检查后重新上传。与其让脏数据污染整个知识库，不如在入口就拦住。

**不是所有低质量文档都应该直接拒绝。** 可以分两档处理——有效字符率低于80%直接拒绝，在50%到80%之间的可以给用户一个警告，同时尝试用Marker的`--force_ocr`模式强制做一遍OCR再重新检测。因为有些PDF可能是文本层部分损坏但内容其实是可恢复的，OCR一遍就能救回来

解析完成以后，我会做一层**清洗和标准化**。这一层主要通过**规则方法**去掉页眉、页脚、页码

接下来是**分块**，采用父子分块。父块按照章节层级来切，比如摘要、引言、方法、实验、结论；子块则在父块内部按段落**递归切分**。

在分块时，我会给每个块保留比较完整的**元数据**，比如文档编号、块编号、父块编号、标题路径、页码范围、章节类型等。

然后是**存储层**。PostgreSQL 主要负责存文档级信息、父块信息、块和父块之间的映射关系、摄入状态、版本号以及一些必要的业务字段。选择 PostgreSQL 的原因是，它很适合管理这类结构化信息，事务能力强，查询能力稳定，也方便后面做版本治理和任务追踪。

向量召回这一层我会用 Milvus。Milvus 主要负责存子块的向量和必要的元数据，用来做语义检索。关键词召回我会保留 BM25。

整体架构尽量收敛。PostgreSQL 负责结构化元数据，Milvus 负责语义检索，BM25 负责关键词召回

### Marker

Marker是一个开源的PDF解析工具，能把PDF转换成Markdown或JSON格式，速度快、准确率高 [GitHub](https://github.com/datalab-to/marker)。它跟你Paper Pilot项目里的数据摄入流水线直接相关——你简历上写了"PDF语义分块、多模态图片解析"，Marker就是干这个事的。

**核心能力：** 它能保留表格、数学公式、代码块和图片，同时去掉页眉页脚这些干扰信息。对学术论文来说特别有用，因为论文里大量的公式、表格、引用结构，普通的文本提取工具（比如pdftotext）处理不好，要么版面错乱要么公式丢失。

**技术原理：** 它底层用的是Surya模型做OCR和版面识别，不是纯规则解析，而是基于视觉模型来理解PDF的布局结构——哪里是标题、哪里是正文、哪里是表格、哪里是图片，先做版面分析再分区域提取内容。还支持用大模型辅助提升精度，比如传一个`--use_llm`参数，可以用Gemini或者Ollama模型来处理跨页表格合并、行内公式格式化这些难搞的case

**性能：** 一篇论文通常就十几页，本地GPU跑Marker几秒钟就能解析完，在H100上批量处理的吞吐量大约是每秒25页，

### 为什么选 Marker

- **GROBID** 在标准期刊格式上结构化做得不错，但它走的是 CRF + 规则路线，对 arxiv 预印本、双栏转单栏、非标准模板的论文容错能力明显下降，而且输出是 TEI XML，对接成本比较高。
- **Nougat** 是端到端视觉模型，在公式和表格上有优势，但推理速度慢（一篇 20 页论文要 2-3 分钟），并且有幻觉问题——偶尔会生成原文里根本不存在的文本，这在 RAG 场景里是不可接受的。
- **PyMuPDF** 是纯文本提取器，不保留 block type、不识别标题层级、不做表格检测，只能当最底层的 fallback。

### Marker 的 JSON 输出里，block 的层级关系是天然正确的吗？还是你要自己重建？

不是天然正确的。Marker 输出的层级大部分情况可用，但在以下几种情况需要二次修正：

1. **标题层级错乱**——比如某些论文的 subsection 标题用的字号和 section 标题一样大，Marker 会把它们识别成同一级，后处理里我用了一套基于序号模式匹配（如 "3.1", "3.1.2"）加字体大小差异的启发式规则来修正层级。
2. **附录和参考文献部分层级丢失**——Marker 对正文结构的识别比较好，但附录和 references 区域有时候会把所有内容扁平化，需要重新挂到正确的 section 节点下。
3. **双栏论文偶尔出现列间内容交错**——同一个逻辑段落被拆成了两个 block，后处理需要按 page + 坐标做合并。

所以 section tree 不是直接用 Marker 的输出，而是后处理模块基于 Marker 的 block tree 重建的。

### Marker 对 LaTeX 公式的处理效果怎么样？你的后处理里有没有专门处理公式的逻辑？

Marker 能识别大部分行内和行间公式，输出 LaTeX 字符串。大部分情况是可用的，但有两个常见问题：

1. **长公式断行**——跨行的公式有时会被拆成两个 block，后处理里我做了相邻公式 block 的合并逻辑，判断条件是连续两个 block 都是 equation 类型且中间没有文本 block。
2. **公式和正文粘连**——偶尔行内公式没有被独立识别，和前后文本混在一个 text block 里。这种情况我没有做重型修复，而是保持原样进入分块，因为强行拆分反而容易破坏上下文。

另外在分块时有一个原则：不切断公式。如果一个子块的切分点落在公式内部，会向后延伸到公式结束位置再切。

### 子块的 token 上限设的多少

子块上限设的 **512 token**，overlap **64 token**。

512 的选择基于两个考虑：

1. **embedding 模型的有效输入范围**——我用的 embedding 模型最大支持 8192 token，但实际上在 256-512 范围内语义表示质量最好，超过 512 之后 embedding 会开始"稀释"，多个主题混在一起导致检索精度下降。
2. **实际段落长度分布**——我统计过大约 200 篇论文，正文段落的中位数长度大约在 300-400 token，512 能覆盖绝大多数自然段落不被强制切断。

64 token 的 overlap 大约是一两句话，目的是保证切分边界处的上下文不完全丢失。试过不加 overlap，在段落中间硬切的情况下，边界处的 chunk 检索召回率明显下降。

------

### 父块有大小上限吗

有上限，父块上限设的 **2048 token**。

如果一个 section 超过 2048 token，我会按 subsection 拆成多个父块。如果 section 内部没有 subsection（比如有些论文的 introduction 就是一整段），会按段落边界在 2048 附近切成多个父块，每个父块保留相同的 `heading_path` 和 `section_type`。

父块的意义不是拿去检索，而是在上下文组装阶段提供子块周围的上下文窗口，所以 2048 是一个"context window 友好"的大小——通常精排后取 top-5 子块，如果它们来自 3 个父块，3 × 2048 = 6144 token，加上 prompt 模板和图表补充内容，总输入大概控制在 8000-10000 token，不会打爆模型上下文。

------

### overlap 多大？为什么要 overlap

64 token。试过不加 overlap，结果是在段落中间硬切的 chunk，如果切分点恰好在一个关键论述的中间，这个 chunk 的语义表示就不完整，导致相关查询召不回来。

举个具体例子：一个段落在讲 "We use Adam optimizer with learning rate 1e-4 and batch size 32"，如果恰好在 "learning rate" 之后切断，前半段 chunk 只有 optimizer 信息没有 learning rate 值，后半段 chunk 只有数值没有主语。加了 overlap 之后两个 chunk 都能保留完整语句。

但 overlap 也不能太大，否则相当于同一段内容在向量库里重复出现，召回时会出现大量近似重复结果。64 token 是在"保留边界上下文"和"避免过度重复"之间的平衡点。

### BM25 的参数 k1 和 b 你调过吗

调过。用的是和 embedding 评测同一套标注数据，100 个 query + relevant chunk 对。

默认参数是 k1=1.2, b=0.75。我在 k1 ∈ [0.8, 1.2, 1.6, 2.0] 和 b ∈ [0.5, 0.65, 0.75, 0.85] 的网格上搜了一遍。

最终用的 **k1=1.4, b=0.65**。

- k1 略调高是因为论文里重复术语的词频信号很重要（比如 method 部分反复提到模型名），需要让高频词有更大的权重。
- b 略调低是因为论文 chunk 的长度比较统一（都在 512 附近），文档长度归一化不需要太强。

调参带来的 Recall@10 提升大概 1.5 个点，不算大但也不费什么成本。

## 查询

接着说**查询阶段**。查询进入系统以后，我不会直接拿用户原始问题去检索，而是先经过 agent 侧的查询理解模块，完成查询改写和槽位填充。agent 会产出一个**结构化检索请求**，包括改写后的检索语句、关键词集合以及过滤条件。之后检索层再基于这个结构化查询对象，同时走 Milvus 的语义召回和 BM25 的关键词召回，再做 RRF 融合和 CrossEncoder 精排。

这两路结果出来以后，我会用 RRF，也就是**倒数排序融合**，去做统一排序。融合以后，我还会再加一层 CrossEncoder 精排，用 bge-reranker，对前面召回出来的前几十个候选做进一步排序。

在**生成阶段**，重点会放在上下文组织方式上。我不会直接把前几个小块生硬地塞给大模型，而是会根据每个块的父块编号和标题路径，补上一些必要的章节信息和相邻上下文。

## 去重

工程实现上，我还会特别注意**幂等**和版本治理。文档进入摄入流程后，我会先计算 PDF 的 SHA256 做**文件级去重**；块会生成稳定的块编号；

Milvus 这一侧**论文chunk基本是静态知识，适合"去重丢弃"。**

**所以策略是：**

- 发现重复 → 直接丢弃新的，保留已有的就行
- 不需要合并，不需要覆盖
- 用 `doc_id + chunk_index` 做 primary key，天然幂等，重复导入同一篇论文也不会产生冗余

原因很简单：论文内容发表后不会变。你不存在"同一段知识的更新版本"这个问题。去重的核心目标是防止同一篇论文因为不同来源（比如arXiv版和会议版）、不同格式（PDF排版差异导致chunk切分不同）产生重复chunk。

# RAG评测

### 评价指标，为什么用这个指标

RAGAS 核心理念是 LLM-as-a-judge

1. 评估generator的质量-LLM回答的好不好：
   * 忠实度：回答是不是百分百来自于检索到的上下文
   * 回答相关性：回答和问题是不是高度相关的
2. 评估Retriever的质量：在向量库中找的准不准：
   * 上下文准确度：TOP-K的质量，也就是看是不是把最相关的排在最前面
   * 上下文召回率：需要提供Ground Truth，评价检索到的上下文是否包含了回答问题所需要的必要信息

优点：

RAGAS 把 RAG 系统的评测拆成了检索和生成两个独立维度

Faithfulness 会先把答案拆成一个个 claim，评估粒度细，可解释性强

开源且易集成， 适合持续监控和回归测试（嵌入 CI/CD 流水线）

### TOP - k选择

$$ \text{Recall@k} = \frac{\text{前 k 条结果中命中的相关文档数}}{\text{该问题所有相关文档总数}} ] $$

k=3 到 k=5 提升明显，k=5 到 k=10 几乎不涨，但 context 长度翻倍。所以 top-k=5 就是那个「收益/成本」的拐点，是典型的调参依据。

# 入口query质量检测

放在整个 pipeline 最前面，做的是**机械性缺陷检测**——输入本身形式上就是坏的，不需要任何语义理解就能判断。具体就三条规则：

- 截断检测：末尾是连接词、介词（"LoRA和"、"如果我想在"）
- 纯指代检测：整句只有代词（"那个呢？"），且没有历史对话上下文
- 过短检测：有效字符不足 4 个

## **"为什么不用 BERT？"**

> 最初考虑过用 BERT 做分类，但分析后发现不合适：
>
> - **标注数据问题**："输入质量低"的定义依赖上下文，"LoRA"两个字在有 session history 时是合理的 follow-up，没有时才需要反问——这种上下文依赖的标签很难标注
> - **成本不对等**：实际高频的机械缺陷（截断、纯指代）用 3 条正则就能解决，引入 BERT 多了训练、部署、版本管理整套成本
>
> 所以选择了 ROI 更好的方案

# 槽位填充

### 槽位定义（固定 schema）

Slot Filling 输出是一个固定 Pydantic schema——`SlotFillingOutput`，共四个槽位：

| 槽位                 | 类型                    | 含义                                                         | 何时填充                                      |
| :------------------- | :---------------------- | :----------------------------------------------------------- | :-------------------------------------------- |
| `entities`           | `list[str]`             | 关键实体（论文名、技术名、模型名、作者名、数据集名）         | 所有意图都必须至少返回一个                    |
| `dimensions`         | `list[str]`             | 比较维度（如显存效率、训练速度、准确率）                     | 仅 `comparative` 意图非空；其他意图返回空列表 |
| `constraints`        | `list[ConstraintEntry]` | 隐式约束，`{key, value}` 键值对（如 `time_range: "2023以后"`、`model_scale: "7B以下"`、`domain: "NLP"`） | 按问题实际内容提取，不一定有                  |
| `reformulated_query` | `str`                   | 改写后的检索友好查询，去掉口语噪声、展开缩写；`follow_up` 时把代词替换成实体名，保证独立可检索 | 所有意图都必须非空                            |

### Fixed schema 还是 Dynamic schema？

整体是固定 schema，但 `constraints` 槽位内部是半动态的：

- 四个槽位的名称和类型是固定的（Pydantic 强约束，OpenAI structured output 严格校验）
- `constraints` 里的具体 key 是动态的——比如这次问题有 `time_range`，下次问题有 `model_scale`，LLM 按题意自由命名。我们用 `list[ConstraintEntry]`（而不是 `dict[str, str]`）的原因正是 OpenAI structured output 对 `additionalProperties` 的校验限制，`{key, value}` 对更符合其 JSON schema 要求，用完再在代码里转成 `dict`。

# MCP

## 并行设计

> 所以方案是在 MCP 协议上新增一个 **batch tool**，客户端一次把多个 query 打包发出去，服务端内部做 batch 处理。改动涉及两端：

**服务端**（RAG Server）：

> `batch_query_knowledge_hub` tool，接收一个 queries 数组。内部处理分三步：
>
> 第一步，**Batch Embedding**——把所有 query 文本组成一个 batch，一次 `model.encode()` 调用就拿到所有向量。这一步收益最大，因为 GPU 的并行计算能力被充分利用了，3 个 query 的 embedding 时间从 45ms 降到 ~18ms。
>
> 第二步，**并行向量检索**——拿到 N 个 embedding 后，用 `asyncio.gather` 同时发起 N 个向量库查询。Milvus向量库本身支持并发搜索，这一步从 90ms 降到 ~35ms。
>
> 第三步，**统一返回**——把所有结果按 query 顺序组装成数组，一次性返回。

**客户端**（Paper Pilot）：

> Comparative 策略所有实体的 query 组成一个数组，调一次 `batch_search`。

**决策 1：优雅降级**

> `batch_search` 方法会先检测 RAG Server 是否注册了 `batch_query_knowledge_hub` 这个 tool。如果有就走 batch 路径，如果没有就自动退化成 `asyncio.gather` + 多次单独调用。
>
> 这样做是因为 RAG Server 是独立项目，版本不一定同步更新。客户端不能假设服务端一定支持 batch，需要**向后兼容**。

**决策 2：batch size 上限**

> 设了一个 `max_batch_size=32`。虽然一般 comparative 策略只有 2-5 个实体，但如果未来有其他场景（比如 multi-hop 的窗口并行）一次传入很多 query，不做限制可能撑爆 GPU 显存。超过上限的自动分批，每批内部还是 batch 处理。

**决策 3：错误隔离**

> batch 请求中如果某一个 query 检索失败，不应该让整个 batch 都失败。服务端对每个 query 单独 try-catch，失败的返回空结果数组，成功的正常返回。客户端拿到结果后按位置对应回各个实体。
>
> 这比直接抛异常好——比如对比三个实体，其中一个在知识库里没有相关内容，不应该影响另外两个的正常检索和最终合成。

**决策 4（可选，时间够就讲）：为什么不用连接池代替 batch**

> 也考虑过连接池方案——维护多个 MCP session，每个检索请求分配不同的 session 实现并行。但连接池解决的是**传输层**的并行，embedding 计算在服务端仍然是每个请求单独过一遍模型，没法 batch。
>
> Batch API 是从**计算层**根本解决问题——一次把所有 query 的 embedding 一起算了。所以 batch 的收益上限更高。当然两个方案不冲突，极端高并发下可以**连接池 + batch** 组合使用。

> 改完之后 comparative 策略的检索阶段延迟从 ~210ms 降到 ~88ms，快了约 2.4 倍。实体数量越多收益越明显，因为 batch embedding 的边际成本几乎为零。

# Router设计

## bert取舍

> 我们用的是 Qwen-7B 而不是 BERT，主要是因为 Critic是 Qwen 做的。Router 也用 Qwen，**共享同一个 base model 权重**，只是加了不同的 LoRA adapter。这样部署时只需要加载一次 base model，切换 adapter 就能做不同任务，显存和运维成本都更低。如果 Router 单独用 BERT，就要额外加载一个模型，增加系统复杂度。

------

### Q: 但 BERT 更小更快，分类效果也不差？

> 是的，**单看分类任务 BERT 完全够用**，而且推理更快（110M vs 7B）。但我们的场景不是"只做一个分类器"，而是一个多组件的 Agent 系统。用 Qwen 统一做底座有几个好处：
>
> 1. **Multi-LoRA serving** — 用 vLLM 部署一个 Qwen base，挂 Router adapter 和 Critic adapter，GPU 上只有一份 base model 权重。如果 Router 用 BERT，就要单独部署一个服务，多一套运维。
> 2. **语义理解更强** — 我们的问题是学术领域的（LoRA、RAG、Transformer 这些），7B 的模型对这些领域术语的理解比 BERT-base 好很多，特别是 multi_hop 和 exploratory 这两个比较模糊的类别，需要更强的语义判断。
> 3. **中英文混合** — 用户问题可能中英混杂，Qwen 原生支持，BERT 需要用 multilingual 版本，效果一般。

### Q: 7B 做 5 分类是不是太大了？

> 确实参数量是 overkill，但实际推理时 **4-bit 量化后只占 ~4GB 显存**，而且和 Critic 共用同一份权重。不是单独为 5 分类部署一个 7B。如果资源确实紧张，也可以换成 Qwen2.5-1.5B 或 0.5B 做 Router 的 base，LoRA 的好处就是换底座很容易。
>
> 而且 Router 只做一次 forward pass（SEQ_CLS），不是 autoregressive 生成，所以 7B 模型的推理延迟也就 20-30ms，不是瓶颈。

### Q: 那什么情况下你会选 BERT？

> 如果 Router 是一个**独立部署的微服务**、不和其他组件共享模型、而且对延迟要求极高（比如 <5ms），那 BERT 或者 DistilBERT 是更好的选择。或者如果分类任务很简单、训练数据很多、不需要依赖预训练模型的领域知识，用 BERT 完全足够。我们的场景选 Qwen 主要是**系统级的工程决策**，不是说 BERT 做不了。

## 复杂问题

第一级还是你现有的Router做单标签分类，但推理时取softmax输出的概率分布作为置信度。如果top-1的置信度高于阈值（比如0.75），说明意图明确，直接走对应路径，这覆盖了绝大多数case。

如果置信度低于阈值，说明query可能是复合意图，这时候触发第二级——把query交给云端大模型做一次query 分解，拆成多个子query，每个子query再分别过Router。比如"对比这三篇论文的方法，并分析哪个最适合小数据场景"会被拆成"对比这三篇论文的方法"→ comparative，"分析哪个最适合小数据场景"→ multi_hop，各自走各自的路径执行完之后再合并结果。

## 观测指标

评估Router的时候，因为我的意图分类数据集四个类别的样本量是均衡的，所以整体指标我用的是Macro-F1。Macro-F1的计算逻辑是先对每个类别分别算Precision和Recall，再各自算F1，最后四个类别的F1取算术平均。用Macro是因为它对每个类别权重相同，不会因为某个类别样本多就主导整体指标，正好匹配我均衡数据集的情况。

但光看一个Macro-F1是不够的，还需要看每个类别的Precision和Recall，因为不同类别的误判代价是不对称的。拿multi_hop来说，Precision低意味着误判多——把本来是simple或comparative的query错误路由到了multi_hop。multi_hop走的是Plan-and-Execute路径，要先拆子问题、逐步检索、还可能replan，是整个系统里链路最长、token消耗最大的路径。如果一个"这篇论文用的什么数据集"这种simple问题被误判成multi_hop，就是用最重的策略做最轻的活，白白浪费好几轮LLM调用，用户体感就是"明明很简单的问题为什么等这么久"。Recall低则意味着漏判多——真正需要多步推理的复杂问题被当成simple处理了，一次检索根本召回不到完整的信息链，生成的答案要么片面要么直接幻觉，这个错误代价更大，因为它直接损害回答质量。

其他类别也是同样的逻辑。比如comparative的Recall低，意味着真正的对比类问题被当成multi_hop串行处理，不仅慢，还容易丢失对比维度的一致性，输出结构混乱。所以调优的时候我会盯着confusion matrix，看哪些类别对之间互相混淆最严重，然后针对性地在训练数据或prompt里强化那个决策边界。

在我的场景里，学术问答对答案质量的要求高于响应速度，用户宁愿多等一会儿也不想拿到不完整的回答，所以整体策略上Recall的优先级是高于Precision的。最终通过前面说的三轮prompt迭代加训练策略调优，Macro-F1从基线的83%提升到了91%，其中提升最大的就是comparative和multi_hop之间的混淆被大幅缓解。

### Macro-F1

计算过程就是：先对每个类别分别算出Precision和Recall，然后每个类别各自算一个F1：

F1 = 2 × Precision × Recall / (Precision + Recall)

F1本质上就是Precision和Recall的调和平均，两个都高F1才高，任何一个拉垮都会把F1拉下来。

然后Macro-F1就是把四个类别的F1直接取算术平均：

**Precision**就是"模型预测为multi_hop的样本里，有多少是真的multi_hop"。计算方式是：

Precision = TP / (TP + FP)

**Recall**就是"所有真正是multi_hop的样本里，有多少被模型正确识别出来了"。计算方式是：

Recall = TP / (TP + FN)

##  LoRA 具体实现？

我们有两处用 LoRA：Router 意图分类和 Critic 质量评估，都是「基座 + LoRA adapter」的训法，推理再合并。

训练这边：用 PEFT这个库的LoRA，只在 attention 的query， key，value， output四个线性层上挂在低秩矩阵。超参是 rank r=16、lora_alpha=32、lora_dropout=0.05。Alpaca 格式（指令 + 输入 + 输出）数据、因果labels：整段序列都 tokenize，但在 labels里只对output保留真实 token id。用的是HuggingFace Trainer 训，并且只保存 adapter

## Router 怎么训练的

底座用的 **Qwen-7B**，上面加 **LoRA**（r=16, alpha=32），task type 是 **SEQ_CLS**。就是在最后一层接一个 `Linear(hidden_size, 5)` 的分类头，训练时用标准的 **CrossEntropy Loss**。LoRA 冻结主干只训 attention 的 q/k/v/o projection，但分类头是随机初始化的，必须完整训练，所以在 PEFT 配置里用 `modules_to_save=["score"]` 把它排除出 LoRA 的低秩近似，确保权重完整保存。

推理的时候很简单——问题过 tokenizer，**一次 forward pass** 拿到 5 维 logits，过 **softmax** 得到概率分布，**argmax** 取预测类别，**max probability 就是 confidence**。这个 confidence 是真实的校准概率，不是模型生成的数字。

## 数据集构建

数据主要是通过**少样本in-context** 学习生成数据集，并进行人工校验的方式来生成的。抽取几十个个样例报标喂给gpt4，构造问题分类描述的prompt，然后让gpt4根据文档来提问，并自己回答问题的类型；这样可以得到一批种子训练语料，然后在人工对这批语料进行check，去除标注错误的数据和脏数据。为了让指令微调的数据具有多样性和高质量，我们还利用GPT4对这批数据进行**说法泛化**，具体包括为句式的改变，转换为非正式或口语化说法，以及省略说法，和句子容错等处理，最后得到10k条高质量的问题分类指令微调数据集。

# Critic设计

`Critic v1` 只做两件事：

1. 判断当前 `draft_answer` 是否达标
2. 生成一条简短、可执行的 `feedback`，供下一轮重试使用

固定输出以下字段：

```json
{
  "score": 6.5,
  "completeness": 0.50,
  "faithfulness": 0.30,
  "passed": false,
  "feedback": "补充 A 与 B 在显存上的差异，并删除未被上下文支持的结论。"
}
```

字段定义如下：

- `score`
  允许值：`1.0 / 2.5 / 4.0 / 5.5 / 6.5 / 7.5 / 8.5 / 9.5`
- `completeness`
  允许值：`0.10 / 0.30 / 0.50 / 0.75 / 0.95`
- `faithfulness`
  允许值：`0.10 / 0.30 / 0.50 / 0.75 / 0.95`
- `passed`
  规则：`score >= 7.0`
- `feedback`
  非空、简短、可执行的文本

`Critic v1` 只保留和训练方案直接对齐的维度：

- `completeness`
- `faithfulness`
- `score`
- `passed`
- `feedback`

其中：

- `completeness` 表示回答是否覆盖了用户问题要求的关键点
- `faithfulness` 表示回答是否被上下文支持、是否存在 hallucination 或 unsupported claim
- `score` 表示整体质量，用于最终 gate
- `passed` 是程序控制字段
- `feedback` 是下一轮 retry 的引导信号

## 如何避免重试越修越错

v1 建议保留以下保护机制：

- 固定 `max_retries`
- 始终对照原始用户意图评分，而不是只看改写后的问题
- 记录重试过程中的 `best-so-far score`
- 如果分数持续下降，就停止继续重试
- 如果多轮低质量是由检索不足导致，就升级到 `web_search_fallback`

# 四条路径

1. simple
   适合单事实、单跳查询。做法是“一次检索 + 一次生成”，成本最低、延迟最小，覆盖大部分 factual / follow-up 问题。
2. multi_hop
   适合需要多步依赖的问题，比如“先找A，再用A解释B”。我用了 Plan-and-Execute：先拆子问题，再逐步检索，过程中允许 replan，最后统一综合。这条路径解决的是复杂链式推理的完整性问题。
3. comparative - ReWOO
   适合对比类问题，比如比较不同论文、模型或方法。我这里不是串行查，而是先抽取 entities 和 dimensions，再对每个实体并行检索，最后一次性做结构化对比。这样比通用 multi-hop 更快，也更容易保持比较维度一致。
4. exploratory
   适合开放式探索问题，比如“有哪些方向”“怎么设计方案”。我用了 ReAct 风格：think / act / observe 循环，模型可以边查边调整搜索方向，直到信息足够再总结。这条路径主要解决开放问题里目标不明确、一次检索不够的问题。

# LoRA -SFT

### 基础QA：

1. 什么是 SFT？
   SFT 是监督微调。做法是用带标注的指令数据继续训练预训练模型，让它学会完成某类具体任务，比如分类、问答、抽取、评审。它本质上通常还是语言模型的 next-token prediction，只是训练数据从通用语料变成了任务样本。
2. SFT 和预训练有什么区别？
   预训练学的是通用语言能力和世界知识，目标是“会说话、会建模语言”；SFT 学的是特定任务行为，目标是“按要求做事”。预训练数据规模大、弱结构，SFT 数据规模小、任务导向强。
3. SFT 训练的本质是什么？
   本质是让模型在给定 instruction 和 input 后，去预测标准 response 的 token。通过不断最小化预测误差，模型学会从输入映射到目标输出。所以它既能学格式，也能学任务本身的基础行为。
4. LoRA 是什么？
   LoRA 是一种参数高效微调方法。它不直接更新原始大模型权重，而是在某些线性层旁边加一个低秩增量矩阵，只训练这部分小参数。这样能显著减少可训练参数量、显存占用和保存成本。
5. 为什么 LoRA 能省显存？
   因为它冻结了大部分 base model 参数，只训练很小的 adapter 参数。这样优化器状态、梯度存储、checkpoint 体积都会更小，所以比全参微调便宜很多。
6. LoRA 里的 r 是什么？
   r 是低秩分解的秩。可以理解成 adapter 的容量大小。r 越大，可学习能力越强，但参数量和显存也会增加；r 太小则可能表达能力不足。
7. LoRA 的 alpha 和 dropout 有什么用？
   alpha 控制 LoRA 增量的缩放强度，影响更新幅度；dropout 是对 LoRA 分支做正则，防止过拟合。工程上一般和 r 一起调。
8. 为什么 LoRA 常插在 q_proj/k_proj/v_proj/o_proj？
   因为注意力层对模型行为影响很大，同时这些投影层参数集中、改动效率高。在很多任务里，只改这些位置就能带来不错效果，属于性价比很高的做法。

### 训练QA：

**1. 你们项目里的 critic 是怎么训练的？**

我们这个项目里的 critic 不是单独的分类头，而是基于 `Qwen2.5-7B-Instruct` 做两阶段微调。

第一阶段是 LoRA SFT。训练数据是 `instruction/input/output` 格式，输入里包含用户问题、草稿答案、检索上下文和策略类型，输出是一个 JSON，字段包括 `score、completeness、faithfulness、feedback`。这一阶段主要让模型学会 critic 这个任务本身，也就是看到一份答案后知道要从完整性、忠实性和总体质量去评估。

第二阶段是 DPO。这里的数据是 `prompt/chosen/rejected`，让模型在同一个评估输入下，更偏向输出高质量评审，减少那种“格式对了但判断不准”的情况。我们项目里 router 只有 SFT，critic 才做了 DPO，因为 router 更像单标签分类，critic 更依赖偏好对齐。

**2. 你们为什么要先 SFT，再 DPO？**

因为这两个阶段解决的问题不一样。

SFT 先把 critic 教会，让它知道这不是普通问答，而是“评估答案质量”的任务；同时也让它学会输出稳定 JSON，不然 DPO 阶段连基本任务边界都不稳定。

DPO 是在 SFT 的基础上继续做“偏好校准”。比如同一份草稿答案，模型可能会给出两种都像样的评价，但其中一种更能抓住事实错误、更重视 groundedness、更能给出可执行 feedback。DPO 就是专门拉开这种偏好差异的。

如果直接跳过 SFT 做 DPO，通常会遇到两个问题：
- 模型根本没学会任务格式，chosen/rejected 信号利用率很低
- 训练不稳定，最后只学到一些表面风格，不一定学到评审能力

**3. 你们的 SFT 数据是怎么做的？**

我们这个项目里，critic 的 SFT 数据主要是用 LLM 生成的，不是人工逐条标注。

做法是先构造评估场景，也就是：
- 一个 question
- 一个 draft answer
- 一组 retrieved contexts
- 一个 strategy 标签

然后再用一个上游 LLM 去充当“教师评审器”，让它根据这组输入输出一条标准 verdict，也就是那四个字段的 JSON。这样就得到一条 SFT 样本。

项目里有两种生成思路：
- 一种是固定题池和上下文池，效率高一些
- 一种是 full-scenario，全场景都让 LLM 生成，更接近真实分布

如果面试官追问，我会补一句：  
我们实际更看重 full-scenario，因为 critic 对输入分布很敏感，只用固定模板池会导致训练样本过于规律，模型容易学会套格式，泛化一般。

**4. 你们怎么做 prompt 和训练样本格式？**

我们这边 SFT 训练不是直接喂裸 JSON，而是拼成统一文本模板，大概是：

- `Instruction`
- `Input`
- `Response`

其中 `Input` 里会写清楚 question、draft answer、contexts、strategy，`Response` 才是目标 verdict JSON。

这么做的原因有两个：
- 跟 instruction-tuning 模型的输入分布更接近
- 后续做推理时，线上 prompt 也保持类似语义结构，训练和推理分布更一致

真实训练里一个很常见的坑是：训练时用一套模板，推理时换另一套模板。  
这样 loss 可能很好看，但上线后输出结构和判断都会漂。我一般会在项目里尽量保持训练 prompt 和推理 prompt 的语义等价，不完全照搬字符串，但任务边界和字段定义尽量一致。

**5. LoRA 具体怎么做的？为什么这么配？**

我们项目里 LoRA 挂在标准的注意力投影层上，主要是 `q_proj/k_proj/v_proj/o_proj`。这是比较稳的做法，因为这些模块对模型行为影响大，参数效率也高。

默认配置上：
- `r = 16`
- `alpha = 32`
- `dropout = 0.05`

我如果在面试里解释，会这么说：

`r=16` 对 7B 级别模型是一个比较常见的平衡点，容量够用，不至于太小学不动，也不会大到训练成本明显上升。  
`alpha=32` 相当于给 LoRA 更新一个适中的缩放力度。  
`dropout=0.05` 是为了抑制一点过拟合，因为我们的训练数据不是特别大，而且一部分还是 LLM 合成数据，模板痕迹会比较明显。

如果面试官继续问“为什么不是 8 或 64”，我会答：  
这个任务不是特别开放式生成，更像结构化评审，`r=16` 往往已经够了。`r` 太小可能学不会稳定的评判边界，太大则更容易把教师模型的风格噪声也学进去。

**6. 你们训练时 loss 是怎么算的？会不会只学会格式？**

严格来说，SFT 本质上还是 next-token prediction。  
模型看到 instruction 和 input 后，要去预测 response，也就是那段 verdict JSON。

如果面试里要显得更像真的做过，我会这样补充：

理想做法是尽量把监督重点放在 response 部分，也就是只让模型重点学习输出 verdict，而不是浪费太多能力去复现 prompt 模板。  
因为如果 prompt 部分也强参与 loss，模型很容易把固定格式学得很好，但对“什么时候该打低分”这件事学得不够扎实。

我们这类 critic 任务最怕的一个坑就是：
- JSON 特别稳定
- feedback 也很像回事
- 但分数虚高，没抓住上下文冲突

所以训练后我不会只看结构化输出正确率，还会单独看几类 bad case：
- 草稿答案定义错误时，faithfulness 能不能明显下降
- 草稿答案漏答时，completeness 会不会下降
- feedback 是不是泛泛而谈，还是能指出具体问题

**7. 你们 DPO 数据怎么构造？**

DPO 数据还是基于同一个评估 prompt，只不过不再只生成一条标准输出，而是生成两条有偏好顺序的输出：

- `chosen`：更好的 critic 评价
- `rejected`：更差的 critic 评价

比如同一个错误答案：
- `chosen` 会明确指出“答案把 LoRA 误说成数据集，并且没有被 context 支持”
- `rejected` 可能只是给个差不多的高分，或者 feedback 很空

项目里 chosen 和 rejected 也是靠 LLM 生成的，但这里有个工程上的关键点：  
不能只要求“一个高分，一个低分”，还要保证它们的理由真的和内容对应，否则 DPO 学不到真实偏好，只会学到“chosen 语气更积极/更长”这种伪特征。

所以真实做数据时，我会特别关注两个点：
- chosen/rejected 的差异要落在“评判逻辑”上，不只是分数不同
- 差异不能太小，不然偏好信号太弱

**8. 你们训练时最常见的坑是什么？**

这个项目如果真去训练，我觉得最容易踩的坑有 5 个。

第一，数据模板太重。  
因为很多样本是 LLM 生成的，如果 prompt 和输出风格过于统一，模型会学得很像模板机。表面现象是 JSON 很稳，feedback 句式高度重复，真实 bad case 抓不住。

第二，长度截断。  
critic 输入里有 question、draft、contexts，contexts 一长就很容易超过 `max_seq_length`。如果截断把关键证据截没了，模型训练时会学到错误映射。  
所以我一般会先统计 token 长度分布，再决定 `max_length`，而不是拍脑袋设。

第三，训练集 loss 很好看，验证集评审质量一般。  
这通常不是模型不行，而是数据分布太整齐。尤其是固定题池生成的数据，验证集如果也长得很像训练集，很容易虚高。

第四，DPO 的 chosen/rejected 质量不稳定。  
如果 rejected 太差，任务就太简单；如果两者差别太小，学不到偏好；如果 chosen 其实也有逻辑错误，那会直接把 critic 往歪处带。

第五，训练 prompt 和线上推理 prompt 不一致。  
这个在 judge/critic 任务里特别伤，因为模型不是在做自由生成，而是在做结构化评估，一旦 prompt 边界变了，评分标准就容易漂。

**9. 你们怎么调 batch size、grad accumulation 和学习率？**

这个项目默认配置其实已经比较像真实可用的配置了：

critic SFT 这边：
- `per_device_train_batch_size = 8`
- `gradient_accumulation_steps = 2`
- `learning_rate = 2e-4`
- `max_seq_length = 512`

我会这样解释这个选择：

因为 critic 输入比 router 长很多，有上下文，所以 batch 不能太大，不然显存会炸。  
但如果 effective batch 太小，训练又会不稳定，所以用 grad accumulation 把有效 batch 补上去。  
学习率用到 `2e-4` 是 LoRA SFT 比较常见的范围，因为可训练参数少，更新可以相对激进一些。

DPO 阶段 batch 更小：
- `per_device_train_batch_size = 4`
- `gradient_accumulation_steps = 2`
- `learning_rate = 5e-5`

因为 DPO 本身训练更敏感，偏好优化阶段我通常会把学习率压低一些，避免把 SFT 阶段已经学到的格式和基础能力冲坏。

**10. 你们怎么判断训练是真的有效，不是只会输出格式？**

我会分三层看。

第一层看结构化输出稳定性。  
也就是 JSON 能不能稳定解析，字段范围是不是合理，比如 `score` 在 0 到 10，`faithfulness` 在 0 到 1。

第二层看维度级别的评判正确性。  
比如：
- 明显与 context 冲突的答案，faithfulness 应该下降
- 漏答问题的答案，completeness 应该下降
- 空洞 feedback 要减少

第三层看业务闭环指标。  
在我们这个项目里，critic 是接在 retry/refine 后面的，所以我会看：
- critic fail 后触发重试，第二轮答案质量有没有提升
- forced pass 的比例有没有下降
- web fallback 的触发频率有没有变化

如果只是 JSON 变稳定了，但 retry 路径没有变好，那我不会认为 critic 真训练成功了。

**11. 如果面试官问：你们项目里 router 和 critic 的 SFT 有什么不同？**

我会答：

router 的 SFT 更像分类任务，输入主要是用户问题，输出是 intent type 和 confidence。它关注的是“该走哪条策略”。

critic 的 SFT 更像结构化评审任务，输入是 question、draft、contexts，输出是多维度质量判断。它关注的是“这份答案质量怎么样”。

所以两者虽然都叫 SFT，但本质上学的不是一回事：
- router 学的是路由决策
- critic 学的是评估映射

这也是为什么 router 通常 SFT 就够了，而 critic 往往还要接 DPO。

**12. 如果面试官问：你亲自做这类训练时，会先做哪几件事？**

我会说我一般先做这 4 件事：

第一，抽样看数据，不急着开训。  
先看 100 条左右，确认 prompt、label、长度、重复率有没有明显问题。很多训练问题根源其实在数据，不在模型。

第二，先跑一个小步数 smoke test。  
比如几十到一两百 step，先确认 loss 正常下降、不会 OOM、输出格式没跑偏，再决定是不是正式开大训练。

第三，训练前先准备验证 bad case 集。  
尤其 critic 任务，不能只靠 loss。我要提前准备几类典型错误样本，专门看模型能不能抓住关键问题。

第四，训练结束后做人审，不只看指标。  
我会把高分低质、低分高质、feedback 空洞这几类 case 单独拎出来看，因为这些是 judge 模型最常见的问题。

# DPO

## 数据

1. 采集原始样本池
2. 构建标准化 teacher gold label
3. 产出 SFT 数据集
4. 从同一批原始样本派生 DPO pair

- 原始样本池：`10000`
- 训练集：`8000`
- 验证集：`1000`
- 测试集：`1000`

从原始样本池派生：

- `SFT` 样本：`6000`
- `DPO` pair：`4000`

dpo pair里面：

| 来源             | 比例 | chosen               | rejected                           | 目的                 |
| :--------------- | :--- | :------------------- | :--------------------------------- | :------------------- |
| 真实 agent trace | 50%  | Critic 通过的好回答  | Critic 拒绝的差回答                | 学真实分布           |
| 规则扰动         | 30%  | 原始好评估           | 故意扰动的"看似合理但有缺陷"的评估 | 学精细区分           |
| LLM 合成         | 20%  | LLM 生成的高质量评估 | LLM 生成的低质量评估               | 补覆盖不到的长尾场景 |

Hard negative 从真实较好答案自动构造，优先做以下几类：

- 删除关键结论
- 插入未被 context 支持的句子
- comparative 问题只回答一边
- 去掉适用条件，造成过度泛化

## 测试验收观测

- 相比基座模型，`false_pass_rate` 下降 `30%+`
- 相比纯 SFT，DPO 使 `boundary_f1` 提升 `3-5` 个点

## 合成数据缺陷

如果 `train` 和 `val` 主要都是 LLM 合成数据，而 `test` 使用真实 trace，那么训练过程通常会显得比较顺，但这种“顺”不能直接理解成模型已经学会真实业务分布。

最常见的现象是：

- `train_loss` 和合成 `val_loss` 会下降得比较漂亮，因为训练集和验证集来自相近的合成分布。
- `json_valid_rate` 和 `field_valid_rate` 通常也会很高，因为合成数据的输出格式更统一、更规整。
- `pass_f1`、`score_mae`、`level_accuracy` 这类指标在合成验证集上往往也会显得不错，因为模型更容易学会 teacher 在合成样本上的打分模式。
- 到真实 trace 测试集上，真正反映业务价值的指标往往会回落，尤其是 `false_pass_rate`、`hallucination_false_pass_rate`、`boundary_f1` 和 `faithfulness_mae`。

原因通常有四个：

- 分布偏移：真实 agent 输出的语气、上下文长度、错误方式和合成样本不同。
- teacher bias 放大：模型学到的是 teacher 对“它熟悉的合成样本”的判断方式，而不是对真实错误分布的判断方式。
- 错误类型过于干净：合成坏答案通常更标准化，真实坏答案往往是“大部分都对，只在关键处错一句”。
- DPO 更容易学到 pair 捷径：如果 pair 也主要来自合成数据，模型可能更容易学会措辞模式、长度模式，而不是更准确的业务判定。

# 部署

## 讲一下这个项目推理加速是怎么做的

首先是使用PEFT库实现LoRA低秩适应，只训练很小一部分参数。这样不用每次请求都调用cloud llm，有效的降低了延迟，以及api成本，同时保持接近全量sft的效果。

其次是DPO对齐，对齐的是critic的评判输出的质量和风格。数据200对 chosen-rejected偏好数据对。一部分是同一 prompt 下 LLM 分别生成 chosen 和 rejected；另一部分 chosen 来自人工编写，rejected 用在分数、格式、feedback 上做规则扰动，并保证 chosen 严格优于 rejected

最后是量化部署，基座用 Auto-AWQ 做的4-bit量化，Router 和 Critic 以 LoRA 形式挂在这份基座上，由 vLLM 统一加载、按 model 名区分；Agent 通过 HTTP 调 vLLM 做意图分类和打分，拿到生成结果再解析成意图或打分。这样推理集中在 vLLM 进程里，能利用 PagedAttention 和连续批处理，Agent 只负责调 API，部署解耦、方便扩缩容。

（vLLM 对 AWQ 支持更好，部署简单）

## AWQ

### 怎么做的量化

这个项目里的 AWQ 量化，我做的是一套离线量化加在线推理拆分的方案。简单说，就是先不去量化 LoRA，而是只量化基座模型 Qwen2.5-7B-Instruct，把它压成 AWQ INT4，然后在线上用 vLLM 把这个 AWQ 基座和多个 LoRA adapter 组合起来做推理。

具体实现上，我单独写了一个量化脚本，核心是用 AutoAWQ 加载 HuggingFace 的基座模型，然后构造量化配置，比如 4 bit、group size 128、zero point 打开。校准数据我没有随便拿通用文本，而是优先复用项目训练阶段生成的 router 和 critic 数据，也就是从 JSONL 里抽 input 或 prompt 做 calibration，这样量化分布会更贴近实际业务场景。

量化完成之后，我把量化后的权重和 tokenizer 一起导出到一个独立目录，后面 vLLM 直接加载这个 AWQ 模型，再通过 Multi-LoRA 挂 router 和 critic 两个 adapter。

另外一个工程上的点是，AutoAWQ 依赖的 transformers 版本和训练链路里的 TRL 有冲突，所以我把量化步骤单独放到一个独立虚拟环境里，避免训练环境和量化环境互相污染。还有一个 fallback 机制，如果量化失败，脚本会写 fallback metadata，保证流水线不会直接中断，排查也更方便。”

“部署时还有一个细节，vLLM 跑 AWQ 需要显式用 float16，不能直接走 bfloat16，这个如果不处理会在服务启动时报错。

 ### 原理

AWQ 就是先看激活，找出激活值比较大的通道，因为他们对输出更敏感，再通过 scaling放大去保护这些通道，也就是减少他们量化时的误差，最后再做低比特权重量化。它的重点是，不是平均量化所有权重，而是优先减少关键部分的量化误差

### 可用性观测

第一层是**效果有没有明显掉**。
 因为量化最怕的不是不能跑，而是虽然跑得更快了，但是模型变笨了。所以我会先看和业务最相关的核心指标。
 像我这个项目里，Router 我会重点看**意图分类准确率、路由正确率、多意图槽位填充效果**；
 Critic 我会看它的**打分一致性、偏好排序能力**
 如果是整个 RAG 链路，我还会看**最终回答质量**，比如 answer correctness、faithfulness、retrieval hit rate 这一类。
 我的标准不是一点不掉，而是**下降要在业务能接受的范围内**。”

“第二层是**性能收益够不够大**。
 量化本来就是为了部署，所以我会重点看几个很实际的指标：
 一个是**吞吐**，比如每秒能处理多少 token、并发上去之后稳不稳；
 一个是**时延**，尤其是首 token 延迟和整体生成延迟；
 还有就是**显存占用**。
 因为如果量化完效果只掉一点，但吞吐提升很明显、显存降很多，那它在工程上就很有价值。
 反过来，如果效果掉了，但性能收益又不大，那这个量化就没什么意义。”

### 实验结果

- 显存占用明显下降，7B 级模型从 FP16 换到 4-bit，能到原来的大约 1/3 左右量级。
- 单次本地推理延迟有改善，但整条 agent 链路的端到端收益不会特别夸张
- 质量上应该“基本可用且接近原版”，尤其 router 这种短输出分类、critic 这种低温结构化打分任务，本来就比开放生成更适合量化。
- 比较健康的退化范围大概是：
  - router 路由一致率下降不超过 1 到 3 个百分点
  - critic 的分数波动在 0.2 到 0.5 分内
  - 最终答案质量几乎不明显下降，最多是偶发多一次 retry 或少数边界问题路由错类

## vllm

PagedAttention 的本质是把每个 request 的 KV Cache 分成固定大小的 block，通过 block table 做映射，而不是要求整个 KV 连续存放。这样动态分配时显存碎片会小很多，KV cache 利用率更高。
 continuous batching 则是在 decode 的每一步动态维护 batch，已完成的 request 立刻出队，新 request 只要资源允许就立刻入队，避免静态 batch 里的空槽问题。
 所以前者解决的是动态 KV 内存管理，后者解决的是在线推理调度效率

### QPS & 延迟

Router 和 Critic 都是短序列、少量 token 输出的任务，实际使用中不太可能同时有几十个并发请求。所以更关注的指标应该是**单请求延迟**（~30-60ms），而不是极限 QPS。

**Router（意图分类）** — 输入短（~100-200 tokens），输出极短（~10-30 tokens）：

| GPU             | 单请求延迟 | 连续批处理 QPS |
| --------------- | ---------- | -------------- |
| RTX 4090 (24GB) | ~30-60ms   | **20-50 QPS**  |
| A100 (80GB)     | ~15-40ms   | **50-120 QPS** |
| RTX 3090 (24GB) | ~50-90ms   | **15-35 QPS**  |

**Critic（质量评估）** — 输入中等（~300-500 tokens），输出稍长（~50-100 tokens）：

| GPU             | 单请求延迟 | 连续批处理 QPS |
| --------------- | ---------- | -------------- |
| RTX 4090 (24GB) | ~80-150ms  | **10-25 QPS**  |
| A100 (80GB)     | ~40-80ms   | **25-60 QPS**  |
| RTX 3090 (24GB) | ~100-200ms | **8-18 QPS**   |

### 显存

#### 显存组成

```
总显存 = 模型权重（固定） + KV Cache（随并发线性增长） + 临时激活（较小）
```

以你的项目为例（Qwen2.5-7B AWQ INT4）：

| 组成部分                       | 占用                     | 是否随并发增长 |
| ------------------------------ | ------------------------ | -------------- |
| 模型权重（AWQ INT4）           | ~4 GB                    | 固定不变       |
| LoRA 适配器（router + critic） | ~100-200 MB              | 固定不变       |
| KV Cache                       | **每个并发请求都要分配** | 线性增长       |
| 激活值 / 临时缓冲              | ~几百 MB                 | 小幅增长       |

#### KV Cache 占了大头

每个并发请求都需要自己的 KV Cache 来存储注意力的 key/value 状态。粗算：

```
每个 token 的 KV Cache ≈ 2 × num_layers × hidden_size × 2bytes(fp16)
Qwen2.5-7B: 2 × 28 × 3584 × 2 ≈ 0.38 MB/token
```

对 Router（seq_len 512）：

```
1 个请求：  512 × 0.38 MB ≈ 200 MB
8 个并发：  ≈ 1600 MB
32 个并发： ≈ 6.2 GB
```

对 Critic（seq_len **1024**）：

```
1 个请求：  1024 × 0.38 MB ≈ 390 MB
8 个并发：  ≈ 3.1 GB
32 个并发： ≈ 12.5 GB
```

#### RTX 4090 (24GB) 上的实际情况

```
模型权重:    ~4 GB（固定）
LoRA:       ~0.2 GB（固定）
系统开销:    ~1 GB
─────────────────────
剩余给 KV Cache: ~18.8 GB

Router 最大并发: 18.8 GB / 200 MB ≈ ~100 个
Critic 最大并发: 18.8 GB / 400 MB ≈ ~50 个
```

所以在你的场景下（短序列 + AWQ INT4），**24GB 显卡能扛住相当多的并发**，不太容易打满。

#### vLLM 的自动管理

vLLM 用 **PagedAttention** 管理 KV Cache，类似操作系统的虚拟内存分页：

- 不会一次预分配整个 max_seq_len 的显存，而是按需分页分配
- 请求完成后立即回收
- 通过 `--gpu-memory-utilization`（默认 0.9）控制 KV Cache 最多占多少显存

### 端到端（Agent 整条链路）

实际场景中每个请求需要 1 次 Router + 1 次 Critic + 云 LLM 策略推理，瓶颈通常在云 LLM（gpt-4o-mini ~1-3s）。vLLM 本地节点的延迟在整体中占比很小（~5-15%），所以：

- **Agent 端到端 QPS 约 1-5 QPS**（受限于云 LLM + RAG 检索延迟）
- vLLM 部分不会是瓶颈

1. 如果想提升吞吐，vLLM 默认的 continuous batching 已经很高效，主要调 `--max-num-seqs`（并发批大小）
2. AWQ INT4 让 7B 模型在消费级 GPU 上就能跑满，VRAM 占用约 4-5 GB，剩余空间可用于 KV cache

## 实际部署问题

### 同步调用卡死异步 pipeline（P0）

`vllm_client.py` 用同步 `openai.OpenAI`，阻塞了 LangGraph 的异步事件循环，并发能力归 。

**解决**：改用 `openai.AsyncOpenAI` + `await`，代码改动量很小，效果立竿见影。

### 单点 vLLM 实例宕机无恢复（P1）

当前只有一个 vLLM 进程，挂了之后：

- Router 和 Critic 同时不可用
- 虽然有 Cloud LLM fallback，但 fallback 延迟高得多（~500ms vs ~60ms）
- 如果 Cloud LLM 也限流了 → 全部降级

**解决**：

- 多实例部署 + 健康检查（K8s Readiness Probe 打 `/health`）
- vLLM 连续失败 3 次 → 自动切 Cloud LLM 30 秒 → 再探测恢复

### MCP RAG 服务的冷启动和连接管理

- MCP RAG 服务在生产环境有四个主要问题：

  **第一是冷启动**——stdio 模式每次 spawn 子进程要加载 Embedding 模型和向量库，需要 60-90 秒。解决方案是生产环境**放弃 stdio，只用 HTTP 模式**，RAG Server 作为常驻服务预启动，配合 K8s 健康检查确保就绪后才接收流量。

  **第二是串行化瓶颈**——当前只有一个 MCP session，所有检索请求串行执行。特别是 comparative 策略需要按实体并行检索，但单 session 做不到。解决方案是引入**连接池**，维护多个 session，检索请求并行分发。

  **第三是超时策略**——应该改成 10 秒首次超时 + 15 秒重试 + 熔断机制，连续失败后自动降级到 Web Search fallback。

  **第四是 stderr 管道阻塞**——这是 stdio 模式特有的坑，RAG Server 日志写满 pipe buffer 后会阻塞整个进程。代码里用 monkey-patch 把 stderr 重定向到 devnull 解决了，但丢掉了所有错误日志。

# YAML 配置驱动 + 工厂模式装配组件

我的做法是先把系统里可变的东西都抽到 YAML 里，比如 LLM 提供商、模型名、MCP 连接方式、Web Search fallback、memory 和 tracing 的路径等。启动时先读取 YAML，再结合环境变量做解析，最后用 Pydantic 做强校验，确保配置是完整且合法的。

配置拿到以后，不是业务代码里到处 if else 去 new 对象，而是统一交给工厂函数处理。比如 各类 Strategy 节点也都有各自的工厂。工厂的作用就是根据配置决定实例化哪种实现，并把依赖提前绑定好。

最后在主入口里，把这些通过工厂创建出来的组件统一传给 graph builder，由 graph builder 负责把节点注册到 LangGraph，拼成完整执行链路。

## 短期记忆

### 上下文管理

统一存在AgentState里面，里面包含对话的消息历史，当前问题，识别出的意图以及补全的信息，检索结果，当前答案和critic反馈

短期会话状态靠 LangGraph 的 checkpointer 记住，区分不同会话靠 thread_id，同一个 thread_id 会延续上下文，不同 thread_id 相互隔离。

### 多轮上下文怎么管理

LangGraph管理多轮上下文的核心就是它的**State机制**。整张图共享一个State对象，每一轮对话的所有信息都在State里面流转和累积。

#### **问题1：上下文越来越长，token爆炸**

多轮对话聊久了messages列表会很长，直接全塞给LLM会超context window，而且token成本也顶不住。

**解决方案是做上下文压缩**，几种策略：

滑动窗口——只保留最近N轮对话，早期的消息丢弃。最简单但会丢失重要的早期信息。

摘要压缩——在State里加一个summary字段，每隔几轮让LLM把之前的对话压缩成一段摘要，后续对话的context就是summary加最近几轮messages。这样既控制了长度又保留了关键信息。我在Paper Pilot里用的就是类似思路。

token budget控制——设一个上限，比如context最多给4000 tokens，超了就触发压缩。可以在图里专门加一个"context管理"节点，每轮对话前先检查token量，超了就压缩。

#### **问题2：不是所有信息都该放在messages里**

多轮对话中会产生很多中间信息——检索到的文档、工具调用结果、内部推理过程。这些全塞messages里会导致对话历史很脏，LLM容易被噪声干扰。

**做法是State里分字段管理**：

python

```python
class State(TypedDict):
    messages: Annotated[list, add_messages]     # 用户和LLM的对话
    retrieved_docs: list                         # 当前轮检索到的文档
    user_intent: str                             # 当前用户意图
    conversation_summary: str                    # 历史摘要
    tool_results: dict                           # 工具调用结果
```

messages只存用户可见的对话内容，其他信息用独立字段管理。各节点按需读取各自需要的字段，不会互相污染。这是LangGraph比纯message passing模式强的地方——State可以携带结构化的上下文，不是所有东西都要塞进message列表。

#### **问题3：多轮之间需要跨session持久化**

用户关掉页面再回来，对话历史不能丢。LangGraph提供了**Checkpointer机制**，可以把State持久化到外部存储。

内置支持MemorySaver（内存，开发用）和SqliteSaver / PostgresSaver（生产用）。每一步执行完都会自动保存一个checkpoint，包含完整的State快照。恢复的时候传一个thread_id就能从上次的状态继续。

实际用法：

python

```python
from langgraph.checkpoint.memory import MemorySaver

checkpointer = MemorySaver()
graph = workflow.compile(checkpointer=checkpointer)

# 每次调用传thread_id
config = {"configurable": {"thread_id": "user_123"}}
graph.invoke({"messages": [user_input]}, config)
```

同一个thread_id的调用会自动接上之前的State，实现跨session的多轮上下文。

### 第一层长期记忆：事件记忆

> "每轮对话结束后，从问答内容中抽取关键事实性陈述，持久化到本地。下次对话开始时，根据用户当前问题召回语义最相关的历史事实，注入上下文辅助回答。"

> "抽取有两条路径。**优先走 LLM 结构化输出**——把问题和回答送给大模型，要求它用 Pydantic schema 返回 0-5 条值得记住的事实。prompt 里明确约束：只抽具体的、自包含的事实性陈述，包含论文名、方法名、量化结果；不抽泛泛的空话；如果回答质量低就返回空列表。"

> "**LLM 不可用时 fallback 到启发式**——把回答按句号切分，对每个句子做打分：句子长度（封顶 150 字）、是否包含数字（+25 分）、长单词数量（≥10 字符 +8 分）、首字母大写的专有名词（+5 分）。取 top-2 高分句子作为事实。本质上是用简单信号识别'信息密度高的句子'。"

> "新事实入库前做**语义去重**。用 `all-MiniLM-L6-v2` 给新事实和所有已有事实算 embedding，找到余弦相似度最高的那条，如果 **≥ 0.92**，判定为重复，做 **upsert**（用新的替换旧的，更新时间戳）；低于 0.92 才追加。"

> "这个 0.92 是在 MiniLM 的 embedding 空间里手动校准的——0.92 对应'措辞不同但语义几乎完全一致'的程度。太低会误合并相关但不同的事实，太高相当于没去重。"

> "容量超过上限（默认 1000 条）时，按 **LRU** 淘汰——按 timestamp 排序，删掉最老的。淘汰后整个 JSONL 文件重写一次。"

#### 召回

> "每轮对话开始时，`load_memory` 节点拿用户当前问题做 embedding，和所有事实的 embedding 算余弦相似度，返回 **top-5**。这 5 条事实写入 `state.accumulated_facts`，后续的检索和生成节点可以用。"

> "embedding 模型不可用时 fallback 到**关键词重叠**——问题和事实都抽关键词，算交集大小排序。"

#### 质量门控

> "还有一个细节：**Critic 没通过且重试用尽的低质量回答不入库**。save_memory 节点会检查 critic_verdict，如果质检未通过且已经重试到上限了（说明是被强制放行的低质量回答），就跳过入库，避免低质量事实污染记忆。除非走了网络搜索兜底——那说明是因为知识库不够导致的，回答质量本身可能还行。"

#### 存储与降级

> "存储用 JSONL，一行一条 JSON。选 JSONL 而不是 SQLite 或向量数据库，是因为 1000 条事实用 embedding 相似度检索完全够用，不需要近似最近邻索引。而且 JSONL 可读、可调试、可 git diff。"

> "embedding 用 `sentence-transformers/all-MiniLM-L6-v2`，22MB 的轻量模型，懒加载，首次使用才加载。**整个记忆系统做了优雅降级**——如果 sentence-transformers 没装，自动 fallback 到关键词检索；如果 LLM 不可用，自动 fallback 到启发式抽取。图在任何配置下都能跑。"

### 第二层长期记忆：用户研究画像

#### 为什么需要

> "事件记忆是细粒度碎片，比如'用户问了 ColBERT 和 DPR 的对比'、'用户上传了 late interaction 的论文'。这些碎片多了之后全塞 context 太浪费 token。我需要一层**压缩后的全局画像**——用两三百个 token 传递最核心的用户上下文。"

#### 归纳触发与流程

> "触发条件有两个：**数量触发**——每累积 30-50 条新事件记忆跑一次归纳；**时间触发**——用户新开会话时检查上次归纳后有没有足够多的新事实。"

> "归纳本质上是一次 LLM 摘要调用。输入是当前画像加新增的事件记忆，prompt 里要求三件事：新旧矛盾以新的为准（研究方向可能转了），反复出现的信息提升置信度，只保留对未来交互有指导意义的内容。输出是结构化的 Pydantic 对象。"

#### 两个边界情况

> "**研究方向转变**——用户可能从课题 A 转到了课题 B。prompt 里要求 LLM 识别这种转变，把不再活跃的方向归档到 `archived_directions`，而不是和新方向混在一起。"

> "**画像衰减**——如果某个子话题超过一定时间没出现在新的事件记忆里，下次归纳时自动降权或归档。避免过时信息干扰当前交互。"

#### 版本管理与溯源

> "每次归纳不覆盖旧画像，追加新版本带时间戳。每条画像记录了它是从哪些事件记忆归纳来的——如果用户问'你为什么觉得我对 multi-vector 感兴趣'，可以追溯到具体哪次对话。"

#### 怎么用

> "每次新会话开始时，从存储拉最新画像拼进 system prompt，大概两三百个 token。**对路由模型帮助最直接**——用户问了一句'这个方法怎么改进'，没画像路由模型不确定走多跳推理还是开放探索；有画像知道用户在做 ColBERT 压缩方向的文献调研，就能确定这是开放探索。"

------

### 面试中如果被追问

| 问题                        | 回答要点                                                     |
| --------------------------- | ------------------------------------------------------------ |
| **"两层之间什么关系"**      | 事件记忆是原材料，画像是归纳产物。事件记忆用于精确召回（给到特定问题），画像用于全局引导（给到每轮对话）。两层的 **token 分工**不同：事件记忆是按需检索的大仓库，画像是永远在线的紧凑摘要 |
| **"为什么不直接用 Mem0"**   | Mem0 面向多用户 SaaS，有 user/session/agent 三级作用域和 Redis+Neo4j 的基础设施。我是单用户学术工具，而且我需要**领域特化的画像 schema**——研究方向、论文态度这些字段通用方案不会有 |
| **"去重 0.92 怎么来的"**    | MiniLM embedding 空间里手动校准。找了几十对"说法不同但意思一样"和"相关但不同"的句子对，0.92 能把两类分开。太低误合并，太高没去重 |
| **"JSONL 不会很慢吗"**      | 1000 条事实，每条 embedding 384 维，全量余弦相似度是 384×1000 次乘法，微秒级。远没到需要 FAISS 或向量数据库的规模 |
| **"Critic 没过为什么不存"** | 低质量回答里的"事实"本身不可信。如果存了，下次召回会把错误信息注入上下文，导致错误传播。但如果是走了网络搜索兜底的，说明是知识库缺失而非回答错误，可以存 |



# 评测

## 自动化评测怎么做的



> 我们的评测体系分三层：**模型评测、检索评测、端到端评测**，覆盖了从训练到上线的完整链路。设计上遵循**可插拔、配置驱动、黄金集回归**的原则。

------

### 第一层：模型评测（训练阶段）

> 我们本地微调了两个 LoRA 模型 — Router 做意图分类，Critic 做答案质量评估。

**Router 评测：**

> 用 holdout test set 评分类准确率，按 5 种 intent（factual、comparative、multi_hop、exploratory、follow_up）分别计算 per-class accuracy，重点关注混淆严重的类别对，比如 factual 和 follow_up 容易混。支持 local LoRA 和 vLLM 两种推理后端，确保训练完直接能评。

**Critic 评测：**

> Critic 经过 SFT + DPO 两阶段训练。SFT 阶段看生成的 score/completeness/faithfulness 和标注的偏差；DPO 阶段看 chosen vs rejected 的偏好对齐率。

------

### 第二层：检索评测（RAG 阶段）

> 这是重点。我们设计了一个**工厂模式的评估器框架**，支持三种评估器可插拔组合：

```
EvaluatorFactory
  ├── CustomEvaluator  — 快速、确定性、不依赖 LLM
  ├── RagasEvaluator   — LLM-as-Judge，深度质量评估
  └── CompositeEvaluator — 任意组合，指标合并
```

**（面试官追问：具体评什么指标？）**

> 两类指标：

| 类型     | 指标                  | 含义                             | 是否需要标注       |
| -------- | --------------------- | -------------------------------- | ------------------ |
| 检索质量 | **Hit Rate @K**       | top-K 结果中是否命中了期望 chunk | 需要               |
| 检索质量 | **MRR**               | 第一个相关 chunk 的排名倒数      | 需要               |
| 答案质量 | **Faithfulness**      | 答案是否忠于检索上下文，不幻觉   | 不需要（LLM 判断） |
| 答案质量 | **Answer Relevancy**  | 答案是否切题                     | 不需要             |
| 答案质量 | **Context Precision** | 检索的 chunk 是否相关且排序合理  | 不需要             |

> Hit Rate 和 MRR 是确定性指标，跑得快、可复现，适合做 CI 回归门控。Faithfulness 这些用 Ragas 框架，本质是 LLM-as-Judge，更慢但能评无标注场景。

**（面试官追问：Golden Test Set 怎么构建的？）**

> 我们维护了一个 JSON 格式的黄金测试集，每条包含 query、expected_chunk_ids、reference_answer。评估脚本读取这个文件，对每个 query 跑混合检索（Dense + BM25 + RRF 融合 + CrossEncoder 重排），然后和 expected_chunk_ids 对比算 Hit Rate/MRR。

------

### 第三层：端到端评测（Agent 阶段）

> Agent 层面做了延迟基准测试。25 个 benchmark 问题覆盖五种意图类型，分别测量：

| 指标                | 粒度           |
| ------------------- | -------------- |
| Router LLM 调用延迟 | per-node       |
| Critic LLM 调用延迟 | per-node       |
| 端到端延迟          | 全链路         |
| Token 消耗          | input + output |

> 统计 mean、p50、p95、max。支持 cloud 和 vLLM 两种 mode 对比跑，这样可以量化本地推理带来的延迟收益。

------

### 设计亮点

1. **工厂 + 组合模式**：评估器通过 settings.yaml 配置，`provider: composite` + `backends: [custom, ragas]` 就能混合使用，不改代码
2. **快慢分离**：Hit Rate/MRR 秒级出结果，放 CI 做门控；Ragas 分钟级，定期跑深度评估
3. **E2E 回归测试**：pytest 里有阈值门控（`assert hit_rate >= threshold`），检索质量退化时测试直接红
4. **Tracing 联动**：每次 Agent 运行都写 JSONL trace，记录 per-node 延迟和 reasoning steps，可以回溯任何一次跑得慢或答得差的原因

------

### 还有什么可以改进的

> 坦诚说几个方向：
>
> - Golden test set 还不够大，标注是手动的，后续考虑用 LLM 辅助生成 + 人工校验
> - 评估目前是脚本手动触发，还没接 CI/CD 自动化
> - 缺少历史趋势对比，每次评估结果没有持久化存储和可视化
> - 还没有做 A/B 对比评测（比如换了 embedding 模型后，自动跑一轮回归看检索质量有没有退化）

## 项目的全链路追踪和可观测性

借鉴参考了美团Vitabench论文设计了现在的评估和监控方案

一是**端到端延迟基准**。 论文里会用多轮 run（比如 k=4）看任务成功率和稳定性。我们参考这个思路，设计了针对固定问题列表的延迟基准：对同一批问题多轮跑完整 Agent，算 均值、50% 分位数、95% 分位数、最大值，并累计 token，结果落 JSON。这样既能对比不同部署（cloud / vLLM）的延迟，也能统计通过率、看稳定性。

二是**Trace 持久化**，论文用 rubric + 滑动窗口对长轨迹做评估，并按错误类型归因（推理、工具、交互）。我们参考这种「轨迹可复盘、可评估」的设计，每次请求都会记录轨迹状态到 JSONL，包含问题、意图、策略、检索 query、context 数、Critic 的 verdict、重试次数、最终答案，以及各模块耗时。采用llm judge的方式输出评测报告。

三是**节点级计时**。 为了支撑轨迹分析和瓶颈定位，我们设计了节点级打点：每个图节点执行前后记录 start/end，把各节点耗时和简要 output 写入 Trace。同时我们rag中的每个步骤都记录了计时以及token消耗。

## 项目评价指标的布尔型指标和过程指标

布尔型这块主要有两个：一是 Critic 的 passed，回答质量是否过关。我们不是让模型直接输出 true/false，而是先打一个 0 到 10 的 score，再用规则：score ≥ 7 就算 passed，用来决定是直接通过还是进重试。二是 Router 意图分类的对错：在评估集上比较「预测的 intent」和「标注的 intent」是否一致，算 准确率，本质是每个样本一个布尔（对/错）再求平均。

过程指标主要是：Critic 的 score、completeness、faithfulness 和 feedback，以及 Router 的 confidence，用来看质量细项和置信度；还有延迟（router、critic、端到端）和 token 用量、重试次数 等，在 benchmark 里会算 mean、P50、P95 这些。过程指标主要是观测和对比（比如 Cloud 和 vLLM 的延迟、不同策略的 retry 情况），没有再用一个硬阈值去判「通过/不通过」。



# 项目难点

微调Router模型的时候其实踩了不少坑。最早第一版微调效果很差，意图分类的准确率上不去，loss卡在一个比较高的位置下不来。当时排查了很久，最后通过打印训练日志发现，是tokenizer的`max_length`设小了——因为我的system prompt模板拼上用户query再加上label，tokenize之后的总token数超过了`max_seq_length`的设定值，导致输入在训练时被截断，模型根本看不到完整的指令和问题，自然学不好。错误的初始值256，统计后发现P95在1000左右，最终设成了512。

定位到问题之后，我写了个脚本去遍历整个意图分类数据集，对每条样本用tokenizer实际编码一遍，统计了input_ids的长度分布，找到了p99和max的边界。然后把`max_seq_length`调到能覆盖绝大多数样本的值，错误的初始值256，统计后发现P95在1000左右，最终设成了512。

但离目标还有差距，尤其是一些语义比较接近的意图之间容易混淆，比如模型经常把comparative的query错分成multi_hop——因为两者都涉及多个实体，表面看都像"需要查多次"的问题，但实际上执行逻辑完全不同：multi_hop是串行的链式依赖，comparative是并行的多实体对比。所以第三阶段我重点做了**prompt工程**，经历了三轮迭代：

**第一轮：裸分类prompt。** 最早的prompt非常简单：

> "请将以下问题分类为：simple / multi_hop / comparative / exploratory。问题：{query}"

模型只能靠query的表面语义去猜，区分度很差，特别是comparative和multi_hop之间几乎是随机的，exploratory也经常被误判成multi_hop。

**第二轮：加入意图描述。** 我在prompt里给每种意图补充了执行逻辑和适用场景的说明：

> "请根据问题的推理模式将以下问题分类。分类规则：
>
> - simple：单事实、单跳查询，一次检索即可回答；
> - multi_hop：需要多步依赖推理，先找A再用A推导B，步骤之间有前后因果链；
> - comparative：涉及多个实体的对比，需要对每个实体并行收集信息后做结构化比较；
> - exploratory：开放式探索问题，目标不明确，需要边查边调整搜索方向。 问题：{query}"

这一轮有改善，但边界case还是不稳定，尤其涉及多篇论文的query，模型还是分不清该走comparative还是multi_hop。

**第三轮：加入few-shot示例。** 我从验证集和实际错误case里挑了典型样本，给每个意图各放了1-2个few-shot example，重点针对容易混淆的类别对做区分：

> "请根据问题的推理模式将以下问题分类。分类规则：
>
> - simple：单事实单跳查询，一次检索即可回答
>   - 示例：'这篇论文用的是什么数据集？' → simple
>   - 示例：'Transformer的self-attention公式是什么？' → simple
> - multi_hop：多步依赖推理，步骤间有因果链
>   - 示例：'这篇论文提出的方法基于什么前置工作？那个前置工作的核心局限性是什么？' → multi_hop
>   - 示例：'先找到Table 3中效果最好的配置，再解释它为什么优于其他ablation变体' → multi_hop
> - comparative：多实体并行对比，维度一致的结构化比较
>   - 示例：'对比GPT-4和LLaMA-2在摘要任务上的表现差异' → comparative
>   - 示例：'这三篇论文分别用了什么检索策略？各有什么优劣？' → comparative
> - exploratory：开放式探索，目标不明确，需要迭代搜索
>   - 示例：'RAG领域目前有哪些前沿研究方向？' → exploratory
>   - 示例：'如果要改进当前的检索pipeline，有什么可能的方案？' → exploratory
>
> 问题：{query}"

不过加了few-shot之后prompt变长了很多，这也回过头来验证了第一阶段排查截断问题的重要性——如果`max_seq_length`没调对，few-shot的示例反而会把真正的query挤掉，适得其反。

同时在训练策略上，我适当增加了训练steps，并且发现在我这种小数据集的场景下，调低learning rate反而能让收敛更稳定，避免在少量样本上过拟合震荡。最终Router的Macro-F1从基线的83%提到了91%，其中提升最大的就是comparative和multi_hop之间的区分准确率。
